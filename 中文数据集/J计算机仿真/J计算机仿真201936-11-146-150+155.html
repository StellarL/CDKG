    摘    要：

    随着远程医疗的发展,越来越多的用户将自己的数据存储在医疗云服务器。云服务器可以维护医疗数据,提高医疗服务质量,降低医疗成本。然而,云存储中安全问题必须得到足够的重视。例如,未经身份验证的用户可以访问云服务器,云服务器未按照用户的要求删除数据。为了解决这些安全问题,提出一直基于区块链的远程数据验证协议。首先用户和云服务器之间通过相互认证来验证彼此的身份。当确认用户身份合法后,创建交易请求;然后云服务器生成一条区块链(证据链)。最后用户可以验证数据删除结果。通过实验分析,所设计的协议可以在没有第三方可信机构的情况下完成数据的公开验证,同时可以抵抗窃听攻击、假冒攻击等。

    关键词：

    医疗云; 公开验证; 区块链; 数据删除;

    Verification Protocol of Remote Data Based on Block Chain

    FU Yue-jun LI Zhong-zheng WANG Shi-jie

    Tongren Cigarette Factory,China Tobacco Guizhou Industrial Guilin University Techonlohy

    Abstract：

    With the development of telemedicine, more and more users store their data in medical cloud. The cloud server can maintain medical data, improve medical service quality and reduce medical cost. However, security issues in cloud storage must be paid enough attention. For example, an unauthenticated user may access the cloud server, and cloud server may dishonestly delete data for financial gain. Therefore, in order to solve these problems, a verification protocol of remote data based on block chain is presented in this paper. First, the user and the medical cloud authenticate each other through mutual authentication. After the user is legally identified, the transaction request is created, and the cloud server generates a block chain(evidence chain). The user can verify the data deletion result. Through the experimental analysis, the protocol designed in this paper can complete the public verification of data without a third-party trusted organization, and can resist the eavesdropping attack, the fake attack.

    Keyword：

    Medical cloud server; Public verification; Block chain; Data deletion;

    1 引言

    随着物联网的流行,大量的数据存放在云平台上。云存储使用范围很广,远程医疗是其中的一个典型应用。但是在云存储中也存在很多安全和隐私隐患。用户最担心的是服务器没有按自己的请求去调用设备,或者返回错误的结果来欺骗自己。数据验证能有效解决这个问题,其中最典型的就是公共验证。为了解决数据删除操作中的数据验证问题,近些年提出了很多数据完整验证方案[1,2,3,4,5,6,7]。数据完整验证的两个典型方法是PDP协议和POR协议。PDP(packet data protocol)是基于公钥技术的协议,支持三方验证。POP(post office protocol)协议是在数据块之间插入一些验证块作为验证依据。下面分析几个具有代表性的协议。

    1)POR 协议。 POR协议[8]是由Juels等人在2007提出的。在POR协议中,验证者需要通过加密秘钥去验证,密钥的用户的文件大小和数量都没有关系。该方法在文件中插入一些随机性的检测块。用户使用加密算法对这些信息加密。验证时,验证者需要向云服务器提出验证申请。云服务器根据申请的内容是否正确返回结果。如果云服务器已修改或删除该文件的一部分数据,返回的结果就有可能不正确。POR协议使用误差校正码来扩展文件,从而避免文件部分删除或篡改导致文件损坏。POR协议的验证次数受限制同时不支持数据动态修改。

    2)PDP 协议。PDP协议[9]是由Ateniese等人在2007提出的。该协议是一个概率性的验证协议,通过访问文件数据块的子集来验证完整性是否被破坏。该协议对每个数据块使用同态验证技术进行标识,并把文件何所有标识都发给云服务器。客户端向云服务器发送随机的验证请求。云服务器对申请的内容进行验证和选择相应的数据块与之验证标识计算出返回结果。客户端根据返回结果判断数据完整性从而不需要下载整个文件验证。

    目前,已有很多学者对此进行研究。Wang等人[10]提出一个在云存储环境的数据动态改动并支持第三方验证。该协议考虑了在远程数据完整性验证过程中的用户隐私问题。Chen等人[11]为基于网络编码的分布式存储系统[12]提供了一种远程数据完整性验证协议。该协议能够防止数据被破坏攻击,并且能够在客户端仅占用恒定的存储空间。Ateniese等人[13]提出两种可证明数据持有协议。它是针对云存储中远程数据的单一副本完整性进行设计的,对数据的公开验证性和动态的数据追加操作。Yu等人[14]提出了一种开放式云存储数据验证方案。用户不再需要从云中下载数据,直接使用云中的小型认证器进行认证。该方案有效降低了通信成本和技术成本,使用户可以有效更改验证标记。Curtmola等人[15]提出将文件存储在一个多副本可证明数据持有系统中具有较高效率,并且使得数据所有者能够高效的验证其文件在多个服务器断的完整性。虽然以上提到的方法具有数据完整性验证协议,但大多数方法运算较为复杂,不适合在面向物联网的云存储环境中使用。因此,能够高效验证物联网设备对云存储数据的完整性是至关重要的。

    本文基于区块链技术,设计了面向云平台的数据验证协议,尤其是降低了用户对云服务器的信任假设。当云服务器发送恶意操作时,任何人都可以根据协议中的区块链验证操作结果。主要创新点如下:

    1)创建了一个基于区块链的远程数据验证系统。如果云服务器没用诚实的删除数据,协议可以为用户提供检查服务器恶意行为的功能。而且,协议不需要任何的可信第三方的支持。

    2)在实现数据公开验证的同时,协议具有通信和计算上的高效性。

    2.2 Merkle树

    Merkle树[16]是一种哈希二叉树,是一种做快速归纳和效验大数据完整性的数据结构。Merkle树是通过递归对哈希节点进行哈希生成的,生成新的节点插入到树中,直到剩余最后一个哈希节点为止。将记录信息哈希化后,叶子节点存储对应的哈希值。然后,相邻叶子节点的哈希值串联哈希后生成父节点。重复同样的操作直到只剩下一个顶部节点为止,即根节点。区块链中的Merkle树存储交易信息。通过交易两两配对构成Merkle树的叶子节点。用户通过Merkle树的根和别的用户所提供的哈希值列表去验证交易是否包含在区块中。

    2.3 区块链技术

    1994年本世聪第一次提出区块链概念[17],并将其用在比特币系统中。区块链是一个对等网络[18]的去分布式数据库。其核心是通过分布式共识机制,实习多方可信和价值互联。区块链技术是随着比特币等货币而被挖掘兴起的技术,以区块链技术构成的系统网络中,有事务被写入区块中,事务数据必须通过共识机制由系统中的所有节点共同维护,并且任何单一节点都无法进行修改。区块和区块之前紧密连接,形成区块链。区块链为每个参与者提供分散信任的账簿,每个用户或节点共同维护并存储这个完全相同的分类账簿,这确保了所有用户和相应区块链中的节点完全一致。在分类账中,交易和历史数据都会永久存在区块中,其持久性基于网络的持久性。如果链上的一个区块中的数据被非法更改,其它节点会识别出非法节点,并删除这条链。此链式具有很高的安全性,保证数据具有不可篡改性。区块链的长度不断增加同时系统的安全性也在不断的增强[19,20]。

    目前,区块链分为公有链、联盟链和私有链。

    公有链:公有链没有访问控制权限,任何人都可以随时进入系统中进行数据操作和发送交易。公有链是完全去中心化的,数据是默认公开的。

    联盟链:联盟链是指由若干个机构共同参与管理的区块链,只允许系统内的机构进行读写和发送交易,共同记录交易。

    私有链:私有链是指其写入权限仅在一个组织手里的区块链。由于参与节点有限和可控,因此私有链交易速度快、隐私性好、交易成本低、不易被恶意攻击。

    3 系统模型和设计目标

    3.1 系统模型

    基于区块链的远程医疗数据验证系统结构如图2所示,其中包括医疗服务器和用户。在注册阶段,用户通过安全的方式向服务器发送注册信息。服务器将收到的用户身份信息嵌入产生的区块中,并形成一条区块链发布在服务器间的区块链网络中。除了注册阶段,所有的通信都在公开信道上。

    文中的系统以比特币系统为模板,信息都以交易的形式发送和接收。当用户发送一个请求交易给服务器,服务器查询身份链验证用户的身份信息。如果用户是合法的,服务器接收交易并验证交易。验证合法后,服务器会调用设备,同时为调用的设备产生一个调用证据,并将其放入区块中形成一条区块链(证据链)。最后,服务器将证据发送给用户,用户根据证据链验证证据是否合法。

    3.2 攻击模型

    文章中,认为医疗服务器是半可信的,它不会泄露数据隐私,但是它会为了自己的利益不诚实的调用正确设备的数据或者产生错误的调用结果去误导用户。因此,存在以下几种攻击类型

    1)外部攻击:

    攻击者窃听和篡改传输在公开信道上的信息,并执行窃听攻击和重复攻击等。

    2)内部攻击:

    服务器在收到用户的调用请求后,为了自己的利益没有调用正确的设备或者发出错误的调用结果误导用户。

    3.3 设计目标

    设计目标如下所示:

    防窃听攻击:通过对公开信道的窃听,攻击者没法获得任何有用信息。

    防重复攻击:过时的信息重新发送时,它将被丢弃。

    完整性:用户可以检测半可信服务器篡改数据的恶意行为。

    可追溯性:如果服务器没有诚实的调用设备,用户可以通过调用证据追溯服务器的责任,并且服务器不能否定。

    4 基于区块链的远程数据验证协议

    4.1 高层次描述

    本文提出一种基于区块链的远程数据验证协议。该协议中,服务器是半可信的,用户不相信服务器是按自己的意愿执行数据操作的。为的了解决这个问题,在先前的研究中,很多学者都是用一个可信第三方来解决这个问题。然而,文中的方案不需要使用可信第三方实现数据的验证。协议的主要过程如图3所示,用户向医疗云服务器提出身份认证请求。用户合法后,构建一个含有设备调用请求的交易并广播,服务器确定交易后,检验交易的合法性并对相应设备发出调用请求。发出请求后,服务器为用户产生一个可以验证调用结果的证据,并将这个证据嵌入区块中形成一条可验证调用数据的区块链(证据链)。通过这条证据链,用户可以验证服务器是否诚实的调用了请求设备。

    图3 协议流程图   下载原图

    4.2 方案

    在本节,提出一个基于区块链的远程数据验证系统。该系统包括几个阶段:注册阶段,身份认证阶段,请求交易生成阶段和验证阶段。

    4.2.1 注册阶段

    在这部分,用户发送自己的身份信息给服务器,服务器将用户和自己共享的秘密BPWi嵌入区块,并公布在所有服务器之间,形成一条具有身份验证功能的区块链。具体步骤如下:

    1)用户生成自己的身份信息IDi,密码PWi以及一对公私钥PKi/SKi,并用指纹传感器录入生物信息Bi。然后,用户计算一个假名PIDi=h(IDi,t)和验证信息 BPWi=h(PWi,Bi)。用户以安全的方式发送IDi,PIDi,BPWi给服务器。BPWi只有用户和服务器知道。

    2)服务器将PIDi,BPWi嵌入区块链中。在服务器之间发布这条区块链(身份链)。

    4.2.2 认证阶段

    这个阶段,利用身份-权限链条完成用户的身份认证。如果用户是合法的,系统会为用户的每一次请求产生一个医疗钱包。 医疗钱包类似于比特币钱包,主要为用户生成公私钥,产生交易。

    1)用户输入自己身份信息IDi,PWi和生物信息Bi并计算验证密码BPWi=h(PWi,Bi)。然后,用户发送IDi给服务器。

    2)服务器生成随机数rs,并发送给用户。

    3)用户计算认证信息AHi=h(BPWi,rs),然后发送AHi,ru给服务器,其中ru是用户产生的随机数。

    4)服务器利用区块链中的BPWi验证AHi,如果验证成功,计算认证信息ATHi=h(BPWi,ru),并发送给用户。

    5)用户计算ATH′i=h(BPWi,ru)。如果等式ATHi=ATH′i成立,系统自动为合法用户生成一个医疗钱包。医疗钱包类似于比特币钱包,主要为用户生成公私钥。

    4.2.3 请求交易生成阶段

    1)当用户发出调用请求,钱包为用户生成一对公私钥PKs,SKi。并创建一个含有用户调用请求的交易。医疗钱包为用户建立一个请求交易Tx=<Adu,Set,SigReu,t1,Ads>,交易包括用户的地址Adu=SHA(PKi),服务器的地址Ads=SHA(PKs),交易内容,交易内容的签名和产生交易的时间戳t1。其中交易内容包括用户的调用请求Set=(PIDi,TID1,...,TIDj),交易的签名为SigReu=SigSKi(Set,t1)。最后,用户在网络中广播Tx。

    2)网络中的其它节点验证用户地址是否正确,如果正确,将请求交易放入区块链中的未交易池中,等待服务器的确认。

    3)服务器利用自己的私钥SKs确认交易Tx后,必须解析这个交易。首先验证时间戳t1,然后使用PKs验证签名SigReu=SigSKi(Set,t1),所有是合法后,服务器才会调用设备。否则,服务器拒绝执行或者发出错误信息。

    4.2.4 证书生成阶段

    1)调用设备后,服务器为调用的设备建立调用证据proofj, proofj=(TIDj,SigReu,Sigs,t1),其中Sigs=SigSKs(TIDj,t1),SKs是服务器的私钥。同时,服务器用Merkle树结构将这些proofj归纳到一个区块并生成根节点rooti。服务器产生的新区块哈希值hi=H(hi-1,tsi,rooti), tsi是当前区块产生的时间戳,其中hi-1是哈希链中的前一个哈希值。最后服务器发送proofj,tsi给用户。这样区块链网络中的每个节点都会存储一条如图2所示的哈希链。

    2)用户收到最终的证据l=(proofj,rooti,hi,tsi),通过l可以验证服务器是否诚实调用设备。

    4.2.5 验证阶段

    本阶段,分为两个部分来验证l的正确性。首先,要证明proofj所在区块在当前的证据链中。因此,用户通过验证如下等式就可以追溯到最新发布的区块哈希值。如下方法

    hi=H(hi-1||tsi-1||rooti) (1)

    hi+1=H(hi||tsi+1||rooti+1) (2)

    ……

    hm-1=H(hm-2||tsm-1||rootm-1) (3)

    hm=H(hm-2||tsm||rootm); 其中hm是最近发布的哈希值。

    当用户接收到l后,可以验证l的正确性。为了不失一般性,用上述证据中proof4作为一个例子来解释验证过程。为了验证服务器调用设备的结果。用户首先计算h3,4=H(proof4),如果h′3,4=h3,4成立,然后用户遍历Merkle树验证根节点。具体过程如下(如图4所示)

    h3,4=H(proof4) (4)

    h2,2=H(h3,3||h3,4) (5)

    h1,1=H(h2,1||h2,2) (6)

    root′i=H(h1,1||h1,2) (7)

    ……

    rooti´=?h0,1 (8)

    只有所有的验证和等式都是成立的,那么用户可以确定服务器是诚实的以及这些证据proofs是可信的。

    5 协议分析

    本节主要对提出的协议进行安全和性能两方面的分析。

    5.1 安全分析

    5.1.1 防窃听攻击

    攻击者在公开信道上窃听通信信息,并且获得信息PIDi,TIDj,AHi,ATHi,proofi,rooti,hi。通过散列函数的单调性,攻击者没法获得BPWi和任何有用的信息。因此,提出的系统能够抵御窃听攻击。

    5.1.2 防重复攻击

    如果对手试图使用过时消息来发起重放攻击,则由于使用新的随机数和时间戳来抵抗该攻击,所以它仍然是计算不可行的。为了完成这一目标,随机数和时间戳是必须的。此外,在必要时还可以使用日期信息来加强这一过程。

    5.1.3 提出的协议满足正确性

    如果医疗服务器被认为是诚实的,它将如实的调用用户请求的设备,然后用户得到的证据是l=(proofj,rooti,hi,tsi).其中:证据proofj,根节点rooti,区块前哈希值hi-1都是合法的。其次,由于H(hi-1||tsi-1||rooti),因此,等式hi'=hi总是成立。换句话说,hi总是验证阶段输出的结果。

    5.1.4 可追溯性

    如果云服务器不可信,用户可以通过证据τ=(proofj,rooti,hi,tsi)实现对云服务器的追溯问责。因为proofj里含有云服务器的签名Sigs和用户的签名SigReu,Sigs只能是服务器生成,所以用户可以证明云服务器是否诚实,同时SigReu是用户计算得到的,云服务并不能伪造它,如果云服务器没有按照要求删除是可以被检验出的。即协议满足可追溯性。

    6 结论

    本文设计了一个基于区块链的远程数据验证协议,采用区块链系统保证用户可以检验医疗云服务器的欺骗行为。如果不诚信的医疗云服务器欺骗用户,用户可以在没有第三方可信机构的帮助下对删除证据进行检验,证明服务器的欺骗行为。与其它相关协议相比,本文不需要第三方可信机构的加入,降低了信任机制,同时,协议的用户验证阶段具有高效性。

    Research on the method of Function Call Path Static Generation for Python

    SUN Ran MU Yong-min SHEN Mei-e

    School of Computer, Beijing Information Science and Technology University

    Abstract：

    With the rising of Python, more and more programmers have chosen Python as the preferred language for application development and scientific research. The paper proposes a static extraction method of Python function call path based on abstract syntax tree. Using the abstract syntax tree of Python as analysis basis, we parsed and extracted key information, extracted function call relation, constructed function call relation model, and then generated function call path of Python. Experimental results show that this method can generate the function call path of Python program more accurately, and provide some technical support for the automatic generation of coverage oriented test cases and the localization of software defects.

    1 引言

    在互联网技术和大数据技术发展迅速的时代, Python作为一种面向对象的解释型计算机程序设计脚本语言, 以其良好的可读性, 可交互性, 可扩展性等特征, 吸引了众多开发者, 近年来已成为最受欢迎的编程语言之一, 逐渐成为科学计算、Web应用和游戏开发之首选语言[1]。对Python程序的理解和测试, 也显得越来越重要。

    目前国内针对Python程序源代码分析方面的研究相对较少, 国外在该方面的研究存在一些, 包括控制流分析、数据流分析[2]等静态分析方法的研究, 以及一些动态分析工具如cProfile。但针对Python的路径测试方法研究较少, 目前Python扩展模块PyCallgraph能够获取函数调用信息, 但提取的是函数包含关系, 没有逻辑控制结构, 得不到函数间调用执行顺序, 不能清晰观测函数调用的执行过程, 对路径覆盖测试帮助不是很大。

    针对上述问题, 提出一种基于Python抽象语法树的函数调用路径静态提取方法。实验证明, 该方法能够较为准确的生成Python的函数调用路径, 为面向路径覆盖的测试用例生成及缺陷定位研究提供一定的技术支持。

    2 相关技术

    2.1 路径覆盖生成技术

    路径覆盖的核心是用最少的测试用例, 发现程序中最多的错误。最小测试用例集合应能够覆盖程序中所有路径。程序中每条可能路径至少执行一次, 规则严格, 测试彻底。当程序规模较大时, 逻辑关系复杂, 路径数随分支数指数增加, 路径数量过于庞大导致完全路径测试无法完成。

    面向函数调用路径的思想是在回归测试和路径覆盖测试基础上发展起来的, 不仅简化了面向基路径的测试问题, 使得面向基路径的测试工作量大幅减少, 而且将面向路径的自动化测试变成可能[3,4,5]。

    考虑到面向过程语言的结构特性, 分支循环的复杂性, 以及对热点路径[6] (包括递归路径、不可达路径) 的处理, 基于函数调用路径的覆盖分析工具Regression Test C已实现。该工具通过动静结合的方法, 生成C语言带有控制逻辑的静态函数调用路径。

    考虑到面向对象语言的语法特点, 研究者提出一种基于Soot的面向Java语言函数调用路径生成方法[7], 利用Java编译优化框架Soot生成三地址文件, 设计多态唯一性静态分析算法和组合算法生成函数调用关系图。

    到目前为止, 研究人员先后提取了C语言[8]、C++、Java、C#[9]的函数调用路径, 但方法在多语言间不能通用, 而且之前并没有对Python进行相关研究。文章将对Python的静态函数调用路径的提取进行研究。

    2.2 Python抽象语法树

    在计算机科学中, 抽象语法树 (Abstract Syntax Tree, AST) 作为程序的一种良好中间表示, 不仅能够适应很多编程语言的语法结构, 而且还能达到对源程序预处理的目的。

    基于Python的抽象语法树AST可表示为一个对象列表, 每个对象包含多个子列表或其对象的引用。每个列表或子列表包含任意数量的节点, 每个节点代表一条语句并标有对应行号和偏移量。图1是通过IDLE获取一段python源代码的抽象语法树过程。

    图1 sub函数抽象语法树文本   下载原图

    调用AST模块parse () 方法, 得到抽象语法树对象expr_ast。其中详细记录了sub函数的信息, 父结点Module, 表示代码整体, 节点主体body包含一个函数定义FunctionDef, 属性包括函数名name、参数列表args、变量名Name和子body。子body包含一个Return结点, 包含一个Sub () 运算。

    以抽象语法树为提取函数调用路径的基础数据, 不仅节省解析源代码的时间开销, 而且能够保证对代码理解的准确性。

    3 关键信息提取

    每个AST对象包含代码段中所有执行信息, 信息量巨大, 存在冗余。如何在大量数据中过滤冗余信息, 提取与函数和控制流相关的关键信息, 是静态提取函数调用关系过程中的重要环节。

    3.1 基于抽象语法树的遍历

    在遍历抽象语法树的过程中, 可能会涉及多个操作, 这些操作大多要求对不同节点进行不同处理。若将所有操作分散到各个不同节点类中, 在导致代码可读性和可理解性变差的同时, 如果增加新操作, 就需要重新编译所有节点类, 会造成程序混乱等现象[10,11]。

    AST是一个深度嵌套的树形结构, 为避免上述遍历过程中可能出现的问题, 以及因递归深度限制带来的不确定影响, 文章采用非递归的访问者模式实现抽象语法树的遍历。

    函数调用路径是以函数为粒度提取出的程序执行路径, 首先需要对AST中多个关键节点信息进行遍历, 关键节点类型如表1所示。

    表1 关键节点类型 导出到EXCEL


    关键字    		描述

    ClassDef    		类定义

    FunctionDef    		函数定义

    Name    		名称属性

    Call    		调用信息

    Expr    		表达式

    For/While    		循环控制字

    If    		分支控制字

    Try/Except    		异常控制字

    Assign    		赋值关键字

    在遍历抽象语法树时, 定义如下几个角色:

    1) 抽象访问者角色 (Visitor) :也可称为访问者工厂, 该角色结合Python反射机制, 为每个节点提供通用访问入口visit () 。

    2) 通用访问者角色 (Generic Visitor) :该角色提供通用的访问方法generic_visitor () 。

    3) 具体访问者角色 (Concrete Visitor) :节点访问的详细操作, 方法结构为visit_xxx () , 如visit_FunctionDef () 和visit_If () , 前者是基于函数定义的访问者, 后者是基于分支if的访问者。

    遍历流程可描述为:对每一个node, 通过visit () 转发到各访问者, 如果存在明确的具体访问者visit_xxx () , 直接访问;如果不存在, 则通过generic_visitor () 访问。

    3.2 基于抽象语法树的关键信息提取

    缩进是python特有的表示语句块的方法, 语法树中的每一个节点都标有行号和偏移量, 可以唯一标识语句块。针对上述特点, 统一采用图2中的格式存储关键信息。

    图2 信息节点存储格式   下载原图

    row表示函数定义所在行, col是缩进列, ClassName表示函数所属类, FunctionName表示函数名, 如 (2, 4) Calculate:add (self, x, y) , 表示add函数属于Calculate类, 从第2行第4列起定义。由于函数可在类内或类外定义, 所以ClassName可缺省, 如 (1, 0) main () , 表示main函数从第一行第0列起定义。

    关键信息提取过程:Visitor从抽象语法树的根节点开始遍历, 每个与函数调用关系相关的节点类型都有专用的类型处理函数, 若不存在, 则调用generic_visitor进行处理, 算法如下。

    输入:抽象语法树文本AST Text

    输出:关键信息列表KeyInfoList, 函数字典FuncInfoDict

    Begin

    tree <-AST Text;//获取ast对象

    FOR node in tree DO//遍历每一个节点

    type <-node.type; //根据节点类型, 调用访问者

    IF existFunc (type) THEN

    visit_XXX (node) ;//如果存在具体访问者

    ELSE generic_visitor (node) ;//如果不存在

    ENDIF

    END FOR

    FuncInfoDict <-completeFuncDict () ;//补全函数字典

    //补全关键信息列表

    KeyInfoList <-completeFuncInfo (FuncInfoDict) ;

    END

    至此, 繁琐的抽象语法树文本转换成能够生成函数调用关系所需的关键信息列表。

    4 函数调用路径生成

    根据关键信息, 结合Python语言的控制结构特性, 设计函数调用关系模型构建算法, 构建全局函数调用关系模型, 通过遍历全局函数调用模型, 获取函数调用路径集。

    4.1 相关概念

    定义1:函数调用关系 (Function Call Relation) , 指两个函数之间的调用, 用R表示, 其中R={ (mi, mj) │mi是源函数, mj是被调函数}。如R={ (m0, m1) , (m1, m2) }表示源函数中函数调用关系为m0→m1→m2。

    定义2:函数调用关系模型 (Function Call Relation Model) , 表示单个函数的函数调用关系, 用三元组T (M, R, S) 表示, M是函数集合, R表示函数间的调用关系, S表示源函数。

    图3 示例源代码   下载原图

    图3代码段中main函数, 用函数调用关系模型T (M, R, S) 表示, 其中 M={main, f1, f2, f3}, R={ (main, f1, O) , (f1, f2, O) , (f2, f3, B) }, S={main}, O表示顺序调用, B表示分支调用。

    定义3:函数调用路径 (Function Call Path) , 根据函数调用关系得到的由程序入口到出口的函数名序列, 表示为Pi={Fi0, Fi1, … Fim}, Fij为函数名, Fij和Fij+1的相邻关系, 表示Fij调用 Fij+1或Fij和Fij+1顺序执行。

    定义4:函数调用路径集, 指全部函数调用路径的集合, 用表示, 其中S是源代码, C是函数调用关系准则, Pi是函数调用路径。

    定义5:函数调用路径图 (Function Call Graph) , 函数调用路径集的可视化图, 表示为一个有向图FCG (V, R) , V是FCG中所有节点的有穷非空集合, R是相连节点之间边的有限集合。

    图4 相关概念示例   下载原图

    图4左侧源代码包含函数main、f0、f1、f2和input。根据函数调用关系得到两条函数调用路径:P1={main, sys:input, f2, f1, f0}, P2={main, sys:input, f2, f0}, 全局函数调用路径集为B={P1, P2}。右侧是函数调用路径图, 图中每一个节点, 除Begin节点和Start节点代表全局FCG起点, End节点代表结束点外, 其它每个节点都表示一个函数, 如“ (1, 0) f0 () ”节点, (1, 0) 表示函数f0 () 定义的位置, 图中“→”表示前驱节点与后继节点的调用关系或顺序执行关系。

    4.2 函数调用关系模型分析

    根据函数调用关系模型定义, 以树形结构存储函数调用关系模型, 如图5所示。

    图5 节点模型   下载原图

    data表示函数名称, leftchild表示后继节点或分支节点, 若存在分支, 则代表分支节点, 若不存在, 则代表后继顺序执行的函数节点;rightbrother表示兄弟节点, 与父节点是并列关系, 在程序中表示多分支结构;flag记录分支深度。

    Python包含常见的控制关键字while、for、if、if-else、if-elif+-else, 但没有switch-case和do-while。Python为循环结构提供一个else语句块为辅助。对于顺序结构, 在语句中一般显示为先后顺序, 即先执行的在前且后执行的在后, 构建模型时较为容易, 这里不再赘述。

    4.2.1 分支关系模型

    Python中的分支语法结构, 大致可分为三种, 缺省型、标准型以及多分支型。

    1) 缺省语法结构:

    if <布尔表达式>:<语句>

    这种语法结构表示要么执行要么不执行, 由于没有else, 构建关系模型时, 自动补全空节点。

    图6 缺省语法结构   下载原图

    图6代码段中缺省else, if中调用函数f1 () , 构建函数调用关系模型时, main节点的leftchild指向函数为f1的节点, f1节点的rightbrother指向空节点, 表示在main函数被调用后, 或者执行f1函数, 或者不执行, Tmain (M, R, S) ={{main, f1}, { (main, f1, O) , (f1, None, B) }, {main}}。

    2) 标准语法结构:

    If <布尔表达式>:<语句F1> else:<语句F2>

    这种语法结构满足if条件时执行if语句块, 否则执行else语句块。若if-else都不包含函数调用, 则关系模型不存在。若if-else各自的语句存在一方没有函数调用, 则是缺省语法结构。若if-else的语句块中都包含函数调用, 其关系模型如图7所示。

    图7 标准语法结构   下载原图

    3) 多分支语法结构:

    if <布尔表达式>:<语句F1>

    elif <布尔表达式>:<语句F2>

    else:<语句F3>

    这种语法结构在关系模型中存在一个节点多个兄弟节点, 若存在一个分支处不存在函数调用, 则该分支为空节点, 如图8所示。

    图8示例代码段, 用Tmain (M, R, S) ={{main, f1, f2, f3}, { (main, f1, O) , (f1, f2, B) , (f2, f3, B) }, {main}}。

    4.2.2 循环关系模型

    对于循环结构, 采用Z路径[12,13]覆盖思想进行简化, 减少路径数量, 简化机制指无论循环的形式及实际执行循环体的次数是多少, 只考虑循环一次和零次情况, 即只考虑执行时进入循环体一次和跳过循环体这两种情况, 在提取函数调用路径中也只考虑这两种情况。

    1) while 语法结构:

    while <布尔表达式>:<语句>

    [ else:<语句> ]

    这种语法结构与if – else的结构类似, 表示程序要么执行要么不执行, 结合Z路径覆盖思想, 只考虑循环一次和零次。

    2) for语法结构:

    for <迭代参数> in <序列>:<语句>

    [ else:<语句> ]

    这种语法结构在python中至少会执行一次, 只需循环一次, 若存在else语句块, 则每次迭代都会执行else语句块, 与顺序结构类似。示例程序和模型见图9。

    4.3 函数调用模型构建

    以关键信息列表为输入, 构建局部函数调用关系模型, 算法3如下:

    输入:关键信息列表KeyInfoList

    输出:局部调用关系模型列表ModelList

    Begin

    FOR i <-0 to KeyInfoList.length DO

    flag <-FALSE

    list <-KeyInfoList (i)

    FOR j <-0 to list.length DO

    IF list (j) .index () ==0 THEN

    model <-createModel (“Begin”)

    ENDIF//如果是第一个节点, 则创建节点

    IF list (j) ==”elif” or list (j) ==”else” THEN

    flag <-TRUE //标识置真

    ENDIF //如果节点类型是分支关键字

    //如果节点是分支或循环的开始或结束

    IF list (j) .isLoopOrBranchFlag () THEN

    IF list (j) .index () ==list.length THEN

    model.insertLeft (“END”)

    ENDIF

    ENDIF

    IF flag==FALSE THEN

    model.insertLeft (list (j) )

    ELSE model.insertRight (list (j) )

    ENFIF

    ENDFOR

    ModelList <-ModelList+model

    ENDFOR

    END

    4.4 函数调用路径生成

    全局函数调用关系模型可由程序内的各个函数的内部调用关系模型组合得到。

    组合过程如下:遍历模型列表获取标识为Begin-Start的作为入口点, 从入口点的模型开始, 将每个函数调用关系模型插入到全局的函数调用关系模型中, 插入的模型的叶结点的左孩子指向被替代的那个函数的左孩子, 具体算法如下。

    输入:ModelList

    输出:GlobalModel

    Begin

    entryModel <-findEntryModel (ModelList) ;

    ModelList.delete (entryModel) ;

    //将局部调用关系模型插入到全局模型中

    FOR i <-0 to entryModel.length DO

    value <-entryMode (i)

    FOR j <-0 to ModelList DO

    model <-ModelList (j)

    IF model.containsHead (value) THEN

    value.data <-model.data;

    model.leaf <-value.subModel () ;

    value.leftchild <-model.leftchild;

    ENDIF

    ENDFOR

    GlobalModel <-GlobalModel+entryModel;

    ENDFOR

    END

    根据全局函数调用关系模型, 遍历全局函数调用关系模型, 获得全局函数调用路径。

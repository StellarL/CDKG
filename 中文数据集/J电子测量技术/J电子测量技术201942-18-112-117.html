    摘    要：

    近年来,工控领域面临着愈发严峻的安全形势。为了提高系统安全性,工控处理器引入拟态安全TOE。拟态安全TOE需要对3路异构处理器的TCP下行数据等长切片,而TOE接收的3路异构处理器下行数据存在时间和帧长差异问题,这极大影响了切片模块的性能。提出了事件驱动的高速切片机制,减少切片模块对系统通信的延迟;通过基于指针的环形链表缓存等长切片值,可以匹配处理器间898μs左右的时延差异。实验表明TCP下行的切片速率832.148 Mbps,满足异构系统需求。

    关键词：

    拟态; 异构; 切片; TCP;

    Design and implementation of TCP downlink high speed slice serving heterogeneous system

    Zheng Baowen Tan Libo Zhang Junjie Xiong Xuejun

    Key Laboratory of Specialty Fiber Optics and Optical Access Networks, Shanghai University Information Technology Innovation Center of Tianjin Binhai New Area

    Abstract：

    In recent years, the industrial control field is facing an increasingly serious security situation. In order to improve system security, the industrial control processor introduces a Transmission Control Protocol Offload Engine(TOE). The mimetic security TOE requires equal length slicing of the TCP downlink data of the three-way heterogeneous processor, and the downlink data of the three-way heterogeneous processor received by the TOE has a problem of time and frame length difference, which greatly affects the performance of the slicing module. Therefore, this paper proposes a high-speed slicing mechanism for event-driven design, which reduces the delay of the slicing module to the system communication. By using the pointer-based circular linked list buffer to lengthen the slice value, it can match the delay difference of 898 μs between processors. Experiments show that the TCP downlink slicing rate is 832.148 Mbps, which meets the requirements of heterogeneous systems.

    Keyword：

    Mimetic; heterogeneous; slice; TCP;

    为了应对遍布网络的系统漏洞和多变的网络攻击模式[1],邬江兴[2,3,4]提出网络空间拟态防御理论——通过建立动态性、异构性、冗余性拟态防御架构,提高系统安全性。

    文献[5]深入分析了拟态防御的功能与结构机理、动态异构冗余构造机理, 针对拟态防御系统的各类防御机制规律可能会被攻击者窃取的问题, 提出基于正态分布的拟态防御动态调度策略,进一步完善拟态防御理论体系。文献[6,7,8,9]对拟态防御架构的各个功能进行建模分析与评估,进一步验证了拟态防御思想的可行性。

    在此基础上,文献[10]提出了一种基于软件定义网络拟态防御的实现架构,验证了拟态防御架构的入侵容忍能力和可用性。文献[11]从硬件角度出发,针对当前工控系统对未知后门和漏洞缺乏有效防御手段,以拟态防御架构为核心,以 FPGA 硬件为基础,搭建了工控拟态安全处理器原型验证系统,通过黑盒测试、注入测试等测试方法,对该系统的功能和性能进行了测试,证明了硬件实现的该工控系统功能的顽健性、合理性以及系统性能的适用性。

    工控系统为了节约处理器处理TCP/IP协议栈时间,提升系统安全性,引入拟态安全TOE(transmission control protocol offload engine,TOE)的设计。拟态安全TOE系统的TCP下行链路采用异构的处理器作为3个服务器,提供相同的业务,通过拟态判决模块比对3路数据,选择安全可靠的一路数据给用户。TOE保序接收的3路处理器数据经常不是等长的,为了简化判决模块的设计,采用等长帧比较方式,因而需要本文设计的切片模块对3路数据等长切片,而TCP下行链路3路异构处理器之间的数据时延和帧长差异动态变化,导致切片延迟很大,数据链路容易堵塞,通信速率远低于千兆以太网线速传输速度。

    本文提出的高速的切片逻辑设计基于Xilinx VC709开发平台,主要解决3路异构处理器之间的数据延迟和帧长差异问题,对3路处理器数据切等长帧。切片逻辑通过事件驱动模式高速切片,采用指针环形链表缓存等长切片的值,可以匹配处理器时延差异,TCP下行的切片速率832.148 Mbps。

    1 切片模块功能及接口定义

    1.1 切片模块功能

    切片模块并不直接对数据处理,只处理数据帧长。系统通过对3路处理器分配相同帧长,实现等长切片功能。在基于FPGA的拟态安全TOE系统中,切片模块实现等长切片功能的逻辑如图1所示。TOE将接收到的保序以太网数据先将其缓存,同时通过FIFO接口主动向切片模块汇报接收到的保序帧长。切片模块根据帧长汇报FIFO完成帧长分配后,通过FIFO接口反馈回TOE。TOE根据切片模块反馈的分配帧长,从缓存中读取相应长度的数据,转发到拟态判决模块。

    图1 拟态安全TOE系统中切片逻辑   下载原图

    1.2 切片模块接口定义

    基于拟态安全的TCP下行切片模块接口如图2(a)所示,拟态安全TOE系统支持1 024并发链接,但是每个并发链接都需要业务匹配。TOE系统根据收发数据的五元组进行业务匹配,而针对每个异构处理器而言,维护了一个并发序号与并发链接的映射关系。所以在切片逻辑中,为简化系统资源消耗,异构处理器的帧长汇报接口通过异构处理器的并发序号以及USER端匹配的并发序号,进行业务匹配。USER端TOE模块根据TCP建链与断链的消息提供了USER并发序号,便于每个并发链接资源申请与释放,如图2(b)所示。如图2(c)所示,CPUx接收帧长队列的数据帧由USER并发序号、CPUx并发序号和接收的保序帧长信息(CPUx_Req_Len)组成。如图2(d)所示,分配帧长队列数据帧由USE并发序号、CPUx并发序号和分配帧长信息(CPUx_Res_Len)组成。

    2 基于事件驱动的高速切片机制

    对3路异构处理器TCP下行数据进行等长切片最大的难点在于接收的异构处理器TCP下行数据帧长动态变化且各个异构处理器的下行数据到来之间存在时间差异。同时为了减少切片模块对系统通信的延迟,本文采用了基于异构处理器帧长汇报事件驱动的切片机制,也即每当异构处理器接收到新的TCP数据并向切片模块进行帧长汇报时,则根据3个异构处理器当前状态立即启动系统高速切片。除此之外,为了克服异构处理器之间通信性能的差异,本文提出了基于环形链表的结构存储3路异构处理器的等长切片数值,3路异构处理器的下行TCP数据帧长队列分配都依据该链表存储的切片数值。

    本文采用事件驱动模式的高速切片机制,其逻辑结构如图3所示。对处理器保序帧长的切片过程主要分5个步骤完成:保序帧长汇报,切片计算,切片缓存,切片分配和分配帧长汇报。

    图3 基于事件驱动模式的切片机制逻辑结构   下载原图

    2.1 保序帧长汇报

    保序帧长汇报部分负责缓存切片机制的对象,即处理器保序帧长信息。高速切片机制获取保序帧长采用事件驱动模式,减少切片模块对系统通信的延迟,即切片机制采用不断轮询处理器接收帧长队列状态的方式,快速响应处理器的帧长汇报,只要处理器接收帧长队列非空(即FIFO的empty信号无效),即刻解析汇报帧长信息。为应对TOE系统1 024条并发链接的帧长汇报,每路处理器设置保序帧长寄存器直接累加记录各自汇报的帧长,不必等待每次汇报的帧长分配后再接收新的汇报帧长,以避免接收帧长队列堵塞;处理器分配帧长后,切片机制立即更新寄存器累计的待切片帧长值,即减少分配的帧长值。由于切片机制同时只能对一路处理器切片,本文采用随机命中的方式作为3路处理器并行调度策略,使得3路处理器负载均衡。

    2.2 切片计算

    切片计算部分主要根据当前处理器保序帧长,计算当前处理器能够切片的长度,并由切片缓存记录,作为所有处理器等长切片的依据。接收3路处理器数据的帧长差异动态变化,以及系统等长切片要求,造成传统的切片长度的计算需要对3路处理器每次汇报的帧长进行复杂的逻辑判断比较,而本文对所有处理器设置统一的切片长度计算准则,简化判断逻辑,即处理器切片缓存中切片长度只要都已分配,即缓存为空,切片长度即为保序帧长值,并将该保序帧长记录于切片缓存区,为后续帧长分配提供依据;如果非空,则切片长度为0;如果有多路处理器切片缓存为空,则取处理器保序帧长最大值作为切片长度缓存。

    2.3 切片缓存

    文献[12]采用环形链表设计动态缓存采集的信号,每个信号申请一个链表资源,通过移动指针实现数据的连续更新,从而实现了极高的效率和较好的稳定性。

    因为系统等长切片的要求,且3路异构处理器的通信性能差异动态变化,本文借鉴文献[12]高效稳定的环形链表设计,设计如图 4所示基于指针的环形链表,缓存切片机制计算的切片长度,作为所有处理器帧长分配的唯一依据。3路处理器帧长分配的依据只需一个环形链表缓存,相比文献[12]的设计,本文的设计节省了2/3切片缓存资源。

    链表设置尾指针(如图4所示的share_tail),作为写地址为切片长度申请内存,更新链表记录的切片长度信息,链表的头指针(share_head)将3路处理器都已分配的切片长度释放,链表为每路异构处理器设置头指针(cpux_head),切片机制以此每路处理器的头指针作为读地址,使得所有处理器以同一地址获取同一切片长度作为分配帧长的依据,实现等长切片功能。同时链表通过处理器头指针指示每路处理器切片缓存的分配状态,即处理器头指针和链表尾指针重合指示其切片缓存都已分配,为空,否则为非空状态。

    图4 基于指针的环形链表   下载原图

    此外,通过调整链表的深度,即调整链表记录的切片长度数,也即TOE接收缓冲区可以缓存的处理器数据量,切片机制可以灵活匹配接收异构处理器数据的时间差异。

    2.4 切片分配

    切片分配部分主要完成处理器保序帧长的分配。为实现等长切片功能,同时应对处理器保序帧长的差异,切片机制依据链表缓存的切片长度进行帧长分配都。即切片机制通过处理器头指针作为读地址查询链表记录的切片长度,当处理器保序帧长值不小于该切片长度,分配帧长即取该切片长度,同时更新处理器切片缓冲状态及其保序帧长寄存器,而保序帧长小于切片长度时,则分配帧长取0,即此时对处理器不分配帧长,继续等待保序帧长增加。

    2.5 分配帧长汇报

    分配帧长汇报部分主要通过构建分配帧长队列,向上汇报对处理器分配的帧长。

    3 测试与分析

    为了尽可能真实模拟异构系统数据时延和帧长差异,测试切片逻辑等长切片功能与性能,本文使用Xilinx公司的Vivado2016.1和Mentor公司的 Modelsim10.5,搭建仿真平台。

    3.1 事件驱动模式下的切片吞吐量时延分析

    本文介绍的高速切片机制采用基于事件驱动的模式,以减少切片模块逻辑对系统通信的延迟。如图5(a)所示,TOE系统TCP下行数据从CPU端发出到USER端接收,吞吐量时延为61 608 ns。如图5(b)所示,当处理器切片缓存为空时,完成一次切片需要272 ns,仅占系统数据吞吐时延的0.44%;而切片缓存不为空时,因为减少了存储切片长度的时钟消耗,切一次片仅需要216 ns仅占系统数据吞吐时延的0.35%。

    3.2 接收数据帧长差异分析

    本文构建了5种帧长差异不同的场景,测试帧长差异对切片功能的影响,5种测试场景如下。

    场景1(Q1,无帧长差异):3路异构处理器每次传输1 460 Byte数据,传输10次。场景2(Q2,无帧长差异):3路异构处理器每次传输1 Byte数据,传输100次。场景3(Q3,无帧长差异),3路异构处理器每次传输2 000 Byte数据,传输10次;场景4(Q4,处理器帧长差异设置100 Byte),CPU1每次传输100 Byte数据,传输20次;CPU2每次传输200 Byte数据,传输10次;CPU3每次传输300 Byte数据,传输10次。场景5(Q5,随机帧长差异在1～1 460 Byte之间):3路处理器都测试100次。所有测试场景每次数据传输间隔均为300 ns。

    测试结果cpux_dec_total_length如图6(a)～(c)所示,通过Q1、Q2和Q3测试场景分析,在3路异构处理器无帧长差异场景下,切片模块能够对各种类型的数据帧长实现等长切片,包括小数据包,大数据包以及巨型帧。cpux_dec_total_length如图6(d)～(e)所示,通过Q4测试在发送数据量1 500 Byte的采样点、Q5场景下发送数据在54 808 Byte处采样点,切片模块均完成了对3路处理器数据等长切片。所以本文的切片模块能够应对能够应对接收异构处理器不同的帧长差异。

    3.3 接收数据时延差异分析

    为了分析系统能够匹配的接收数据时延差异,本文设置切片缓存链表深度N=64。链表记录的每个切片长度阈值1 460,接收数据缓冲区最大可以缓存64×1 460=93.44 KB数据,此时系统可以匹配的时间差异大小最大为t(t=93.44 KB/v,v表示系统切片速率)。

    此外针对时延差异过大边界场景,系统可以通过改变链表深度N,即通过改变缓冲区缓存接收数据量,灵活匹配延迟差异,增强系统的鲁棒性;对于内存受限的系统,时延差异过大导致出现切片缓存链表满的异常情况时,本文通过超时异常处理机制处理,清空链表,恢复系统正常通信。由于导致系统异常的场景很多,不同的诱因,处理方法有别,限于篇幅,本文不细述。

    为了分析接收数据时延差异对切片模块功能的影响,本文设置4种不同的时延差异场景:T1设置3路处理器发包时延间隔分别为300、400和500 ns;T2设置300、500和700 ns;T3设置300、600和900 ns;T4设置3路数据发包间隔随机,随机时延范围(300～700 ns)。3路处理器发送数据总量均为50 KB,均无帧长差异,测试结果如图7所示。由测试结果cpux_dec_total_length可知,基于环形链表的切片缓存设计能有效匹配异构处理器间不同的时延差异,实现等长切片功能。

    3.4 整体性能测试与应用

    本文通过模拟真实场景,测试切片逻辑在帧长和时延差异共同影响下的系统整体性能,并分析其应用前景。测试设置4种场景加以对比:固定帧长+固定时延(S1)、固定帧长+随机时延(S2)、随机帧长+固定时延(S3)、随机帧长+随机时延(S4),分别统计他们在传输32、64、128、256和512 KB数据量下,切片的速度。固定帧长统一设置1 024,随机帧长设置范围1～1 460,固定时延设置500 ns,随机时延设置范围300～700 ns。每个采样点经过10次实验取平均值得到,S3和S4场景下,发送总的数据量随机,不易控制,通过取最接近采样点的值,10次实验取平均值得到。场景S4模拟的是切片模块实际的整体性能,如图8所示,S4曲线渐渐趋于稳定达到832.148 Mbps,即切片模块整体性能。

    图8 不同场景下切片速率比对   下载原图

    通过4种场景对比分析,在时延和帧长固定场景下,切片速率完全可以达到千兆以太网线速速度。由S1和S3以及S2和S4对比知,帧长差异对切片逻辑性能影响更大些,切片模块在接收数据量时,由于时延和帧长随机,S3和S4在传输小数据量的时候会出现起伏波动,在数据量较大时表现稳定。针对时延差异,切片逻辑可通过缓冲区进行平滑处理时延的差异,由3.3节知,目前系统最大可以匹配93.44×8/832.148=898 μs左右的时延差异,且可通过调整环形链表深度,灵活配置,使得切片速率接近线速速度;而帧长差异,由于拟态安全系统的等长切片功能限制,目前采用选择等待的方式,造成一定的时延,故而切片速率有所下降,针对其他应用场景,可对帧长差异做进一步优化,实现线速传输。

    4 结 论

    本文采用基于事件驱动模式实现高速等长切片功能,减少了切片对系统通信的延迟。同时针对处理器时延差异,本文通过基于指针的环形链表结构存储等长切片值,可以匹配处理器间898 μs左右的时延差异;针对不同的数据帧长差异,切片机制均能实现等长切片功能。最后搭建仿真平台测试,切片模块整体切片速率可达832.148 Mbps,已经接近千兆以太网线速传输速度,能较好地克服异构处理器间通信差异,满足异构系统需求。

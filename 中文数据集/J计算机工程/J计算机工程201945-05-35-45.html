    摘    要：

    基于神威太湖之光和神威蓝光超级计算机的巨量内存故障统计数据, 建立P级超级计算机的内存失效时间模型。采用序列规则挖掘方法, 分析内存失效序列模式, 得到CPU节点上内存失效序列与后续内存失效的关联关系。通过协同分析方法研究并行应用的内存故障与内存失效特征, 结果表明计算-访存-I/O密集型应用对内存故障影响较大, 而应用类型对内存失效的影响有限, 内存失效可能与内存芯片自身的可靠性有关。

    超级计算机; 内存故障; 内存失效; 统计数据; 失效模型; 关联关系; 协同分析;

    Supercomputers Memory Faults Analysis Based on Statistical Data

    LIU Ruitao CHEN Zuoning

    State Key Laboratory of Mathematical Engineering and Advanced Computing National Research Center of Parallel Computer Engineering and Technology

    Abstract：

    Based on the massive amount of statistical data about memory faults on Sunway TaihuLight and Sunway BlueLight supercomputers, the memory failure time model for Petascale supercomputers is built.By sequential rule mining, the sequential pattern of memory failures is analyzed and the correlation relationship between memory failure sequences and the following memory failure on CPU nodes is found.The characteristics of memory faults and failures on parallel applications are studied by the co-analysis method.Results show that computing-memory-I/O intensive applications have large impact on memory faults while the type of applications has limited impact on memory failures, which, however, may have correlation relationship with the reliability of memory chips.

    Keyword：

    supercomputer; memory fault; memory failure; statistical data; failure model; correlation relationship; co-analysis;

    0 概述

    P级超级计算机现已达到数十万处理器规模, 虽然在系统部件设计中采用多种有效措施提高部件可靠性, 但是系统规模和复杂度的增长速度仍然远超系统部件可靠性的改善速度。随着不断增加的系统规模和复杂度, P级计算机面临极其严峻的可靠性与可用性挑战[1,2]。单一的检查点技术已不能完全适应下一代超级计算机系统的可用性要求。以失效预测为基础的轻量级故障预处理技术已逐渐成为构建高可用系统的关键技术之一。此外, 应用与故障特征结合的针对性容错策略也成为容错设计的重要技术途径。

    在P级超级计算机中, 内存是系统的主要部件之一, 通常使用商业化的动态随机存取存储器 (Dynamic Random Access Memory, DRAM) 。内存故障对系统可靠性和可用性影响较大。内存故障特征的发现可提升系统可靠性, 降低容错开销, 为故障预测及应用与故障特征结合的针对性容错策略的制定提供基础支撑, 对系统容错设计具有重要的参考价值。

    针对超级计算机中的故障分析, 研究人员已经开展了许多相关研究。研究人员针对蓝色基因超级计算机中粗粒度故障类型 (例如人工、环境、网络、软件、硬件) 进行故障分析[3], 并研究非严重故障与严重故障之间的关联关系以及网络、应用和I/O故障的时空特征[4], 同时发现了整机故障间隔时间的分布规律[5]。最近, 研究人员分析了泰坦超级计算机的通用图形处理器 (General Graphic Processor, GPU) 软错特征和GPU错误之间的关联[6], 但缺乏对GPU错误的精确时间记录, 只能以24 h作为最大时间精度。研究人员在2台超级计算机上分析了DRAM老化和DRAM提供商对DRAM故障的影响, 发现DRAM故障发生的位置呈现均匀分布[7], 而通过机器学习技术分析DRAM故障, 可有效提升DRAM故障预测性能[8]。

    针对大规模集群系统故障, 研究人员分析了Mercury高性能集群系统的SCSI、NFS、PBS、互连、内存和CPU缓存等故障分布[9], 其中内存故障包括内存单错和内存多错等。研究人员也分析了DRAM可纠错与不可纠错之间的关联, 以及温度等因素对DRAM故障的影响[10]。国内研究人员还利用故障注入研究了内存故障对高可用服务器的影响[11]。

    本文介绍神威系列超级计算机中基于统计数据的故障分析框架。基于大规模统计数据, 从时间上分析内存失效特征, 建立内存失效时间模型, 为内存失效热点及可靠性分析提供理论基础。采用序列模式挖掘方法进行全系统范围内CPU节点上内存失效序列与后续内存失效的关联分析, 为内存失效预测奠定理论和工程基础。基于故障记录和作业与资源使用记录的协同故障分析方法, 分析典型大规模并行应用对神威太湖之光系统内存故障和失效的影响。

    1 相关研究

    1.1 神威系列超级计算机

    神威蓝光计算机系统是1台P级超大规模并行处理计算机 (Massively Parallel Processing, MPP) 系统, 由计算与互连系统、存储系统、维护监控系统、电源系统和冷却系统等组成。计算与互连系统主要由8 704个CPU节点组成。CPU节点采用高性能通用多核处理器申威1600, 片上集成16个同构计算核心, 多路存储控制器与多路系统互连接口, 内存采用DDR3-1066存储器芯片, 组成高效计算单元。

    神威太湖之光计算机系统是1台P级超大规模并行处理计算机系统。该系统包括40 960个CPU。CPU节点采用异构众核处理器申威26010, 配置大容量DDR3-2133存储器芯片。CPU片上集成260个异构核心, 采用计算阵列和分布式共享存储相结合的异构众核体系结构。神威太湖之光超级计算机系统结构如图1所示。

    1.2 基于统计数据的故障分析框架

    超级计算机中越来越复杂的系统故障为深入理解和发现系统故障规律带来了极大挑战。系统中故障具有多样性、瞬时性、不确定性和不可回溯性, 故障的在线采集与存储、分类梳理和多方法故障分析, 成为深入理解故障的基础与重要保证。为便于在超级计算机中进行大规模故障数据分析, 本文设计基于统计数据的故障分析框架, 如图2所示。

    图2 基于统计数据的故障分析框架   下载原图

    该框架由故障相关统计数据采集、数据统一存储、数据规格化、故障特征分析和故障模型发现等5个部分组成。

    1) 故障相关统计数据采集主要由维护诊断系统、资源管理系统、作业管理系统模块组成。维护诊断系统负责全系统在线监控与维护诊断。资源管理系统负责计算资源的按需分配并记录资源使用情况。作业管理系统负责作业的运行管理与控制, 并记录作业的运行情况。例如, 在神威系列超级计算机中采用一套覆盖包括主机计算系统在内的全系统监控与维护诊断基础设施, 采用分布式可扩展架构进行全机主要目标的监控、诊断与维护[12]。

    2) 数据统一存储基于数据仓库技术, 实时存储各类监控目标的RAS事件、作业运行记录和资源使用记录, 建立系统的大规模故障分析统计数据库。

    3) 数据规格化是按照多种故障分析方法的输入数据要求, 进行统计数据的预处理 (主要包括故障分类、故障数据过滤筛选与压缩等) 、故障序列数据生成、应用与资源使用关联等。

    4) 故障特征分析采用多种统计建模、数据挖掘及协同分析方法, 分析故障发生的内在规律和特点。

    5) 故障模型发现通过对故障特征归纳, 总结出各类故障的特征模型, 为系统容错提供理论参考。

    为便于故障分析, 根据故障严重程度和处理方式, 故障又分为非严重故障和严重故障。非严重故障是指不会导致系统失效的非正常状态或可由硬件自行纠错的非正常状态 (例如可纠正的DRAM单错) 。严重故障是指立即会导致系统失效的非正常状态或必须由软件系统进行容错干预的非正常状态 (例如不可纠正的DRAM多错) 。

    内存故障是指包括内存非严重故障 (DRAM单错) 和内存严重故障 (DRAM多错等) 的内存部件故障。内存失效是指包括DRAM多错等的内存严重故障。

    2 内存失效时间模型

    以内存多错类为主的内存严重故障会直接导致内存失效。由于无法通过硬件自身进行纠错, 因此会造成其所在的CPU节点 (包括CPU、内存和设备接口等) 失效。本文从时间上分析神威系列超级计算机的内存失效特征, 建立P级超级计算机内存失效时间模型, 为内存失效热点及可靠性分析奠定基础。

    2.1 系统运行与维护特征

    笔者对神威蓝光和神威太湖之光的日常运行与资源使用模式进行长期监测 (神威蓝光系统2011-12-01—2016-12-01、神威太湖之光系统2016-07-01—2017-10-01) 。从日常作业监控数据统计中显示, 蓝光系统资源使用率达到90%, 太湖之光系统资源使用率达到95%[12]。

    神威蓝光和神威太湖之光的系统日常运行维护有一个共同特点, 即当某个CPU节点连续多次出现严重故障或者运行作业挂起时, 系统管理员将其拔插并与备件机仓中状态良好的CPU节点进行置换, 从而消除应用运行环境的故障热点。根据管理与维护人员经验, 该类CPU节点是故障多发节点, 属于永久故障部件。长期来看, 经过日常的维护与管理, 主机系统的CPU节点基本上都保持较好的运行状态。

    2.2 分析方法

    Exponential分布 (T～E (λ) ) 、Lognormal正态分布 (T～LN (μ, σ2) ) 、Weibull分布 (T～W (m, η) ) 和Gamma分布 (T～Γ (α, λ) ) 是具有代表性的寿命分布[13]。这4种分布的密度函数如表1所示。

    表1 4种分布的密度函数 导出到EXCEL


    分布名称    		密度函数及参数

    Exponential分布    		f (t) =λe-λt, λ>0, t>0

    Lognormal分布    		f (t) =12πσtexp{- (lnt-μ) 22σ2}, t>0, σ>0,
    μ为对数均值, σ为对数标准差

    Weibull分布    		f (t) =mη (tη) m-1exp[- (tη) m], t>0,
    m为形状参数, η为特征寿命

    Gamma分布    		f (t) =λαΓ (α) tα-1e-λt, t>0,
    Γ (·) 是Gamma函数

    采用这些数学模型, 选取神威蓝光5 a (2011-12-01—2016-12-01) 和神威太湖之光3 a (2014-07-01—2017-10-01) 的内存失效统计数据, 对比分析各CPU节点上内存的失效间隔时间特征。采用最大似然估计法对每个分布的参数进行拟合, 使其尽量逼近实际的累积失效分布数据。为检验分布与实际数据的符合度, 使用K-S (Kolmogorov-Smirnov) 检验进行模型评估, 将P值作为模型拟合度评估参数。P值越低, 认为符合度越差;反之, 则越好。一般情况下, P值大于阈值0.05, 认为该分布与实际数据相符合[14,15]。

    2.3 内存失效时间分析

    在神威太湖之光系统和神威蓝光系统上各随机选取4个CPU节点, 进行内存失效拟合分析。根据实际失效数据, 采用最大似然估计法, 计算各寿命分布参数, 并绘制2个超级计算机中随机选取的4个CPU节点的内存失效间隔时间拟合曲线, 如图3所示。从图3可以看出, Lognormal正态分布、Weibull分布与实际CDF内存失效间隔时间分布拟合度相对较好, Gamma分布符合度一般, 而Exponential分布符合度较差。

    图3 随机选择的CPU节点内存失效间隔时间分布   下载原图

    表2是实际失效间隔时间经验分布和根据实际失效数据拟合的各寿命分布情况。使用最大似然估计法对Exponential分布、Lognormal正态分布、Gamma分布和Weibull分布进行参数估计, 同时, K-S检验产生的P值也在表中列出。以神威太湖之光的CPU节点 (3-0-16-3-0) 的内存失效数据为例, 分析如下:Exponential分布的K-S检验p-value=9.115 993e-08, 小于阈值0.05, 与实际失效数据最不符合;Gamma分布的K-S检验的p-value=0.015 038 35, 小于阈值0.05, 与实际失效数据也不符合。Lognormal正态分布和Weibull分布都符合实际失效数据, 其K-S检验的p-value值分别是0.826 657 5和0.763 346 9。其中, Lognormal正态分布符合度最好, 其次是Weibull分布。

    3 内存失效关联分析

    从神威蓝光及太湖之光系统的故障分布情况来看, 内存故障 (包括DRAM单错与DRAM多错等) , 占据了P级超级计算机故障的很大部分 (神威蓝光系统的91%和太湖之光系统的48%) [12]。内存单错一般可通过ECC机制进行自动纠错。若为内存多错类严重故障, 则直接导致系统失效。在内存失效序列与后续内存失效的关联关系上进行分析, 可以为内存失效预测奠定理论和工程基础。

    以神威太湖之光系统为例, 分析3 a (2013-07-01—2016-07-01) 时间区间的内存失效数据, 以CPU为单位进行内存失效数分布统计, 其中发生过内存失效的CPU有23 621个, 占CPU总数 (40 960个) 的57.67%。

    3.1 原理与方法

    序列模式挖掘分析用于发现序列数据集中的序列联系, 产生的序列规则可以描述事物之间的关联或因果关系, 可以使用序列规则来描述内存失效与后续内存失效的关联。假设内存失效与后续内存失效具备关联关系, 那么具备序列规则:

    <内存失效, 内存失效, …, 内存失效>→<内存失效>

    该规则的含义是一个或多个内存失效的顺序发生, 会导致后续内存失效发生。该序列规则可以用于判定内存失效序列与后续内存失效的关联关系。

    使用序列规则形式化表示内存失效序列与后续内存失效的关联, 表示为<X1, X2, …, Xi>→<Y>, 其中, Xi和Y是项, <X1, X2, …, Xi>是按照时间顺序排列的i序列, <Y>是1序列, 若Xi是内存失效, 则<X1, X2, …, Xi>表示内存失效序列, Y是内存失效, i≥I, i∈。

    若∃i∈N, 使得<X1, X2, …, Xi>→<Y>成立, 则说明内存失效序列与内存失效的因果关系, 且对于支持度和置信度具有以下要求:

    {s (<X1, X2, ⋯, Xi>→<Y>) =σ (<X1, X2, ⋯, Xi, Y>) cpu_count≥min_supportc (<X1, X2, ⋯, Xi>→<Y>) =σ (<X1, X2, ⋯, Xi, Y>) σ (<X1, X2, ⋯, Xi>) ≥min_conf

    其中, σ (S) 表示序列S的频繁度计数, 即出现故障序列S的CPU个数, cpu_count是所有的CPU个数, min_support是最小支持度, min_conf是最小置信度。一般地, 若支持度较小, 置信度较大, 则认为序列规则成立。

    本文使用GSP[16]算法对故障数据库中的内存失效进行挖掘分析。数据预处理包括:1) 数据过滤与排重。通过对故障数据库中的其他非DRAM多错类故障进行过滤, 并对同一时刻、同一位置的相同故障排重, 筛选出DRAM多错类故障数据。2) 数据排序。对该数据分别依次以CPU序号和故障发生时间进行排序, 得到DRAM多错类故障的序列数据库。3) 故障类型规约与映射。将各种DRAM多错故障统一映射到MEM_FAIL类型, 并将数据排序中的故障序列数据集进行类型映射后得到规约后的序列数据集, 如表3所示。

    4.1 协同分析方法

    协同分析方法使用了超级计算机的RAS日志、作业日志和资源使用日志以及应用负载特征。RAS日志由系统故障采集框架, 结合系统的各类软硬件传感器来生成。作业日志由作业管理软件生成。资源使用日志由统一资源管理软件生成。应用负载特征分析一般由应用开发人员在深入理解并行应用对系统各类资源使用模式的基础上, 由应用开发人员提供。

    RAS日志包括各类系统故障日志, 通过过滤与数据排重只保留内存故障事件。作业日志与资源使用日志通过作业ID进行关联, 可获取并行应用规模 (CPU节点数量及位置) 、运行时间等信息。协同分析框架如图7所示。

    4.5 讨论

    如上文所述, 内存故障包括内存单错和内存多错等。内存单错由硬件自行进行纠错, 不会导致失效。内存多错等严重的不可纠内存故障会导致内存失效。从图8发现, 计算-访存-I/O密集型应用最易导致内存故障多发, 访存-通信-I/O密集型应用次之, 计算密集型应用对内存故障的影响最小。从图9中发现, 虽然计算-访存-I/O密集型和访存-通信-I/O密集型应用相对于计算密集型应用会造成更多内存失效, 但总体上, 应用类型对内存失效的影响有限。结合结论2, 内存失效的发生与之前发生的内存失效序列有关, 即内存失效的发生可能与内存芯片自身的可靠性特性有关。

    对比分析图8和图9相应的机仓内存故障和失效情况, 发现内存故障多不一定内存失效就多, 内存故障数少不一定内存失效就少。通过序列模式挖掘分析, 得出内存单错不会导致内存多错[12], 因此, 认为内存单错类故障与内存多错类故障是相互独立的2类故障。

    结论3 在P级超级计算机中, 计算-访存-I/O密集型应用最易导致内存故障多发, 访存-通信-I/O密集型应用次之, 计算密集型应用对内存故障的影响最小。在内存单错可纠时, 应用类型对内存失效的影响有限, 内存失效的发生可能与内存芯片自身的可靠性特性有关。

    通常, 单纯计算密集型应用访存负载较小, 计算-访存-I/O密集型应用、访存-通信-I/O密集型应用访存量较大。结论3与经验上认为的各类部件使用频次和负载越大, 越容易发生故障的观点一致。换言之, 使用频率和负载较小的部件, 其故障发生的概率也相对较小。但由于实际中应用类型的限制, 并不能获取各种单一类型应用的故障分布情况。

    此外, ECC纠单错机制对提升系统可靠性非常关键。对内存失效敏感的应用类型, 可结合内存失效间隔时间模型进行检查点优化。在资源分配时, 尽量扩大负载较重的资源类型的配额, 并利用各类不同应用的混搭, 均衡使用各类资源, 从而延长作业运行环境的平均无故障时间。

    5 结束语

    本文介绍了基于统计数据的超级计算机故障分析框架。基于大规模统计数据, 建立P级超级计算机的内存失效时间模型, 即内存失效间隔时间可用于对数正态分布定量描述, 以及超级计算机中的内存失效热点和可靠性分析。采用序列规则挖掘方法发现CPU节点上内存失效序列与后续内存失效的关联关系, 且内存失效序列越长, 后续发生内存失效的可能性越高。该结论可用于内存失效预测。使用协同分析方法分析并行应用的内存故障与内存失效特征, 获知计算-访存-I/O密集型应用对内存故障影响较大, 应用类型对内存失效的影响有限, 内存失效可能与内存芯片自身的可靠性有关。上述结论为超级计算机的内存失效热点分析、内存失效预测及面向应用类型的容错策略提供了理论与工程基础, 是下一代超级计算机容错设计的重要依据。

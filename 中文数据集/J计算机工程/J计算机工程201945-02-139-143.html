    摘    要：

    软件定义网络架构中流表项的无意识性使攻击者可对其进行篡改, 导致网络中出现规则冲突。针对现有规则冲突检测机制检测时间过长的问题, 提出一种快速的规则冲突检测机制。通过压缩流表项, 建立基于端口的规则拓扑, 根据该拓扑计算端到端的可达性, 从而快速检测网络中的规则冲突。仿真结果表明, 在网络拓扑和流表项数量相同的条件下, 相比现有的Netplumber检测机制, 该机制的检测时间可降低约15%。

    软件定义网络; 流表项; 规则冲突; 编码压缩; 冲突检测;

    A Fast SDN Rule Conflict Detection Mechanism

    HAO Wei YI Peng JIANG Yiming

    National Digital Switching System Engineering and Technological Research Center

    Abstract：

    Due to the unconsciousness of the flow entry in the Software Defined Network ( SDN) , an attacker can tamper with the flow entry eventually causing a rule conflict in it. For the detection time of the existing rule conflict detection mechanism is too long, a fast rule conflict detection mechanism is proposed. By compressing the flow entry, the mechanism establishes a port-based rule topology and directly calculates the end-to-end reachability. Based on the topology, rule conflicts in the network can be detected quickly. Simulation results show that under the same condition of network topology and the same number of flow entny, compared with the existing Netplumber detection mechanism, the proposed mechanism can reduce the detection time by about 15%.

    Software Defined Network (SDN) ; flow entry; rule conflict; coding compression; conflict detection;

    0概述

    软件定义网络 (Software Defined Network, SDN) [1]是一种新型的网络体系结构, 可实现网络数据平面和控制平面的解耦合[2]。SDN通过分离控制平面和数据平面以及开放通信协议, 打破了传统网络设备的封闭性。同时, 南北向和东西向的开放接口及可编程性也使得网络管理变得更加简单和灵活。在SDN中, 控制平面通过Openflow[3]协议与数据平面建立通道, Openflow协议将流表项装载到相应设备的转发流表中, 当传输的数据包包头匹配某一流表项时, 将执行相应的动作。SDN这种管控机制大幅提高了网络管理运营的灵活性, 但也给攻击者进行网络攻击提供了便利[4]。目前, 开放网络基金会 (Open Networking Foundation, ONF) 对于北向接口尚未制定相应的标准, 因此, 攻击者可以通过软件编程轻松地对网络发起攻击, 这种方式要比传统网络中侵入防火墙修改安全规则容易得多, 因此, 本文在研究规则冲突时并没有考虑修改安全规则的问题。在SDN网络中, 交换机等网络设备对控制器下发的流规则是完全信任的, 一旦交换机中的流规则遭到了攻击者的恶意篡改, 整个网络的安全性能将面临严重威胁。

    目前用于规则合法性检测的工具普遍存在准确度低、时间消耗过多的问题, 因此, 迫切需要一种时间消耗少、可靠性高的算法来解决规则冲突检测问题。本文提出一种高效的流规则冲突检测算法。利用控制器掌握全局视图的特点, 在控制器中通过对全网规则的匹配域进行编码, 实现对流表的压缩, 并以端口为节点建立端到端网络路径, 从而对全网规则进行快速冲突检测。

    1 相关研究

    攻击者可以通过改写数据包头域, 达到绕过控制器中的安全规则实行恶意攻击的目的。图1所示是一个简单的网络拓扑, 其中包含有3台交换机、1个控制器以及4个客户。定义一条安全规则:客户1不能访问客户3。在图1中, 客户1向交换机发送数据包, 攻击者向交换机中添加Rule1, 对数据包的包头域进行修改, 修改后的原IP地址变为客户4的IP地址。同时, 在另一台交换机中添加Rule2, 将符合匹配关系的数据包目的IP地址改为客户3的IP地址, 使客户1的数据通过交换机最终到达客户3。在这个例子中, 虽然所有的规则都没有违反安全规则, 但是规则之间的依赖性最终导致规则冲突的问题。

    图1 规则冲突示意图   下载原图

    为解决规则冲突问题, 文献[5]针对开源控制器NOX[6]设计出一种安全内核FortNOX, 该架构采用一种基于别名集的验证算法。该算法通过将规则中相关的地址类信息进行编码, 然后与合法规则集进行比较, 从而发现规则冲突的现象。FortNOX算法可以找到一些由流表修改造成的规则冲突, 但在复杂实例中, 这种方法易产生误报、漏报。文献[7]开发了一种新型检测工具Veriflow, 该工具位于控制器和网络设备之间, 通过编码建立trie树来模拟新规则加入后的转发情况, 从而实现网络检测。该方法的检测准确度和可靠性较高, 但时间消耗较大, 难以在实际网络中应用。文献[8]将传统网络中的头空间解析算法[9]应用到SDN网络中, 首先利用头空间解析算法生成授权规则空间和存在重写操作的规则空间, 然后对这2个规则空间进行对比, 从而发现网络的规则冲突情况。由于头空间解析算法是对全网规则进行遍历解析, 因此这种方法在准确度上是无可比拟的, 但是对规则进行遍历存在内存消耗和时间消耗过多的问题。综上所述, 目前用于规则合法性检测的工具普遍存在准确度低、时间消耗过多的问题。本文提出的流规则冲突检测算法通过对流表的压缩, 并以端口为节点建立端到端网络路径, 可快速实现冲突检测。

    2 流表压缩

    在SDN网络中, 流表是交换机的重要组成部分, 是实现数据匹配、查找、转发等一系列操作的基础。与传统路由表相比, SDN中的流表并不仅仅维护几个固定协议, 而是将所有可以将流进行区分的字段都放在其中, 一个流表中通常包含很多流表项, 流表项就代表转发规则。图2所示是一个流表项的标准结构。从图2可以看出, 一个流表项包含3个部分:

    1) 匹配域, 主要用来对数据包进行查找和匹配。

    2) 计数器, 负责统计该流表项进行匹配操作的总次数。

    3) 动作集, 定义了数据包成功匹配后的处理方式, 一般有重写、转发、丢弃等。

    本文主要关注数据在网络中的匹配和相关动作, 可以忽略计数器, 因此将流表项简化为匹配域和动作集。

    图2 流表项结构   下载原图

    匹配域是数据包在交换机中执行相关动作的主要依据。Openflow协议中定义了多种不同的匹配类型, 包括交换机的输入、输出端口, 源、目的IP地址, 以太网源、目的地址等。Openflow v1.0[10]定义了12个匹配域, 在以后的新版本中, 匹配字段的个数也越来越多。本文的检测方法主要针对修改源、目的IP地址的恶意行为, 因此可以简化匹配域为源、目的IP地址。通过二进制转换将十进制的IP地址转换为二进制bit位表示。用Matchfiled来代表匹配域, 例如Matchfiled (001*) ={0011, 0010}, 其中, *代表通配符。使用Action代表动作集, 动作集包含重写操作Modify和输出Out (port) , port表示输出的端口号。

    在SDN网络中, 一个流表中包含多个流表项, 且存在一部分具有相同输出端口并且匹配域之间的汉明距离只有1的流表项, 也就是匹配域之间只有一个bit的差别, 那么这些流表项就可以使用统一的通配符来定义, 从而对流表项进行压缩处理。本文参照文献[11]中的压缩策略, 对流表项进行压缩。首先, 将一个流表分成几个部分, 并对每一条流表项bit位进行排序, 让所有的通配符表示的bit位排到最末端。然后, 对重新排序后的流表项进行合并, 合并后的规则将组成端口内数量最小的流表, 并且这种方法能够应用现有的编程工具实现。最后, 本文对排序后的流表项进行复原, 得到最终统一格式的流表项, 从而大幅减少匹配次数。

    文献[12]提到的检测算法, 没有考虑规则之间、匹配域之间存在的覆盖关系, 因此每次检测都需要按照规则的优先级顺序进行排序, 这种方法增加了管道图的数量, 降低了检测效率。因此, 本文首先按照特定动作对规则进行合并, 在指定动作的前提下, 可以忽略优先级的顺序。在实际检测过程中发现, 产生虚报的概率在可控范围内。同时, 减少流表数目为构建端到端路径降低了复杂度。

    压缩流表项的具体方法如图3所示。为了更加清晰地说明, 将匹配域中的IP地址简化为4位, 图中给出了5条流表项的匹配域IP地址。这5条流表项根据通配符的具体位置可以分成尾端后缀和非尾端后缀两类, 这两类匹配域分别采取不同的操作。对于第1类规则, 不需要对其进行排序操作, 直接进入合并阶段, 对于0、1都满足的比特位, 直接使用通配符表示, 从而实现对规则的压缩。而对于第2类规则, 必须对原有的bit位进行重新排序, 将原规则中的通配符位统一调换到末端, 并对排序后的匹配域进行压缩处理, 在图3中将第2列和第4列进行了对换, 压缩处理后最终得到两条规则。最后, 对存在重排序的匹配域进行恢复, 而压缩后的第2部分, 将第2列和第4列进行转换可以得到原始的排序, 最终得到压缩处理后的规则。可以看出, 采用这种方法后, 原有的5条规则变成3条规则, 达到了压缩流表项的目的。

    3 转发路径图构建

    网络规则冲突检测, 其核心思想是验证网络的可达性[13]。头空间解析算法基于流表项, 这样做无疑增大了转发表规模, 使得验证时间开销加大, 而veriflow等工具使用数据平面转发设备作为拓扑图节点, 对于以流规则为基础的SDN网络, 检测粒度没有达到理想的要求。本文提出一种依靠交换机端口建立网络拓扑的方法, 该方法在保证检测精度的基础上, 提高了检测效率。

    交换机端口之间存在2种关系, 本文称之为内部连接和外部连接。内部连接是指在同一个交换机中, 由流表规则控制;外部连接是指端口处于不同的交换机中, 它们之间通过物理链路进行连接。为了叙述方便, 定义网络拓扑图为一个有向无环图pDAG (V, E) [14], 在这个图中每一个交换机端口为一个节点。E是端点之间的连接, 对于一个交换机的规则r以及下一跳交换机中的规则r', r和r'之间相匹配, (r, r') 是其中的一条边。

    定义RFINAL为每一个点的最终匹配域, 而没有发生变化的匹配域用Rm来表示, 如果发生了重写操作, 可使用RRW来表示重写后的匹配域, 如式 (1) 所示。

    如果RFINAL和R'FINAL相互匹配, 可以用式 (2) 来表示, 那么这2个端点之间就可以建立一条边, 也就是交换机及端口之间的内部连接。

    具体的算法流程如算法1所示, 首先获取任意节点的匹配域作为起点, 并将其中的匹配域转换为一个三元组, 将该三元组与下一条节点中的匹配域进行交运算, 如果交集不是空集, 那么这2个节点之间就可以相连, 将这种算法称为转发路径构建算法。通过重复这一运算, 可以得到所有端到端间的可达性, 最终建立一张节点之间的拓扑图。

    算法复杂度分析:假设有n个端口, 平均每个端口有k个流表项, 根据算法1的描述, 任选一个端口开始遍历, 查找交换机中的流表项进行匹配。

    4 冲突检测

    本文关注的策略冲突是网络实际转发过程中产生的与安全规则相冲突的转发路径, 即控制器下发的每一条流规则都没有直接违反安全规则。但是, 在网络实际运行过程中, 对流表项的修改, 最终导致某些路径绕过安全规则, 产生规则冲突。这种规则冲突必须要通过修改流表项才会发生, 因此, 在进行检测的过程中只要关注那些具有重写操作的流表项, 以及这些流表项产生的转发路径。将这些路径与安全规则进行比较, 最终发现规则冲突。

    5 仿真结果与分析

    为了评估本文检测方法的效果和性能, 实验系统搭建在Xeon E5-2470 CPU和16 GB内存的Ubuntu12.04主机上。为与Netplumber进行对比, 实验网络采用Stanford主干网[15]拓扑, 为了获得更好的实验结果, 在网络中添加了不同数量的规则, 并且采用多次实验取平均的方法, 确保实验数据的准确性。

    6 结束语

    SDN网络的出现极大地提高了网络管理的开放性和动态性, 但是这些特性也给SDN网络带来新的安全威胁。本文针对网络中改写流表项所导致的与安全规则产生冲突这一问题, 提出一种根据流表项特点以交换机端口为节点的路径构建算法, 进而针对网络中的修改规则建立部分网络拓扑。该算法在保证细粒度检测的基础上减少了检测时间。实验结果验证了该算法的正确性和高效性。下一步将考虑在开源控制器上开发基于该算法的APP, 进一步拓宽应用场景。

    摘    要：

    为解决舰船一体化网络系统的测试充分性问题, 设计网络化故障注入平台。基于网络故障注入和网络攻击2个层面, 构建故障影响因子, 形成故障模型库。利用tc、netem和netfilter功能模块和工具进行网络故障注入, 基于tfn2k架构模拟网络攻击, 根据用户配置的故障注入参数向Agent发送负载攻击指令的数据包, 并接收处理Agent定时采集的本机性能检测数据。实例验证结果表明, 该平台能够有效扩展故障模型库, 实现测试数据的自动生成和注入监测, 提高测试充分性和测试效率。

    关键词：

    一体化网络系统; 故障注入; 模型库; 注入机制; 注入平台;

    Design of Fault Injection Platform for Integrated Shipboard Network System

    ZHANG Ya

    Jiangsu Automation Research Institute

    Abstract：

    In order to solve the test adequacy problem of the integrated shipboard network system, a network fault injection platform is designed.Based on network fault injection and network attack, the fault impact factor is constructed to form a fault model library.Network fault injection is simulated with the functional modules and tools including tc, netem, and netfilter.Cyberattacks are simulated based on the tfn2 k architecture.The data packet carrying the attack command is sent to the Agent according to the fault injection parameters configured by the user, and the performance monitoring data of the local device regularly collected by Agent is received and processed.The example verification result shows that the platform can effectively extend the fault model library, realize automatic generation and injection monitoring of test data, and improve test sufficiency and test efficiency.

    0 概述

    随着信息技术和网络技术的发展, 舰船信息化建设得到极大提高。在服务化技术的推动下, 面向服务的一体化网络架构逐渐成为新的应用模式, 其具有灵活性、互用性等分布式架构的特点, 同时能满足已有系统组件和新软件的灵活重组和信息共享需求。文献[1,2,3]分析了国外一体化应用现状和发展趋势, 提出构建我国新一代网络技术应用的建议。文献[4,5]从信息传输网络、平台网络互连、通信管控和集成等出发, 研究国内一体化网络应用需要解决的问题。文献[6,7]从一体化应用需求和目标入手构建不同类型的开放式网络架构并进行验证。上述研究表明, 一体化网络系统的网络通信状态、软件部署状态以及软件技术体制均发生了变化。

    一体化网络的这些变化, 使传统的软件测试技术无法完全满足测试需求, 而故障注入技术能够为复杂网络环境下的网络容错机制的验证提供技术支持。文献[7]从故障管理角度出发, 提出网络服务故障管理系统设计框架。文献[8]研究故障的隔离和恢复, 构建了网络服务故障管理系统模型。文献[9]给出从操作系统故障层面恢复网络服务的方法。文献[10]重点介绍了多层级网络故障管理。文献[11]提出在虚拟化系统层面使用软件故障注入方法, 对虚拟机和管理器实现故障注入, 并且在不同虚拟程度的平台上进行对比总结。文献[12]提出使用一种可靠性模型来测试云环境的可用性, 根据不同场景调用不同函数, 通过注入故障破坏云组件, 记录故障影响并建模。文献[13,14,15]对故障传播和防火墙层的故障注入进行了研究。另外, 也有研究者从故障注入原型框架、注入处理器设计等层面进行研究, 在一定程度上解决了网络测试难题。但是, 基于软件的注入大多针对操作系统级的故障, 且多数只能注入瞬时故障, 体系化的网络故障注入研究还较少。

    舰船一体化网络系统架构的复杂性给软件测试带来了挑战, 如何确保一体化网络系统的安全性和可靠性是当前测试技术研究的重点之一。目前虽然存在一些网络故障诊断和恢复等方面的研究, 但多数偏重网络架构层面安全性设计和故障分析设计。同时, 通过硬件、网口、串口等插拔操作, 或使用现有故障注入工具可以模拟一般常见故障, 但无法精确地验证系统容错措施。为此, 本文提出一种故障注入机制, 并开发了原型机。系统通过交互界面灵活设置故障注入的时间、位置、方式等, 同时实时监控和查询软件运行状态。

    1 故障类型

    1.1 失效机理和影响因子分析

    在一体化网络环境下, 各类应用软件模块封装成独立服务, 通过支撑总线执行调用和管理, 集成到一体化计算环境中实现特定的领域任务。一体化网络系统的管理对象是系统各设备和实体。当某个模块出现故障时, 系统应能正确监测, 并据此调整操控流程, 对意外状态进行容错处理。系统上层业务软件主要通过接收和发送网络报文实现数据交互。网络报文是应用层软件能否实现正常业务流程的关键, 所以系统失效的一个重要因素是网络环境, 其中网络拓扑结构、通信设备的工作机制、设备间的通信协议均会影响系统运行。例如当网络拓扑结构设计不合理时, 会出现网络风暴, 导致系统故障。通信设备内外部环境原因产生传输的数据出错, 此时系统将启动重发机制, 这会导致数据延迟。对于高实时性要求的系统, 数据的延迟会造成严重后果。硬件层面无法过滤的网络问题会逐层传播到应用层, 进而影响应用软件。对于强实时性系统, 需通过技术手段模拟网络通信异常, 验证其安全处理措施的有效性。

    被测软件环境的变化, 意味着影响其可靠性的因素也发生了变化。在一体化网络环境下, 存在着外界环境因素、人为使用因素、底层硬件因素以及网络因素。在这些因素影响下, 被测软件本身完成功能和任务的情况直接反映其可靠性。另外, 除了应用软件本身, 服务支撑系统各种支撑机制和功能的优劣也影响了任务能否顺利执行。当被测软件运行于某一特定环境或接收特定的输入时, 各种故障影响因子会被激活, 若软件未进行容错处理, 将会导致软件发生失效, 如图1所示。

    1.2 网络故障类型分析

    一体化网络系统的分布式处理结构依赖于网络的支持, 可以概括为网络层和应用层2层架构。网络层通过统一的接口支持多类型网络接入, 实现数据、话音和视频等业务的传输。应用层是对各类业务的实现和管控, 是业务实体的载体。网络系统的适配器、激励器、传感器和外部设备均通过统一的接口接入网络。网络负载增加, 丢包、延时等网络拥堵现象成为可能影响系统运行的环境因素。精确模拟各种网络拥堵现象, 是强实时性系统需要解决的一个问题。

    网络报文在集成环境中易受到底层网络环境的影响。网络传输协议及交换报文格式协议故障主要通过截获并解析系统之间的数据包, 对协议中的控制字段、数据字段进行篡改。将一组简单数据进行组合, 定义为元数据, 用以说明数据资源的相关特性。通过约束其取值范围, 实现对接口数据的单因素异常注入, 验证系统容错能力。很多数据之间存在约束关系, 如数值约束、时序约束、结构约束等。特殊的约束关系对应特定的应用领域, 结合约束规则验证系统在交互数据时, 是否违反领域规则。网络通信和业务整合是系统集成的主要任务, 根据一体化网络系统的运行层次, 分别向通信网络、系统、中间件及软件层注入故障。

    网络传输的可靠性由通信设备及网络模块保障。如果网络受到其他因素干扰影响信息传递的及时性, 将会引起严重后果, 所以当系统进行软硬件设计时, 必须考虑网络传输的因素, 进行冗余设计。结合实际应用背景, 分析一体化网络系统所面临的典型故障, 将其分为网络传输故障和网络攻击故障, 如图2所示。网络传输故障是指信息在网络传递过程中的异常, 主要包括数据延时、包丢失、包乱序、包损坏以及包篡改;网络攻击故障是指由通信网络中存在的病毒和攻击导致的网络故障, 包括ICMP攻击、TCP攻击、UDP攻击以及混合轮流攻击等。

    2 网络故障注入技术

    2.1 故障注入平台架构设计

    网络故障注入平台主要包括数据库模块、网络传输故障生成模块、网络攻击报文生成模块、平台故障Agent管理模块4个部分。数据库模块用于完成软件运行状态相关数据的存储, 与用户交互界面进行数据交互来实现运行状态的查询功能。网络传输故障注入模块实现网络信息延时、丢包、乱序、数据包篡改、包损坏的故障注入。网络攻击报文生成模块主要用于生成流向目标主机的DOS类型攻击报文, 包括ICMP攻击、TCP攻击、UDP攻击以及混合轮流攻击。平台故障Agent管理模块主要负责部署在注入机上的故障注入模块与配置在目标主机的Agent之间的通信, 包括故障注入指令的发送、软件运行状态数据的接收, 通过建立Socket通信, 实时获取被测软件在目标主机上运行所占用资源消耗的数据。

    基于Web的用户交互界面采用Apache+MySQL+PHP软件架构来实现, 可通过Web对相关参数进行配置来发起不同类型的故障注入, 实时查看被测软件当前的运行情况, 并可通过与数据库交互以实现不同被测软件在不同故障注入情形下运行状态的查询。

    平台故障Agent包括故障注入模块和运行状态管理模块。运行状态管理模块包括软件运行状态数据采集、实时发送和摘要发送。实时发送的数据包括被测软件在目标主机上运行所占用的线程数、内存、网络上下行流量等资源消耗数据。统计计算资源在设定观测时间内的最小值、最大值、均值等数据, 与故障注入的配置参数合并发送给注入机, 存储到数据库中。故障注入平台模块的隶属关系如图3所示。

    2.2 网络传输故障生成模块设计

    网络故障注入主要利用tc、netem和netfilter功能模块和工具进行模拟。tc是流量控制, 主要用来管控netem的运行方式。通过调用netem各类配置, 模拟带宽受限、传输延迟、数据丢包等各种网络传输性能瓶颈。netfilter可实现安全策略应用中如数据包过滤、数据包处理、地址伪装、透明代理、动态地址转换等功能, 通过配置框架后的中间件方式, 为每种网络协议定义一套钩子函数。钩子函数在数据包传输通过协议栈的5个定义点时被调用, netfilter在每个定义点上截取所需网络包, 对网络包进行规则检查, 决定网络包的后续流向, 进行原数据包向上传输, 或是进行定制修改后重新上传, 或是将该数据包直接丢弃构造丢包状态。

    2.2.1 网络传输故障生成模块

    如图4所示, 网络传输故障生成模块将接口1接收到的数据包通过接口2转发给目标主机, 在转发过程中对数据包进行故障注入操作。流量控制主要基于netem架构来进行配置, 数据包从接口1进来后, 经过流量控制规则判断, 将不满足规则的数据包舍弃处理, 选择满足需求的数据包交由转发模块进行传输, 可模拟延时、丢包、包损坏、包乱序等故障。

    图4 网络传输故障生成模块实现机理   下载原图

    各类故障具体如下:

    1) 数据延时

    数据延时是指数据在网络介质中通过网络协议进行传输时, 由于信道干扰以及设备反应缓慢等原因造成的网络延迟。本文使用tc流量控制对netem运行方式进行管控, 将经过接口2传输的数据包进行截取处理, 在数据流中设置传输时延值和波动值进行延迟的调控, 形成数据延时故障。调用数据流时延命令的格式为[16]#tc qdisc add dev eth1 root netem delay [时延值] [时延波动值];调用停止延迟命令的格式为#tc qdisc del dev eth1 root netem delay [时延值] [时延波动值]。

    2) 包丢失

    数据包在传输过程中, 由于网络链路故障、物理设备失效、带宽受限导致网络拥堵等因素可能出现包丢失。本文使用tc管控netem的运行, 对数据流进行截取处理, 配置不同的丢包率对经过接口2的数据包进行随机丢弃。调用丢包命令的格式为[16]#tc qdisc add dev eth1 root netem loss [丢包率];调用停止丢包命令的格式为#tc qdisc delete dev eth1 root netem loss [丢包率]。

    3) 包乱序

    数据包在网络传输中按照一定的次序进行发送。对经过接口2的数据包截取后, 可以打乱其原有发送次序, 构造包乱序故障。本文将一部分相互关联的数据包即时发送, 其余的数据包进行延时发送。调用包乱序命令的格式为[16]#tc qdisc change dev eth1 root netem delay [延迟] reorder [比例] [相关性];调用停止包乱序命令的格式为#tc qdisc delete dev eth1 root netem delay [延迟] reorder [比例] [相关性]。

    4) 包损坏

    数据包在传输过程中可能发生损坏, 出现包损坏故障。本文使用netem规则中的corrupt来设置包损坏的比例等故障信息, 注入包损坏故障。调用包损坏命令的格式为[16]#tc qdisc add dev eth1 root netem corrupt [损坏比例];调用停止包损坏命令的格式为#tc qdisc delete dev eth1 root netem corrupt [损坏比例]。

    5) 包篡改

    负载篡改类故障由netfilter架构进行模拟, 可利用其包处理功能来改变数据包的字段标志值。netfilter在数据包流过协议栈的5个关键点PRE_ROUTING、LOCAL_IN、POST_ROUTING、LOCAL_OUT、FORWARD时, 调用钩子函数, 勾取传输网络包, 比较规则链, 对网络包进行继续上交、修改、或丢弃处理。

    PRE_ROUTING:数据包首次进入网络层后, 进行版本号与数据校验和的验证。当数据包经过此定义点时, 在此点进行目的地址转换。

    LOCAL_IN:此定义点主要进行传送至本机数据包的过滤处理。将需要转发的数据包直接经过此检测点, 传输至FORWARD检测点, 继续进行定制规则链的包过滤。

    POST_ROUTING:此定义点主要处理即将通过网络层传输的数据包, 根据需求进行数据包内置的源地址变更和地址伪装处理, 过滤处理后继续传输数据包。

    LOCAL_OUT:此定义点主要处理本机进程传输的数据包, 过滤处理OUTPUT包。

    通过编写包处理函数, 将my_hookfn在PRE_ROUTING处完成注册, 转换接收数据包的源地址, 截取客户端的请求指令, 修改正在传输的数据包, 实现网络故障的注入。

    2.2.2 网络攻击报文生成模块

    网络攻击报文生成模块主要集成多种DOS攻击类型, 采用基于tfn2k架构来模拟网络攻击, 将主控端和代理端均设为平台主机实施协同攻击。TFN客户端部署在主控端, 实现对攻击对象的配置和攻击方法的管理。TD程序植入代理端, 接收TFN控制指令, 通过伪造源地址的方式向目标主机发起攻击。通过灵活配置攻击方式 (ICMP攻击、TCP攻击、UDP攻击以及混合轮流攻击) , TFN客户端向TD程序单向发送控制指令, 实施网络攻击。当攻击完成后, 控制主控端关闭两者的单向通信。

    2.2.3 平台故障Agent管理模块

    当用故障注入模块注入故障时, 平台故障Agent管理模块根据用户配置的故障注入参数向运行在目标主机的平台故障Agent发送负载攻击指令的数据包, 攻击指令利用套接字 (Socket) 按指定协议进行封装。该数据包发送至目标主机的平台故障Agent软件, 由该软件对数据包按指定协议格式进行解析得到攻击模式和参数等并发起本机资源消耗类攻击。

    平台故障Agent管理模块的另一个重要功能是接收并处理Agent定时采集的本机性能检测数据。该数据同样通过Socket按指定协议类型进行封装后传输至平台故障Agent管理模块。该功能与故障注入功能应绑定不同的端口从而保证通信独立性。

    实验结果分析如下:

    1) 注入数据延时故障

    验证对象对在一定时间内接收的数据的完整度进行判断, 若接收的数据包不全, 则终止当前业务流程, 给出通信异常提示。该验证对象对数据包延时造成的网络故障具备一定程度的容错能力:当数据包延迟时间较短时, 只会轻微影响进程和任务间的通信;当数据包延时较为严重时, 会使得任务失败, 然后重做任务, 大量增加任务的执行时间。

    2) 注入包丢失故障

    验证对象对包内数据进行判断, 若数据满足结构约束则正常进行业务流程;若丢失的数据为结构判断体, 则重复业务流程直至中断处理。该验证对象对网络故障都有一定的容忍限度:当网络故障未超过限定阈值时, 只是轻微地影响任务的执行时间;当网络故障超过限定阈值时, 将对应的节点视为有问题的节点, 并会引起任务失败, 干扰系统的业务流程。注入包损坏故障与包丢失率较大时的实验结果类似。

    3) 注入包乱序、包重复故障

    验证对象对数据时序进行严格的判断, 未按照正常时序发送的数据包, 全部判断为异常数据, 超时未收到正常数据, 则终止业务流程。对于包重复的数据, 仅接收处理第1条正确数据, 进行业务流程, 后续重复的数据包由于不满足业务流程的需求, 按照异常数据处理。

    4) 网络攻击

    使用原型系统对验证对象进行了TCP攻击、UDP攻击以及混合轮流攻击3类测试。在测试初期, 验证对象网络流量运行正常;启动攻击后, 业务终端的网络吞吐量产生大幅下降, 出现数据丢包和超时异常, 发现验证对象未进行相关的安全性策略设计。验证对象在受到网络攻击后, 出现严重故障, 业务流程无法正常进行。

    5 结束语

    本文提出一种针对一体化网络系统的故障注入方法, 根据舰船一体化网络系统的特征, 构建网络故障、软件故障和硬件故障模型库, 重点针对网络故障和网络攻击2个层面, 分别采用netem和netfilter架构进行流量控制和负载篡改, 基于tfn2k架构模拟网络攻击。在此基础上, 开发原型机并进行实例验证, 结果表明, 故障注入机理在网络故障注入和检测能力方面比较优越, 可以实现测试数据的自动生成和注入监测, 有效提高测试效率。下一步将细化舰船一体化网络系统的架构特征, 增强故障模型的通用性, 完善原型系统, 优化工具的框架结构, 解决故障注入结果的自动监测和报告生成问题。

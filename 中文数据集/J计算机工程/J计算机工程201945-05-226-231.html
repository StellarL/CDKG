    摘    要：

    传统流媒体网络因流媒体数据量较大的特点, 使服务器的负载重、主干网压力大, 从而导致网络吞吐量降低。为此, 提出P2P流媒体资源调度算法。通过引入节点能力度和数据块优先级概念, 构建多层管理流媒体系统模型。在此基础上, 给出多源节点协同调度算法。实验结果表明, 与基于数据块优先级的资源调度算法、先来先服务算法相比, 该算法能有效缩短系统启动时延, 提高播放连续度。

    节点能力度; 资源节点; 数据段优先级; 媒体码率; 贡献因子;

    PAN Xianghui CHEN Yajun WU Xueyi

    Abstract：

    Due to the large amount of streaming media data, the traditional streaming media network has a heavy load on the server and a large pressure on the backbone network, so that the network throughput is low.Therefore, a P2 P streaming media resource scheduling algorithm is proposed.The concept of peer capability and data block priority is introduced to build a multi-layer management streaming media system model.On this basis, a multi-source node collaborative scheduling algorithm is given.Experimental results show that compared with Resource Dispatch Based on Data Priority (RDDP) algorithm and First Come First Served (FCFS) algorithm, the algorithm can effectively reduce the system startup delay and improve the playback continuity.

    0 概述

    随着网络技术的发展, 流媒体系统得到广泛应用。P2P系统节点不仅可以向邻居节点请求缺失的数据块, 也可以上传数据块给邻居节点。请求节点的数据源包括资源服务器和邻居节点, 当邻居节点和服务器都向请求节点提供数据块时, 普通的邻居节点先于服务器提供数据资源, 且能够降低服务器的负载[1]。因此, 如何提高请求节点在邻居节点中的数据分配量以及请求资源的成功率将直接影响服务器的负载和用户节点的播放质量。本文提出一种基于节点能力度和资源优先级的多源协同调度 (MCPR) 算法。通过定义节点的能力度和数据块的优先级, 给出相应的资源调度算法。

    1 相关工作

    P2P技术应用于流媒体系统, 不仅能够降低服务器的负载, 而且可以增强客户节点的服务质量, 如降低响应时延、提高播放连续度等。相对于传统C/S结构, 节点负载不均衡, 单点存在移除的风险, 即结构不稳定。资源 (任务) 调度是流媒体系统的核心问题, 把客户节点的请求提交到合适的服务节点, 其性能由节点的能力和数据块的选择决定。多数系统考虑如何节省服务器的带宽资源, 例如, PPLIve在实施了一系列优化措施后, P2P视频系统的带宽节省率约90%。目前, 国内外研究者提出利用节点的上行带宽降低服务器负载[2,3,4], 如AnySee、PRIME。文献[2]通过结合P2P技术和直播流媒体系统, 构建基于数据驱动的覆盖网络DONet, 以适应实时多媒体数据的传播和分发。文献[5]提出基于先来先服务 (First Come First Served, FCFS) 算法, 即媒体提供者优先满足先到的节点请求, 但没有从提高客户端服务质量的角度进行资源调度。上述研究通过充分利用节点的上行带宽资源降低服务器的负载, 从而提高流媒体直播系统的服务质量。另外一种降低服务器负载的方法是提高系统的稳定性, 如基于网状结构的稳定的流媒体系统[6]。如果客户节点因为某个服务节点失效或没有相应的节点提供服务, 那么把请求转发到服务器。

    上述研究利用节点带宽资源和稳定的系统网络拓扑结构降低服务器负载, 但是没有考虑到客户端节点的服务质量、服务节点的负载均衡、优质服务节点资源调度和数据源节点的性能, 如播放连续度、启动时延等。其中, 影响资源调度的因素包括数据块的稀缺性、紧急性和新鲜性等。关于数据块的优先级, 文献[7]提出基于数据块优先级的资源调度 (Resource Dispatch Based on Data Priority, RDDP) 算法, 但考虑到了数据块的紧急性和稀缺性对优先级的影响, 没给出定量计算, 仅用平衡因子来测量二者的定量关系, 遗漏了时间因素对紧急性定量的影响。文献[8]在评价数据块紧迫性时只考虑播放点和请求数据块的距离, 并没有考虑到分发节点的上行带宽。文献[9]引入能力因子作为量度源节点综合性能的参数, 并给出贡献因子 (Contribution Factor, CF) 及其定量计算, 但该算法仅考虑了系统调度性能的节点, 限制系统的整体调度性能, 从而加大启动延迟和丢包率, 降低了系统的吞吐量。关于P2P流媒体的资源调度问题只从节点能力或数据块紧迫性等因素来优化系统性能, 为此, 本文提出基于节点能力度的P2P流媒体资源调度算法。

    2 节点能力度与数据段优先级

    2.1 节点能力度

    影响节点能力度的因素包括上传带宽、在线时间、相对距离。如果源节点的丢包率较高, 请求节点在等待的时间范围内收不到数据块, 会加重该源节点的负载, 即丢包率高、能力度低。源节点的在线时间Ti越长, 其上传带宽就持续有效, 即服务能力就增强。相对距离Dij越长, 媒体流传输时间越长, 表示该源节点的服务能力越低。节点能力度可表示为:

    {Ci= (Τi×UB2) / (L (i, t) ×Dij×CF (i, t) ) CF (i, t) = (∑Νi∈Ρ∑j∈Rjdij) / (Bt/b)           (1)

    其中, UB表示节点i的上传带宽, Ti表示节点i的在线时间, Dij表示请求节点j和源节点i的相对距离, L (i, t) 表示发送节点i在t时刻的丢包率, Ci表示i节点的能力度, CF (i, t) 表示节点i在时刻t的贡献因子, dij是0-1指示器, 说明源节点Pj是否正确响应请求节点Pi的请求 (0是错误, 1是正确) , P为向源节点Ps发送数据块请求的节点集合, Rj是请求节点向源节点请求的数据块的集合, B是源节点Ps的上传带宽, t是一个调度周期, b是一个数据块的大小。

    分配数据传输任务给能力度大的节点, 会使能力度大的节点总被分配新的任务, 而能力度小的节点却很少被请求上传数据。发送节点的贡献因子越大, 会降低该节点的能力度, 使分配的任务减少, 从而降低负载。由于在计算节点能力度时节点i的上传带宽所占比重较大, 因此上传带宽在式 (1) 中是UB2。

    2.2 数据段优先级

    在调度周期为T的时间内调度任务, 可以视为按照数据段的优先级填满调度窗口Wi[4]。优先级定义影响到请求端媒体播放的连续度、带宽利用率、骨干网负载等, 数据段的优先级计算如式 (2) 和式 (3) 所示。与文献[5]算法相比, 本文提出的基于优先级的数据段调度算法更注重数据段调度的流媒体播放质量等系统性能的影响。其中, 影响性能和优先级的因素有数据的紧迫性U、稀缺性F、新鲜性R, 而这些因素在不同的情况下对系统服务质量的影响程度不同, 因此式 (2) 设计了α、β、θ调节因子, 用来控制紧急性U、稀缺性F、新鲜性R在数据段优先级的计算中所占比重。

    dij=αU+βF+θR s.t.dij∈Di (2)

    Ui= (WT×v) / (idi-idc) s.t.α+β+θ=1 (3)

    其中, dij表示第i个调度周期数据段j的优先级[8], ts为起始时刻, tc为当前时刻, v为播放速率, 可以看出, 数据段的紧急性取决于一个调度周期内调度窗口的大小WT和媒体播放的速率v, 以及请求数据段编号idi和当前正在播放的数据段的编号idc。播放速率v越大, 紧迫性越大。idi-idc的差值越大, 即请求数据段i的编号越靠近当前播放数据段的编号, 请求数据段i的紧迫性越大。参数紧急性U关系到请求节点播放的连续性, 其是重要的性能参数。如果在调度周期T内不能获得请求数据段, 那么请求节点放弃向P2P网络请求, 转向流媒体服务器 (Media Server, MS) 请求。假设请求节点的连续性降低到一定程度, 势必会加重MS的负载, 同时大量的数据段要经过主干网传输, 会加重主干网的负载压力。请求节点播放连续性的控制是系统性能的核心因素, 本文从2个方面保证连续性, 具体描述如下:

    1) 通过多源节点协同调度和客户端缓冲区进行分区, 提高媒体传输的速率。

    2) 提高紧迫数据段的优先级, 即提高式 (2) 中α的值, 增加紧迫性U在数据段优先级中的所占比重。

    在本文算法中, 节点分为管理节点 (Manage Peer, MP) 和资源节点 (Resource Peer, RP) 2种类型, 1个MP管理50～100个RP节, 即一个节点最多有99个邻居节点。在资源节点集中越稀缺的资源优先级越高。资源的稀缺度取决于RP的个数, 以及在一个MP所管理的RP集合中含有资源即数据段i的RP个数。NPIk表示资源节点i中含有数据段k的邻居节点集合, |NPIk|表示集合中元素的个数, |NPI|表示RP节点i的邻居节点个数。如式 (4) 所示, |NPIk|越小, F值越大, 说明资源k越稀缺, 结合式 (1) , 可知该数据段的优先级越高。

    3 多源节点协同调度算法

    3.1 P2P流媒体系统模型

    P2P流媒体系统中的节点类型分为流媒体服务器、资源节点、管理节点、代理服务器 (Agent Server, AS) 、媒体服务器。其中, 流媒体服务器拥有数据文件资源, 管理节点负责收集终端资源节点的信息, 代理服务器负责MP, 并拥有其id、IP等信息, 媒体服务器拥有多媒体数据文件。

    当请求节点Pr进入P2P流媒体系统时, 先向代理服务器提交请求任务。管理节点负责众多的RP, 并拥有RP的id、ip、带宽、缓冲区大小等媒体资源节点信息, RP拥有多媒体资源文件。Pr把请求提交给AS后, AS在其索引文件中查找距离Pr最近的MP, MP拥有RP的描述信息, AS把Pr的请求转发给MP, 因为MP掌握资源节点RP的资源描述信息, 如果RP中拥有Pr请求的资源, 那么把该RP加入到邻居集合中, 即NPi={RP1, RP2, …, RPn}, 表示请求节点Pr已找到请求的资源节点并与其建立了邻居关系。此时, 由相应的MP帮助Pr和NPi中的节点建立连接, 为多源节点数据传输做好准备。如果MP中并没有Pr请求的资源, 那么由AS把请求转发给下一个邻近的MP, 以此类推。P2P流媒体系统模型图1所示。

    3.2 多源协同调度算法

    P2P流媒体系统的核心问题是数据调度, 其影响系统的流畅度和资源利用率等性能指标[10]。数据调度涉及数据段和源节点2个方面问题, 即一个调度周期Ti内调度数据段的集合Di={d1, d2, …, dn}, n≤U, 其中, n为一个调度周期内所能调度的数据段个数, 每段所含字节数相同, 大小均为k。问题1是要解决请求节点的数据段按照优先级的计算排序的问题。问题2是如何把高优先级数据段的请求任务分配给节点能力高的资源节点, 并且实现多个资源节点的协同调度。

    为保证系统有较低的启动时延, 客户端节点开辟数据缓冲区, 并把该缓冲区分为b1、b2、b3这3个部分[11]。其中, b1为调度窗口, 即为本调度周期Ti内所调度的数据集Di, 其大小定义为w;b2为紧急窗口, 即本调度周期Ti内当前缓冲区可以没有, 但必需在下一调度周期Ti+1内调入的数据段集合Di+1, 大小为w/3;b3为预测窗口, 预测将会调入缓冲区但并不紧迫的数据集, 大小为3w。本文把请求节点缓冲区分为3个部分, 并由相应3种源节点集合并行协同提供数据调度, 不仅快速响应请求节点, 而且不影响请求节点的播放质量。

    媒体文件分成若干个数据段, 每段数据有唯一的段号[12,13]。既要考虑到当前调度周期的数据需求, 也要考虑到下一个调度周期的数据需求[14], 即要充分保证媒体播放的连续性。本文提出把客户端缓冲区分区的思想, 即把缓冲区分成buff0、buff1、buff2这3个部分, buff0填充当前调度周期内需要的数据段集, buff1填充一下调度周期的数据段, buff2为预测缓冲区, 存放预测但并不紧迫的数据。buff0数据段个数n=Bd×T/k, Bd是请求节点Pr的下行带宽, T为调度周期, k为一个数据段的大小。如果Pr向单个源节点请求, 那么可知为保证媒体播放的连续性Bdt+k≥vt, 即t≤k/ (v-Bd) , 当Pr向m个源节点请求时, t≤m×k/ (v-Bd/m) 。如果是单个源节点, 那么媒体连续播放的时间t很小, 即较难保证媒体播放的连续性, 因此本文提出多源协同调度算法, 以提高系统播放的连续性。

    图2所示给出多源协同调度模型。buff0对应当前调度周期Ti内的缓冲数据段集Di, 向该缓冲区上传数据的节点存储在队列Q1中。buff1对应于下一个调度周期Ti+1所缓存的数据, 其也维护一个服务节点队列Q2。Buff2维护一个服务区节点队列Q3, 用于预测数据的上传任务。Q1中节点的能力度均大于Q2, Q2的能力度大于Q3, 使能力度大的节点服务于当前紧迫的高优先级的数据, 且各队列各司其职, 从而提高播放的连续度和降低启动时延。因为队列本身是一种FCFS结构, 所以可以利用这种结构的算法特点把高优先级的节点放在队头, 低优先级的放到队尾。如果要降低启动时延又要保证连续度, 那么必须满足如下公式:

    bi×Δt+ (T-Δt) ×r≥ B (6)

    其中, Δt为启动时延, bi为服务节点上传带宽, T是一个调度周期, r是媒体码率, B是buff0缓冲区大小。

    图2 多源协同调度模型   下载原图

    根据式 (6) , 有:

    Δt≥ ( B-T×r) / (b-r) (7)

    式 (7) 反映了节点能力度中的重要参数b上传带宽和启动时延Δt之间的内在约束关系。可以看出, 为降低启动时延, 节点能力度中的参数b越大越好。由式 (1) 中的UB2可以看出, 节点能力度主要取决于上传带宽b, 因此, 在Q1中把能力度最大的节点放在队头, 其余放在队头后面。本文算法是用贪心算法选择出队头服务节点, 用堆排序建立大顶堆。多源节点协同调度算法MCPR具体步骤描述如下:

    步骤1 请求节点Pr提出请求数据段集合{s1, s2, …, sn}。

    步骤2 根据第3.1节P2P流媒体系统模型确定Pr的邻居节点集合 {p1, p2, …, pm}, 初始化队列Q1、Q2、Q3, 按照式 (1) 计算Pr邻居节点的能力度, 并用堆排序建立有序集<p′1, p′2, …, p′m>, 把该有序集分为3部分, 分别入队到Q1、Q2、Q3。

    步骤3 由式 (2) ～式 (5) 计算Pr请求数据段优先级序列<s′1, s′2, …, s′n>, 同一数据段可能拥有多个源节点, 对同一数据段请求分配给能力度较大的节点。每一队列有一作业调度线程, 该线程采用时间片轮转法分配给每个节点一个时间片t, 其优点是均衡化服务节点负载, 本文在式 (1) 中运用贡献因子抑制节点负载, 使用时间片轮转算法, 可使队列中的节点维持较高的节点能力度。

    步骤4Q1中共有m个节点, 由步骤3得到Pr请求数据段优先级序列<s′1, s′2, …, s′n>, 其中把n个数据段分配给Q1中队列前面的n个节点, 并行上传数据。

    步骤5 按照缓冲区的划分, 在第1个调度周期内填充满buff0, 共有m1个节点协同填充满buff0, 该协同调度机制的优点是快速填充满缓冲区, 从而减少客户端节点的启动时延。同时, buff0的滑动窗口移动到buff0的末尾。

    步骤6Pr启动媒体播放, 同时滑动窗口继续向后滑动buff1共有m2个源节点共同完成协同填充, m=m1+m2, 把缓冲区分成2段的优点是可以提高媒体播放的连续性。

    4 实验结果与分析

    4.1 实验配置

    本节验证资源调度算法的优越性, 特别是在节点能力度和数据块优先级方面, 并从平均启动延迟、服务器负载、客户端连续性等方面具体测量[15], 使用文献[5,7]算法进行对比。

    本文系统环境为:操作系统Win10, 处理器AMD双核主频3.30 GHz, 内存8 GB, 网络仿真器采用OPNET。网络层拓扑包括5个核心路由器和50个域路由器, 核心路由器和核心路由器之间、核心路由器和域路由器之间的带宽是20 Gb/s, 域路由器之间带宽是50 Mb/s, 各域中终端节点1 000个, 每个节点的输出带宽为[0 Kb/s, 7 000 Kb/s], 接入带宽为[300 Kb/s, 3 000 Kb/s], 每个域设置1个媒体源服务器, 流媒体速率为500 Kb/s, 数据块大小为1 300 bit。

    4.2 评价指标

    在上述实验环境下, 本文多源节点协同调度算法MCPR与文献[5]提出的FCFS算法、文献[7]提出的数据块分发调度策略RDDP进行比较。本文利用播放连续度作为评价指标, 该指标的定义为客户节点实际获得的数据段的数量和其请求数据段的数量比值。

    4.3 结果分析

    本文实验结果具体分析如下:

    1) 流媒体播放速率的增加对播放连续度的影响。播放连续度反映了媒体数据块传输到缓冲区是否有效, 和实际到达的时间有直接关系, 即媒体的播放有严格的时间限制。如果在最迟播放时间内数据块未到缓冲区, 那么会造成请求数据段播放迟滞, 从而影响系统播放的连续性。本节实验的目的是改变流媒体的播放速率, 本文算法、RDDP算法、FCFS算法的节点平均流媒体播放连续度对比结果如图3所示。

    图3 不同算法连续度和流媒体播放速率的关系   下载原图

    从图3可以看出, 随着播放速率的增加, 本文算法、RDDP算法、FCFC算法的连续度呈现降低趋势, 且开始降低的幅度比较小。FCFS算法的连续度显著下降, 本文算法和RDDP算法缓和下降, 且下降幅度较小。当速率为450 Kb/s时, 本文算法、RDDP算法、FCFS算法的连续度分别是81.5%、78%、57%。实验结果表明, 开始播放速率低, 且没有达到100% (约89.5%) , 说明有一定的丢包率, 即当速率为100 Kb/s时, 有限的缓冲区和下行带宽并不能提供连续的数据供给, 此时再提高播放速率必然会使连续度降低。随着播放速率的持续提高, RDDP算法考虑数据块的优先级, 紧迫的数据被及时调度, 丢包率没有显著提高。本文算法用高优先级的多源节点协同提供数据, 不仅弥补了下行带宽的不足, 而且保证了高优先权的数据被及时调入缓存, 同时连续度降低的幅度要低于RDDP算法。与本文算法、RDDP算法相比, FCFS算法数据供给效率最低, 且未考虑数据块和节点的优先级。

    2) P2P网络节点数目和平均启动时延的关系。图4所示为不同算法的P2P网络节点数目和平均启动时延的关系。可以看出, P2P网络规模较小, 且媒体资源少。客户节点向媒体服务器发送请求。随着客户节点的增多, 请求逐渐增多, 服务器的负载也增加, 因为客户端的请求主要发送到媒体服务器, 所以平均启动时延逐渐增加。当节点数为300时, 本文算法的平均启动时延达到最大值。客户端的请求主要发向流媒体服务器, 本文算法和RDDP算法的平均启动时延接近, 但曲线均低于FCFS算法的平均启动时延曲线。这是因为本文算法和RDDP算法均考虑所请求数据块的优先级, 紧迫的数据被及时请求到, 而FCFS算法并没有考虑到数据段的优先级。当节点数达到300～550时, P2P网络规模有较大提高, 且积累了丰富的媒体资源, 客户端的请求主要转发到P2P网络, 此时3种算法的平均启动时延均有显著下降, 本文算法的曲线明显低于RDDP算法和FCFS算法, RDDP算法的曲线均低于FCFS算法。当节点数为550时, 本文算法平均启动时延优于RDDP算法和FCFS算法。根据节点能力度计算公式, 选优质的服务节点向客户提供服务, RDDP算法和FCFS算法把请求转发到P2P网络, 丢包率有一定程度的缓解, 从而降低启动时延, 但均未考虑到如何从服务节点选择的角度降低启动时延, 降低丢包率。当节点数达到550～800时, 在系统的平均启动时延方面, 3种算法均有略微提高, 这是因为随着P2P网络规模的扩大, 请求节点的数量也增多, 即P2P网络的负载, 特别是上行带宽的压力逐渐增大, 提高了系统的平均启动时延。本文算法考虑到服务节点的均衡化问题, 利用式 (1) 中贡献因子去抑制负载, 以控制负载均衡。RDDP算法考虑到数据块的紧迫性和稀缺程度, 丢包率和启动时延均低于FCFS算法。

    3) P2P网络节点平均上行带宽和系统丢包率的关系。图5给出不同算法P2P网络节点平均上行带宽和系统丢包率的关系。可以看出, 当P2P节点数在150以内时, 本文算法保持较低的丢包率。客户端的请求发向媒体服务器, 媒体服务器具有丰富的媒体资源来响应客户端的请求。当节点数在150～300时, 丢包率略有下降。当节点数在300～600之间时, RDDP算法和FCFS算法丢包率显著增加, RDDP算法从24%上升到43.5%, FCFS算法从25%上升到58.5%, 此时, P2P网络集聚了较为丰富的热门媒体资源, 客户端的请求由P2P网络响应, 但相对于媒体服务器的带宽和处理能力, 服务节点的能力显然不足, 因此丢包率呈现上升趋势。本文算法运用多源节点协同调度思想, 充分挖掘P2P网络的潜力, 相对于RDDP算法和FCFS算法, 丢包率并没有呈现显著的增加幅度, 验证了P2P网络多源协同调度在提高系统平均上传带宽方面具有显著而有效的作用。当节点数从600上升至800时, 3种算法的丢包率均有不同程度的下降, 本文算法的下降趋势尤为显著, RDDP算法因为考虑到了数据块的优先级, 下降趋势也较为显著, FCFS算法下降趋势最弱。

    5 结束语

    现有的P2P流媒体系统在数据资源调度和系统管理方面强调节点或数据的优先级, 忽略了多节点间的协同, 导致媒体播放连续度低、启动时延大。为此, 本文提出基于节点能力度的P2P流媒体资源调度算法。根据数据段和节点的特点, 定义数据段和节点的优先级计算公式, 并构建多源协同调度模型, 以提高数据的传输效率。实验结果验证了算法的有效性。下一步将研究P2P流媒体系统在复杂环境下的服务质量及在不同自治域节点管理模型的稳定性。

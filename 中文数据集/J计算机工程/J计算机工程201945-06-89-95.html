    摘    要：

    在移动应用的开发周期中, 使用图形用户界面 (GUI) 自动化测试技术可提高APP的开发测试效率。针对基于录制回放的GUI自动化测试方法存在不适用黑盒测试场景或识别正确率较低等不足, 提出基于图像文字识别和应用页面布局的控件识别算法。根据移动应用的页面布局结构和手势操作坐标精确识别控件, 采用控件树遍历的方式解决遍历过程中的页面布局重叠问题。分别在Android和iOS 2个平台进行实验, 结果表明, 与基于坐标和源代码的控件识别算法相比, 该控件识别算法具有更高的正确率。

    图形用户界面; 自动化测试; 控件识别; 文字识别; 页面布局;

    APP Control Recognition Algorithm Based on Text Recognition and Page Layout

    DING Shiju GU Naijie HUANG Zhangjin HOU Jin

    School of Computer Science and Technology, University of Science and Technology of China Anhui Province Key Laboratory of Computing and Communication Software

    Abstract：

    In the development cycle of mobile applications, the use of Graphical User Interface (GUI) automated testing technology can improve the efficiency of the development and test of APP.Still, the GUI automated testing method based on recording playback is not applicable to black box test scenarios, and its recognition accuracy rate is not high.To deal with the advantages, this paper proposes a control recognition algorithm based on text recognition in images and application page layout.Based on the page layout structure of mobile applications and the accurate recognition control for the coordinates of gesture operation, the overlapping problem of page layout in the process of traversing is solved by traversing the control tree.Experiments are carried out on Android and iOS platforms respectively, and results show that the control recognition algorithm has higher accuracy than the control recognition algorithm based on coordinates and source codes.

    Keyword：

    Graphical User Interface (GUI) ; automated testing; control recognition; text recognition; page layout;

    0 概述

    随着经济的增长和移动互联网的高速发展, 移动客户端日渐普及, 移动应用数量呈爆炸式增长。在移动应用产品发布之前, 往往需要进行大量重复的测试, 为了提升测试的效率, 在图形用户界面 (Graphical User Interface, GUI) 的测试过程中[1], 自动化测试被广泛使用。

    已有的GUI自动化测试方法主要是通过录制回放的方式, 该方法记录用户与被测应用的交互并转化成脚本, 然后执行脚本回放交互的内容。其中, 最重要的是识别方法, 即自动识别出交互操作对应的可视化组件。常见的控件识别方法主要有基于坐标的控件识别方法、基于源代码的控件识别控件与基于控件树遍历的控件识别方法。这些识别方法都存在各自的不足, 如对跨设备测试支持不够、不适用黑盒测试场景或识别正确率较低。

    针对上述控件识别方法存在的不足, 本文提出一种基于图像文字识别和应用页面布局的控件识别方法。该方法根据手势操作的坐标和移动应用的页面布局来唯一确定控件, 无需修改源代码, 且支持跨设备录放。

    1 相关工作

    近年来, 在Android和iOS基于录制回放的自动化测试领域中, 研究人员提出了多种自动识别控件的方法, 主要分为基于坐标的控件识别方法、基于源代码的控件识别方法、基于控件树遍历的控件识别方法3种类型。

    1.1 基于坐标的控件识别方法

    基于坐标的控件识别方法通过记录坐标的位置进行录制回放。例如自动化测试工具Jacareto[2]、MonkeyRunner[3]或Pounder[4], 它们在录制时记录鼠标点击坐标和键盘事件, 在回放中使用捕获的信息创建新的触屏和键盘事件。阿里云测[5]是市场上广泛使用的测试平台, 支持Android和iOS的黑盒测试, 包括采用录制回放方式的功能测试, 黑盒测试是在不考虑代码结构的前提下, 根据需求和运行环境对应用程序进行测试[6]。其中, 阿里云测提供了按控件录制和按坐标录制2种方式, 但其官网中说明按控件录制会存在无法识别的情况。而按坐标录制就是通过手势操作在屏幕中的绝对坐标代替控件属性作为脚本参数。

    这种基于坐标的控件识别方法简单快捷, 但在不同分别率的手机中控件的缩放比例会发生变化, 导致控件识别不准。因此, 其无法进行跨设备测试, 脚本的复用率低, 并且不支持 GUI 控件层次的断言。

    1.2 基于源代码的控件识别方法

    文献[7]提出的基于录制回放的Android自动化测试方法, 通过对Android应用的源代码进行插桩, 在页面布局的层级结构间加入一个模拟布局。该模拟布局覆盖Android UI事件处理器, 如onTouchEvent () 和onKeyDown () 方法以及ViewGroup的监听器, 在事件传递过程中进行拦截, 然后保存事件对应的控件属性并生成Robotium[8]脚本, 在回放中通过控件属性识别控件并执行操作。这种控件识别方法由于需要对程序源代码进行修改, 在GUI黑盒测试的场景中并不适用。

    XCTest[9]是Xcode上自带的自动化测试框架, 在Xcode7之后可以支持UI测试。该框架提供一种基于控件的录制回放测试方法, 在该方法中, 录制进程随待测应用一起运行, 监听用户对手机的操作, 自动生成测试脚本, 识别到操作的控件后使用属性name或text标识控件。但是对于一些非标准控件会识别错误, 如属性内容为空。另外, 该框架会将中文名称的控件转换成unicode编码, 但其中大小写会有问题, 需手动修改。XCTest也需要APP的源代码, 不适用黑盒测试环境。

    1.3 基于控件树遍历的控件识别方法

    APP的GUI控件树是指应用程序界面的树形控件布局结构。UIAutomatorViewer[10]是Android SDK中自带的一个GUI工具, 用来扫描和分析Android应用程序的UI控件。其使用adb工具连接手机查看Android应用的控件树。同时, 该工具同步映射手机当前的界面至PC端, 当鼠标移动至映射的界面上时, 会根据当前坐标在控件树中遍历进行识别控件, 并标出控件的范围。

    Appium[11]测试框架在OSX平台的客户端中有一个Inspector功能, 该功能提供了一种iOS录制回放测试方法。与UIAutomatorViewer类似, 该方法将手机屏幕映射至MAC端, 监听电脑端鼠键操作, 根据坐标在控件树中遍历识别控件并生成测试脚本。

    但在使用UIAutomatorViewer和Appium的过程中发现, 页面布局重叠会导致识别控件的错误率较高。本文的主要工作是在基于控件树遍历的识别方法基础上解决页面布局重叠问题, 为此, 提出一种基于文字识别和页面布局的控件识别方法。该方法根据手势操作的坐标和移动应用的页面布局来唯一确定控件, 且无需修改源代码, 并且支持跨设备录放。相较于基于源代码的识别方法, 可以适用在黑盒测试场景。相较于基于坐标的控件识别方法, 基于控件树遍历识别到控件后, 可以获得控件的各个属性, 在回放过程中依据控件属性或控件路径定位可以支持跨设备测试, 同时还能进行UI控件层次的断言验证。

    2 APP页面布局结构

    上文分析了已有的控件识别方法的不足及原因。根据本文提出的基于文字识别和页面布局的控件识别方法, 本节介绍APP的页面布局结构。经过研究发现, iOS应用界面结构与Android应用类似[12], 而2个平台中应用的页面布局重叠问题本质上也相同, 以Android应用为例能够说明问题, 因此, 为了节省篇幅, 本文的分析工作将以Android应用为基础。

    2.1 控件树

    在Android中, 用户界面中的每个控件都是视图 (View) 类的子类, View占据屏幕上的矩形区域并负责绘制。Android提供了一组预构建View, 可用于构建用户界面, 如Button、CheckBox和TextView等。 用户界面是由具有不同层级的View组合成的一个控件树, 控件树中的根结点是一种Layout View, Layout (布局) 是ViewGroup的子类, 用于控制子视图在屏幕上的位置。View及其子类一起负责Android的界面展示和事件处理[13]。图1为使用工具UIAutomatorViewer查看Android应用的控件树示例。

    图1 UIAutomatorViewer控件树示例   下载原图

    2.2 页面布局重叠

    如图1所示, 在PC端上使用UIAutomatorViewer查看Android应用的控件树时, 每个结点都具有一个bounds属性代表它在手机界面中的矩形范围, 其值形如[x1, y1][x2, y2], 其中, [x1, y1]代表控件的左上角坐标, [x2, y2]代表控件的右下角坐标, 2个值可以唯一确定控件在手机界面上的范围。这里定义页面布局重叠是指控件树中不同结点间的范围相互有交叉重叠。

    对于UIAutomatorViewer中由鼠标位置识别控件的功能, 分析其源码可知, 其识别原理是依据鼠标的位置和控件范围的值作为判断条件在控件树中进行深度优先遍历搜索控件。在遍历过程中, 判断是否进入当前结点继续进行遍历的条件是坐标是否在该结点范围内, 这里定义其为遍历搜索条件。当遍历到叶子结点时, 保存叶子结点的范围大小, 最终识别的控件是遍历搜索条件为真的所有叶子结点中范围面积最小的一个。

    在实际的识别过程中发现, UIAutomatorViewer的识别错误率较高, 其原因在于页面布局重叠会导致最终识别的控件并非界面上鼠标位置显示的控件, 而是未显示在界面上的其他页面的控件或者是当前页面上相邻的控件。图2是页面布局重叠导致的一个识别错误情况示例。图中箭头指向鼠标的位置, 矩形框是识别出的控件的范围, 界面右侧部分是识别控件在控件树中的位置和各属性值。从图2可以看出, 箭头对应的控件是“确认选股”按钮, 而识别出的控件是“交易”按钮, 查看控件树可以发现, 该控件是另外一个页面中的控件。

    3 控件识别算法Finter

    上文介绍了APP的控件树并分析UIAutomatorViewer识别控件错误率高的原因, 页面布局重叠阻碍了该控件识别方法应用于实际自动化测试。在此基础上, 本文提出一个新的识别算法Finter用以解决重叠问题。

    3.1 控件树抽象

    为更好地描述页面布局重叠问题, 以图形化的形式介绍APP页面布局对应的控件树。图3是一个控件树的图形化示例。在图3中只画出了能够说明问题的部分结点。

    图3 控件树图形化示例   下载原图

    为进一步分析问题, 现对控件树中的结点作如下定义。

    定义1 (页面结点) 控件树中包含至少一个完整页面的结点。这里的完整页面是指该结点的矩形范围与手机屏幕尺寸大小一致, 如图3中编号1～编号6的结点。页面结点中可以嵌套页面结点, 控件树中的页面结点集合用大写字母P表示。

    定义2 (控件结点) 控件树中结点范围小于手机屏幕尺寸的结点。在Android应用中其又可以分为代表Layout类和代表View类的结点, 如图3中编号7～编号17的结点。控件结点也可以嵌套控件结点, 控件树中的控件结点集合用大写字母W表示。

    定义3 (控件属性) 用来表示不同控件具体特征的变量。对于控件树中的任意一个结点t, 具有属性集合{parent, childrens, text, class, bounds, rect}, 其中, t.parent、t.childrens表示结点t的父结点和子结点, t.text表示结点t的文字内容, 如果没有文字其值为空, t.class是结点t代表的控件的类型, 如TextView、ImageView等, t.bounds表示结点t的矩形范围, 其值为x、y、width、height的组合, 表示结点t矩形范围的位置和长宽。此外, 省略了一些未用到的属性, 并增加一个rect属性用来表示结点布局范围的抽象表示, 代表在界面中的一个矩形区域内的点集。

    3.2 重叠情况分析

    本节在分析大量APP页面结构的基础上, 对页面布局重叠情况进行了整理。显然, 当2个结点的rect交集不为空时即代表其相互重叠。此外, 在控件树中, 子结点布局范围是父结点布局范围的子集。因此, 可得定理1和定理2。

    定理1 在控件树中, 结点s和t重叠, 当且仅当s.rect∩t.rect≠Ø。

    定理2 对控件树中∀t, 如果t.parent≠null, 则有t.rect⊆t.parent.rect。

    由定理1和定理2可知, 当一个结点t和其兄弟结点的子结点布局重叠时, 必然也和其兄弟结点重叠。因此, 在解决重叠问题时可将问题简化为解决兄弟结点间的重叠, 以下分析中的重叠均指兄弟结点间的重叠。页面布局重叠可以是相同结点类型自身间的重叠, 也可以是不同结点类型间的重叠。因此, 按照3.1节中的定义可将重叠情况按结点类型分为3类:页面结点与页面结点重叠, 页面结点与控件结点重叠, 控件结点与控件结点重叠。

    在对APP页面结构的研究中发现, 一些页面具有特殊属性, 如:页面局部透明, 页面结点为空页面。此处对页面的特殊情况进行整理并在下文的算法设计中将其考虑在内。

    特殊情况1 页面结点之间重叠, 其中一个页面为部分透明, 这是由于该页面是以弹出框形式出现, 其子结点是一个控件结点, 没有覆盖整个手机屏幕, 且在控件树中没有属性可以标识这种情况。图4是特殊情况的示例, 图4 (a) 为特殊情况控件树的一部分, 其中结点1和结点2都是页面结点。图4 (b) 为特殊情况界面, 当页面2的内容TextView6显示在界面上时仍能够看到页面1的内容。

    3.3 控件识别算法

    本文在UIAutomatorViewer算法基础上对页面布局重叠问题进行优化, 设计了控件识别算法。算法设计中需要使用文字识别技术, 即利用计算机自动识别图像中的文字, 在算法实现中采用光学字符识别 (Optical Character Recognition, OCR) 技术。

    在对控件树进行深度优先遍历的过程中会出现一个情况。在选择当前结点的子结点作为递归遍历的结点时, 存在多个满足遍历搜索条件的结点, 即兄弟结点间页面布局重叠且输入的坐标在重叠的范围内。针对该情况, 本文提出一种基于文字识别的解决方案。假设当前结点为m, 在m的子结点中满足遍历搜索条件的结点为r、s、t 3个结点。首先求出该3个结点的重叠区域rect, 然后在r、s、t的子孙结点中分别找一个结点p, 满足如下2个条件:1) p.rect∩rect≠Ø;2) p.text≠null。然后根据p.rect的值在手机界面图像对应的区域进行文字识别, 若识别结果与p.text相同, 则p所在子树的根结点即为下一层进行递归的结点。

    基于上述分析, 本文设计了控件识别算法Finter, 该算法以递归的方式进行深度优先遍历。overlapnodes每次递归时初始为空集, 保存相互重叠的结点。choose保存子结点中继续进行遍历的结点, 初始为null。在每次遍历到一个结点时, 首先将包含point的所有子结点加入到overlapnodes中 (第5行～第11行) , 如果为空页面则跳过 (第7行、第8行) 。当overlapnodes中的结点不止一个时, 若均为控件结点且无文字则返回面积最小的结点 (第 13行、第14行) , 否则就使用文字识别算法判断并更新choose (第15行～第23行) , overlaps保存重叠结点间的相交范围, GETOVERLAPS函数为计算重叠范围 (第15行) , 具体见算法1。如果当前结点的子结点中没有包含point的结点, 则当前结点作为结果返回, 否则在choose所在的子树继续进行遍历 (第24行～第27行) 。

    算法1 Finter算法

    针对上3.2节的3种特殊情况, 对算法进行特殊处理。对于特殊情况1, 在计算重叠结点间的重叠区域时, 选择各重叠结点的子结点互相计算重叠区域, 重叠区域计算见于算法2。对于特殊情况2, 当重叠结点中的页面结点为空结点时, 遍历过程中忽略该结点。当特殊情况3发生时, 选择面积较小的结点, 这样选择的原因是考虑到实际情况中假设2个结点均可点击, 只有面积小的结点在面积较大的结点上方才能保证都可以被点击到。

    3.4 算法优化

    在算法中, 图像文字识别技术的准确率决定了识别控件的正确率。现有的OCR技术如TH-OCR (清华OCR) [14]、ABBYY Finereader[15]、谷歌的Tesseract-OCR[16]等的识别率已经达到比较理想的效果, 但仍会有识别错误的情况。因此, 为了提高文字识别的正确率, 本文对Finter算法进行了优化。

    1) 图像放大

    使用OpenCV函数库对手机图像进行放大处理, 然后在放大后的图片中进行图像文字识别可以增大识别率, 在放大图片的同时对识别的区域范围也同比放大。

    2) 包含判断

    使用包含判断的方式来判断文字识别结果。由于手机界面的内容比较丰富, 在一块区域内除去文字内容可能会存在其他图案, 这导致识别的结果除了文字内容还有乱码, 因此, 在判断识别结果时不进行直接比较, 而是看文字识别的结果是否包含已知文字。

    3) 正确率判断

    由于不能保证百分之百文字识别准确率, 因此设置正确率判断。当使用文字识别算法对多个重叠结点文字识别时, 若均未完全正确, 选择识别率高的结点作为识别正确的结果。同时, 为识别率增加一个下限, 当低于这个下限时, 直接认为其识别错误。如当识别内容为2个中文汉字时, 认为必须全部识别正确, 对于3个中文汉字必须至少识别正确2个汉字, 对于4个及以上的汉字识别正确率不能低于75%。

    4 算法实现

    本文将上文提出的控件识别方法针对Android应用、iOS应用分别进行实现。同时使用控件树中的路径作为控件标识, 并基于开源框架Appium进行测试。在实现过程中使用了一些优化方法以提高该方法的可用性。

    4.1 控件路径

    控件路径是一个控件从GUI 控件树根节点到自身的绝对路径。而控件树在计算机中是以XML格式保存的, 因此控件路径以XPath格式作为描述形式。XPath是XML路径语言, 它是一种用来表示 XML 文档中某部分位置的语言。使用XPath作为定位控件的方式具有唯一性, 可以精确定位控件。

    开源测试框架Appium支持Android、iOS 2个平台的移动应用自动化测试, 它可以使用XPath进行控件定位。同时支持Webview页面, 将Webview的页面结构也加入到控件树中。在算法实现中, 将控件识别的结果以XPath形式保存, 并生成基于Appium的脚本用来测试控件识别的结果是否正确。

    4.2 缓存识别结果

    由于对图像进行文字识别是一个耗时的操作, 在算法中尽量减少文字识别的次数。如果在同一个页面中每次识别控件都要进行文字识别将会严重影响算法的性能。本文采用一种缓存文字识别结果的方式减少文字识别次数, 以图3为例, 结点2和结点4是2个重叠的完整页面, 假设页面2为当前手机显示的界面, 当定位页面2内的控件时, 识别算法会每次都在结点2和结点4间进行重叠判断和文字识别。针对这种情况, 本文采用2种方式缓存识别结果:一是当进行一次重叠判断后, 保存当前控件树和重叠结点, 下一次遍历到该控件树和之前判断过的重叠情况后直接选择上一次的结果;二是每次对控件进行文字识别后, 保存该结点的文字识别结果, 如果其他重叠情况发生时需要对该控件进行文字识别, 可以直接读取结果。这2个方法都需要保存控件树作为缓存结果, 因为不同手机界面对应的控件树也不同。

    5 实验结果与分析

    5.1 实验环境

    本文实验分别针对Android系统手机和iOS系统手机进行测试, 主要测试目标为控件识别的正确率。Android测试环境如下:搭载Android4.4系统分辨率为720像素×1 280像素的小米4和搭载Android7.0系统分辨率为1 080像素×1 920像素的荣耀8;iOS测试环境为一部搭载iOS10系统的iPhone5s和一部搭载iOS11系统的iPhone6。

    5.2 识别正确率实验

    实验在Android、iOS 2个平台分别选取4个应用, 每个应用选取5个页面。由于在APP中只有少数页面存在布局重叠, 因此为了体现实验的对比效果, 只选择具有布局重叠的页面进行测试。每个页面等距离划分为11×6的网格, 对其中50个交点坐标进行控件识别, 识别次数共1 000次, 结果取识别正确次数所占的百分比。判断识别是否正确的标准是将控件的XPath结果用Appium框架编写脚本, 当脚本运行的结果和测试人员操作手机的结果一致时, 则表示控件识别正确。将Finter算法在Android平台和iOS平台的识别结果分别与UIAutomatorViewer和Appium的识别结果做对比, Android应用的对比实验数据如图5所示, iOS应用的对比数据如图6所示。

    UIAutomatorViewer是在所有页面中找到一个面积最小的控件, 因此, 即使存在页面重叠, 在很多情况下也能识别到对应控件。而Appium存在重叠情况时识别效果会非常差, 查看Appium Inspector的识别算法可知其识别过程是默认不存在重叠, 即在遍历控件树时返回第1个搜索到的包含当前坐标的控件, 因此, 其识别正确率较低。

    5.3 跨设备对比实验

    除了针对控件识别的正确率实验, 本文将提出的控件识别方法与其他控件识别方法也做了对比实验。根据相关工作的分析, 基于源代码的控件识别方法理论上不会存在方法的错误, 前提是在白盒测试环境下且APP的开发符合测试需求规范, 即每个控件都有一个唯一的name或label, 可以唯一标识控件的属性。

    但是很多测试场景是在黑盒测试环境下进行而且很难满足测试人员对开发规范的需求。因此, 本文与同样在黑盒测试环境下基于坐标的控件识别方法进行对比实验, 针对基于坐标的识别方法存在跨设备不足的问题进行对比测试, 证明本文方法可以有效支持跨设备自动化测试。

    6 结束语

    本文提出一种新的APP控件识别方法。该方法采用控件树遍历的方式, 为解决遍历过程中的页面布局重叠问题。实验结果表明, 与基于坐标和源代码的控件识别方法相比, 本文方法可以在黑盒测试环境下精确识别控件, 并允许跨设备录制和回放, 增加了脚本的复用率, 同时支持控件级别的断言形式。但本文方法受限于文字识别的正确率, 下一步将针对OCR技术进行算法优化, 在控件识别过程中对OCR进行针对性的机器学习训练, 用以提升识别的准确率。

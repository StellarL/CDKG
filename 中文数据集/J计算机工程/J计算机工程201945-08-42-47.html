    摘    要：

    为满足基于Docker容器的应用在Kubernetes集群和Docker Swarm集群之间的迁移需求, 研究A-Migrator异构容器云应用迁移系统。通过Kubernetes和Docker Swarm的应用编排调度策略, 给出两者之间编排信息转换的可行方法, 并提出基于镜像预同步的应用迁移技术。实验结果表明, A-Migrator异构容器云应用迁移系统可实现基于Docker容器的应用在2个集群之间的迁移, 且引入镜像预同步技术后应用迁移时间平均减少60.33%。

    关键词：

    Swarm集群; Kubernetes集群; 异构容器云; 调度策略; 迁移技术;

    Application Migration Method for Heterogeneous Container Cloud

    YANG Kaiqi YAO Pei ZHAO Yulong TANG Lingtao

    Institute of Jiangnan Computing Technology

    Abstract：

    To meet the migrating needs of applications based on the Docker container between Kubernetes clusters and Docker Swarm clusters, an application migration system for A-Migrator heterogeneous container cloud is developed.Through the application orchestration and scheduling strategies of Kubernetes and Docker Swarm, the paper gives a feasible method of converting orchestration information between the two and the technology of application migration based on pre-synchronization of images.Experimental results show that the application migration method for A-Migrator heterogeneous container cloud can migrate applications based on Docker container between Kubernetes clusters and Docker Swarm clusters successfully, and the time used to migrate applications decreases by 60.33% on average after introducing the technology of image pre-synchronization.

    Keyword：

    Swarm clusters; Kubernetes clusters; heterogeneous container cloud; scheduling strategy; migration technology;

    0 概述

    云计算作为一种新兴的计算模式, 近年来受到越来越多的关注和应用。云计算的核心支撑技术是虚拟化技术, 该技术使得多租户机制成为可能[1], 其在为虚拟应用弹性分配资源的同时也简化了应用的复制和扩展过程。目前主要有2种主流虚拟化技术:硬件级虚拟化技术和操作系统级虚拟化技术[2,3]。前者在物理服务器上运行Hypervisor (也称虚拟机监视器) , 虚拟机利用Hypervisor运行在物理服务器上, Hypervisor能为虚拟机分配适量的内存、CPU、网络和磁盘;后者则是在操作系统级虚拟资源、封装程序以及这些程序的依赖来组装成容器。

    容器虚拟化技术凭借其快速部署、轻量化以及高效等优点成为虚拟化技术中的研究热点[3,4], 近几年在广度和深度2个方面都有长足发展, 已经成为创建、发布和运行分布式应用的事实规范, 逐渐成为云计算领域中最具有深远影响的变革技术。这些影响主要体现在简化部署、快速启动和服务组合等方面。容器的轻量、标准化和可移植性使得企业能够快速地按需创建配置环境、部署管理应用, 以大幅提升交付效率和运维效率, 从而加速业务创新。

    然而企业希望运行多种异构技术, 目前云计算服务市场发展较快, 国外的云服务提供商主要有亚马逊、微软、谷歌等, 国内的云服务提供商主要有阿里、华为、美团等。为利用就近计算优势, 提高业务应用的全域服务能力以及避免被特定的云计算服务提供商锁定, 企业需要将自己的业务应用部署到多个地域、多个云计算服务提供商的数据中心内[5]。但是企业在跨域传输数据时需要付出很大的代价:一方面, 跨域数据传输不仅消耗时间长, 而且消耗宝贵的网络带宽资源, 性价比低;另一方面, 在安全规则限制下某些数据不允许被跨域传输。因此, 现在学术界和工业界都提倡在异构容器云之间进行应用的迁移。比如企业A在杭州和福州等地都有自己的数据中心, 在杭州数据中心采用的是基于Docker Swarm[6]编排的容器云服务, 而在福州数据中心采用的是基于Kubernetes[7]编排的容器云服务。当企业A需要在杭州和福州数据中心之间进行应用迁移时, 因为容器服务的异构性而无法实现。

    本文通过研究Kubernetes和Docker Swarm的应用编排调度策略, 开发一种A-Migrator (Application-Migrator) 异构容器云应用迁移系统, 并通过引入镜像预同步技术加快在Kubernetes集群和Docker Swarm集群之间的迁移。

    1 相关研究

    1.1 Kubernetes集群和Docker Swarm集群

    Kubernetes、Docker Swarm和Mesosphere[8]是目前3个主流的Docker容器编排调度引擎。而其中Kubernetes和Docker Swarm是2个最常用的在集群中创建容器的引擎。

    1.1.1 Kubernetes集群

    谷歌在2014年的谷歌开发者论坛上发布了开源的Docker容器集群管理引擎Kubernetes。目前, Kubernetes由Linux基金会下的Cloud Native Computing Foundation管理。Kubernetes使用Docker打包、实例化及运行应用程序, 通过集群的方式管理跨主机的容器, 可以解决位于不同主机上运行容器之间的通信问题[9]。Kubernetes提供高度的互操作性以及自我修复、自动升级、回滚、存储和编排等功能。Kubernetes在企业中使用比较广泛, 如国内的阿里和华为等公司。

    1.1.2 Docker Swarm集群

    Docker Swarm是Docker公司于2014年开始研发的第1个容器集群项目, 是该公司对容器生态支持的核心方案。Docker Swarm在Machine所提供功能的基础上优化主机资源利用率和容错服务, 支持用户创建可运行Docker Daemon的主机资源池, 然后在资源池中运行Docker容器。随后, Docker公司推出Swarm mode, 并从Docker 1.12版本开始, Swarm mode成为Docker Engine的一部分。Swarm mode引入服务的概念, 并提供众多的新特性, 如具有容错能力的去中心化设计、路由网络、动态伸缩、内置服务发现、负载均衡、滚动更新和安全传输等。

    1.2 云联邦

    目前对多数据中心的容器管理主要以云联邦 (Cloud Federation) 的形式进行[5,10,11], 如Kubernetes集群联邦、Hashicorp Nomad和Mesos集群联邦等。

    1.2.1 Kubernetes集群联邦

    Kubernetes集群联邦 (如图1所示) 在架构上同Kubernetes集群相似, 由一个集群联邦的API server提供一个标准的Kubernetes API, 并且通过etcd来存储状态。不同的是一个通常的Kubernetes集群只是管理节点计算, 而Kubernetes集群联邦管理所有的Kubernetes集群甚至是多个数据中心之中的Kubernetes集群。

    Mesos集群联邦 (如图3所示) 将资源分为主节点资源和从节点资源, 通过Framework来与多个主节点交互, 并自动将工作负载调配到多个集群中。

    虽然目前Kubernetes集群联邦、Hashicorp Nomad和Mesos集群联邦都能对多个数据中心的容器进行管理, 但都局限于同构容器编排引擎, 无法调度异构容器编排引擎之间的资源, 不能实现基于Docker容器的应用在Kubernetes集群与Docker Swarm集群之间调度。文献[12]提出基于日志重放的容器迁移方法, 该方法的迁移时间随着容器运行时间的增加而增加, 然而, 企业级应用往往需要长时间运行, 因此该方法存在耗时长的问题, 而且该方法仅适用于同构的容器云环境;文献[13,14]利用Checkpoint-Restart技术来进行容器应用迁移, 但是该方法需要在迁移前后的环境中文件系统一致, 并且要求网络一致, 无法适用于不同云提供商、跨地域的多数据中心的应用迁移;文献[15]提出的应用迁移三层模型将应用迁移过程分为Base Layer、Application Layer和Instance Layer这3层, 该方法虽然能在Base Layer层保证迁移前后文件系统的一致性, 但是由于未考虑到Docker Swarm集群和Kubernetes集群的编排异构性, 无法实现迁移前后编排引擎对应用的接管。因此, 文献[12,13,14,15]提出的方法同样都无法解决基于Docker容器的应用在异构容器云之间的迁移。本文设计的A-Migrator有效地利用了Docker Swarm和Kubernetes的编排规则, 能实现异构容器云之间的应用迁移, 并且通过镜像预同步技术大幅减少了应用迁移时间。

    2 A-Migrator系统

    本文通过A-Migrator系统进行跨异构容器云迁移。A-Migrator总体架构如图4所示。

    图4 A-Migrator总体架构   下载原图

    在A-Migrator系统中, Client为A-Migrator的客户端, 通过命令行与其他模块进行交互;IPS (Image Pre-Synchronization) 为镜像预同步模块, 实时监测数据中心新建镜像操作, 并将新建镜像同步到远端数据中心的镜像仓库;SwarmNetes为编排信息转换模块, 实现Kubernetes编排信息与Docker Swarm编排信息之间的转换, 并将转换后的编排信息同步到远端数据中心;镜像仓库为数据中心使用的私有镜像仓库, 负责存储镜像文件;日志收集模块收集系统的所有日志信息。A-Migrator进行应用迁移的过程参考2.2节中对基于镜像预同步的应用迁移技术的论述。

    2.1 编排信息同步技术

    基于Docker 容器的应用在跨异构容器云迁移时, 编排信息的同步通过SwarmNetes模块来实现。SwarmNetes架构如图5所示。其中, 虚线所示方向为应用从Kubernetes集群向Docker Swarm集群迁移时编排文件转换的过程, SwarmNetes从输入的Kubernetes编排信息文件中提取应用信息, 并保存在中间表示层SwarmNetes对象中, 通过转换将SwarmNetes对象转换为Docker Swarm集群可以识别的Swarm对象, 最后根据Swarm对象输出Docker Swarm集群的编排文件;实线所示方向为应用从Docker Swarm集群向Kubernetes集群迁移时编排文件转换的过程, SwarmNetes从输入的Docker Swarm编排信息文件中提取应用信息, 并保存在中间表示层SwarmNetes对象中, 通过转换将SwarmNetes对象转换成Kubernetes可以识别的Kubernetes对象, 最后根据Kubernetes对象输出Kubernetes集群的编排文件。

    图5 SwarmNetes架构   下载原图

    SwarmNetes实现编排信息转换算法如算法1所示。

    2.2 基于镜像预同步的应用迁移技术

    对于未采用镜像预同步技术进行应用迁移的方法, 本文称为传统方法, 如图6所示。

    图6 传统应用迁移方法   下载原图

    传统应用迁移方法主要有以下5个步骤:

    步骤1 将源数据中心应用的编排信息文件1转换为目的数据中心能识别的编排信息文件2, 并将编排信息文件2同步到目的数据中心。

    步骤2 在目的数据中心根据编排信息文件2创建应用时, 会从镜像仓库2拉取应用所依赖的镜像文件, 若镜像仓库2中没有该应用所依赖的镜像文件, 则进行步骤3;若镜像仓库2中有该应用所依赖的镜像文件, 则进行步骤4。

    步骤3 从源数据中心的镜像仓库1同步应用所需的镜像文件到镜像仓库2中。

    步骤4 从镜像仓库2中拉取应用所需的镜像文件。

    步骤5 根据编排信息2和镜像文件创建并运行应用。

    传统方法实现应用迁移所需的总时间T1为:

    T1=TOrcSync+TImageSync+TAPPRun

    TOrcSync=TOrcConvert+TOrcRep

    其中, TOrcSync为编排信息同步的时间, TOrcConvert为SwarmNetes在源数据中心将编排信息1转换为编排信息2的时间, TOrcRep为将编排信息2从源数据中心复制到目的数据中心所用的时间, TImageSync为镜像仓库1和镜像仓库2进行镜像同步的时间, TAPPRun在目的数据中心通过编排信息2和镜像文件创建应用所用的时间。

    A-Migrator引入镜像预同步技术来实现基于Docker容器的应用在异构容器云之间的迁移。基于镜像预同步的应用迁移技术 (如图7所示) 在源数据中心引入镜像预同步 (IPS) 模块, IPS实时监测源数据中新建镜像的操作, 若监测到有新建的镜像文件, 立即监测目的数据中心的镜像仓库2中是否有该镜像文件, 若没有则将该镜像文件同步到镜像仓库2中。

    图7 基于镜像预同步的应用迁移技术   下载原图

    基于镜像预同步的应用迁移技术步骤如下:

    步骤1 在源数据中实时检测新镜像文件的生成, 若发现新建立的镜像文件, 立即同步到目的数据中心。

    步骤2 通过2.1节提出的编排信息同步技术, 将源数据中心应用的编排信息文件1转换为目的数据中心能识别的编排信息文件2, 并将编排信息文件2同步到目的数据中心。

    步骤3 在目的数据中心根据编排信息文件2创建应用时, 会从镜像仓库2拉取应用所依赖的镜像文件, 若镜像仓库2中没有该应用所依赖的镜像文件, 则进行步骤4;若镜像仓库2中有该应用所依赖的镜像文件, 则进行步骤5。

    步骤4 从源数据中心的镜像仓库1同步应用所需的镜像文件到镜像仓库2中。

    步骤5 从镜像仓库2中拉取应用所需的镜像文件。

    步骤6 根据编排信息2和镜像文件创建并运行应用。

    基于镜像预同步的应用迁移技术实现应用迁移需要的总时间T2为:

    T2=TOrcSync+TAPPRun

    应用迁移时镜像同步阶段占用了主要时间, 从T1、T2的表达式可以看出, 基于镜像预同步的应用迁移技术, 大幅缩短了基于Docker容器应用进行跨容器云迁移时的总时间。

    3 实验结果与分析

    将基于Docker容器的应用在Docker Swarm集群和Kubernetes集群之间迁移时, 需要确保迁移后异构容器编排引擎能正确接管应用, 即编排引擎对应用的副本控制、应用访问存储、组成应用的容器之间的通信、外界对应用的访问、应用访问配置信息等都能正常进行。本文选取Docker容器中的4个典型应用来验证A-Migrator的可行性和有效性, 这4个应用分别为:

    1) Counter。Counter应用统计访问数据库的次数, 并在Web页面上显示。该应用依赖redis:3.0和tuna/docker-counter23:latest这2个镜像, 镜像文件大小分别为91.5 MB和696.0 MB。Counter用tuna/docker-counter23:latest镜像运行3个Web容器副本。Counter应用的成功迁移能代表对异构容器编排引擎副本控制规则的转换。

    2) Wordpress。Wordpress是一款个人博客系统, 使用PHP和MySQL数据库开发。该应用依赖wordpress:4.8-apache和mysql:5.6这2个镜像, 镜像文件大小分别为408.0 MB和256.0 MB。Wordpress应用在Docker Swarm中使用Named Volumes来提供存储卷, 在Kubernetes中使用Persistent Volume和Persistent Volume Claim来提供存储卷, 该应用的成功迁移代表迁移后应用能正常访问存储。

    3) Gitlab。Gitlab应用用于仓库管理, 使用Git作为代码管理工具, 并在此基础上搭建Web服务。该应用依赖redis:latest、postgresql:9.5和gitlab:8.13这3个镜像, 镜像文件大小分别为133 MB、232 MB和770 MB。Gitlab应用在Docker Swarm中使用env文件来提供环境变量和配置信息, 在Kubernetes中使用ConfigMap资源对象来提供环境变量和配置信息, 该应用的成功迁移代表迁移后应用能维持正确的配置信息。

    4) Vote。Vote负责统计对Dog和Cat这2个对象的投票结果, 并在Web页面上显示。该应用依赖redis:alpha、postgres:9.4、docker/example-voting-app-worker、docker/example-voting-app-vote和tmad ams333/example-voting-app-result这5个镜像, 镜像文件大小分别为27.8 MB、263.0 MB、574.0 MB、83.5 MB和226.0 MB。Vote应用在Docker Swarm中使用默认的ingress网络来确保多个容器之间的正常通信并提供外界访问, 在Kubernetes中使用Pod资源对象来确保多个容器之间的正常通信, 并使用Service资源对象来提供外界访问。该应用的成功迁移代表复杂应用在迁移后各个容器之间能正常通信, 以及复杂应用能正常对外提供服务。

    3.2 实验结果

    实验中分别用传统方法和A-Migrator方法将Counter、Wordpress、Gitlab和Vote这4个应用从Kubernetes集群迁移到Docker Swarm集群, 迁移所用时间如图9所示。传统方法迁移所用时间分别为121.71 s、117.46 s、193.25 s和185.10 s, 而利用A-Migrator方法进行迁移分别只需要46.97 s、54.28 s、62.06 s和82.08 s。

    分别用传统方法和A-Migrator方法将Counter、Wordpress、Gitlab和Vote这4个应用从Docker Swarm进群迁移到Kubernetes集群之中, 迁移所用时间如图10所示。传统方法所用时间分别为128.34 s、120.25 s、184.02 s和191.22 s, 而利用A-Migrator方法进行迁移分别只需要45.72 s、53.32 s、62.73 s和79.52 s。

    4 结束语

    为满足将基于Docker容器的应用在不同集群互相迁移的需求, 本文提出基于镜像预同步技术的A-Migrator异构容器云应用迁移系统。该系统通过A-Migrator能够实现基于Docker容器的应用在Kubernetes集群和Docker Swarm集群之间的迁移。实验结果表明, 与传统方法相比, 该系统具有较好的时间性能。下一步将研究如何降低A-Migrator带来的网络带宽消耗, 以减少不同数据中心之间网络频繁传输的影响。

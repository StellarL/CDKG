    摘    要：

    未来网络的网内存储优化存在依赖特定协议、基于网络高层次结构实现、缺乏实际部署应用的问题。为此, 采用协议无感知转发 (POF) 软件定义网络技术, 设计控制面与数据面的功能和交互方式, 构建一种应用于数据链路层的协同存储架构。通过扩充POF相关指令与处理流程, 基于该协同存储架构提出一种有效利用周边节点缓存资源的协同存储方法。实验结果表明, 该方法能够对上层未来网络应用提供透明的协同存储支持, 比基于CCNx、NDN的协同存储方法具有更高的转发效率。

    关键词：

    未来网络; 软件定义网络; 协议无感知转发; 信息中心网络; 协同存储;

    Collaborative Caching Method Based on Software Defined Network

    ZHU Xiaodong WANG Jinlin WANG Lingfang DING Li

    National Network New Media Engineering Research Center, Institute of Acoustics, Chinese Academy of Sciences University of Chinese Academy of Sciences

    Abstract：

    There are some problems in future network memory optimization, such as dependence on specific protocols, implementation based on high-level network structure and lack of practical deployment applications.To solve this problem, Protocol Oblivious Forwarding (POF) which is a kind of Software Defined Network (SDN) technology is used, the function and interaction mode between control surface and data surface are designed, and a cooperative storage architecture applied to data link layer is constructed.By extending the POF instruction and processing flow, a collaborative caching method based on this architecture is proposed to utilize the buffer resources of peripheral nodes effectively.Experimental results show that the proposed method can provide transparent collaborative storage support for future network applications and has higher forwarding efficiency than the collaborative caching method based on CCNx, NDN.

    Keyword：

    future network; Software Defined Network (SDN) ; Protocol Oblivious Forwarding (POF) ; Information Centric Network (ICN) ; collaborative caching;

    0 概述

    随着大数据、云计算等网络技术的快速发展, 以及网络视频、微博等新兴业务的呈现与增长, 当前渐进式变革的网络架构优化已越来越难以满足需求, 亟需对未来网络发展进行进一步研究。目前, 对未来网络研究的热点如信息中心网络[1]等, 以内容作为网络的中心, 关注于信息本身。而多种信息中心网络的架构, 如命名数据网络 (Named Data Network, NDN) [2]、内容中心网络 (Content Centric Network, CCN) [3]、移动性优先 (Mobility First, MF) [4]等都对网络节点的存储能力提出了需求。

    软件定义网络 (Software Defined Network, SDN) [5]将网络的控制与转发行为进行分离, 使网络设备更加灵活, 开放网络的可编程性, 实现高效快速的业务链部署, 并有效满足复杂网络对灵活性及可扩展性的要求[6]。目前, 已有很多关于软件定义网络的研究, 如文献[7,8]基于软件定义网络技术扩展了网络节点的存储能力。

    针对单节点存储能力不足、节点间存储内容冗余等诸多问题, 需要合理的方法来优化, 这一方面的研究比较缺乏。而且已有信息中心网络的协同算法缺乏对底层实际应用部署的考虑。因此, 目前尚缺乏在低网络层次直接协同利用多网络节点的存储资源, 并为上层有存储需求的网络应用提供承载的相关研究。另外, 网络节点的协同存储能力在如海云协同[9,10]等研究中有实际应用的需求与价值。

    基于软件定义网络技术, 本文构建一种应用于数据链路层的协同存储架构, 并提出一种应用于该架构的算法, 以有效协同利用各个节点的存储资源。以CCNx[11]应用为例进行系统对上层未来网络应用有效承载性的验证, 通过实验对比将存储决策由控制面下沉至数据面对数据包转发速度的影响, 最后在数据链路层对比本文算法与采用CCNx和NDN协议实现协同存储方式的转发效率。

    1 相关研究

    基于软件定义网络技术对网络节点存储能力的扩展, 目前已有的研究主要聚焦于单节点的存储扩展。文献[7]提出一种存储与转发分离的架构, 将存储设备与交换机的交换端口直连, 并定义数据传输的私有协议, 在不影响交换机线速转发功能的前提下, 支持较大的存储容量。文献[8]提出直接在软件定义网络交换机内部的数据平面利用交换机容量实现对存储的支持, 并同时考虑缓存替换策略。然而, 上述方法尚缺乏将各个节点存储资源协同利用的有效方式。

    对于协同存储方面的研究, 文献[12]提出下一跳缓存策略, 在兴趣包命中节点下一跳缓存数据, 可将热点内容推送至网络边缘。文献[13]提出基于哈希的邻域协同存储, 通过哈希确定每个数据块在邻域中的存储位置。文献[14]提出基于流行度的协同存储策略, 将热点内容放置于靠近用户侧并尽量降低网内缓存冗余。然而这些协同算法都是针对某一协议实现, 依赖于协议本身, 缺乏通用性, 且均为研究性质算法, 未考虑在实际系统中的应用。同时, 若协同存储算法运行于应用层或网络层之上, 则不可避免会带来过多的协议解析开销, 影响系统效率, 且并不适用未来网络替代TCP/IP协议栈的可能发展方向。在数据链路层实现协同存储, 能够灵活支持上层应用, 并具备高效的转发速率。

    综上, 目前仍缺乏协同利用已扩展存储功能的网络节点存储能力的方法, 且在设计上应尽量考虑在较低网络层次如数据链路层实现, 而且应当不受限于具体的上层协议, 并对上层应用提供灵活的支持, 同时具备较快的转发速率。

    2 本文方法设计

    2.1 协议无感知转发

    华为提出的协议无感知转发 (Protocol Oblivious Forwarding, POF) [15]技术, 其转发设备对数据报文协议和处理转发流程没有感知, 网络行为完全由控制面负责定义。该技术作为对OpenFlow协议的增强, 拓展目前软件定义网络的应用场景, 为实现真正灵活的可编程软件定义网络奠定基础。POF通过{位移, 长度}的匹配模式实现无需升级即可支持多种网络协议, 并对协议本身无感知。同时, POF定义了一组流指令集以实现更高的性能和更灵活的转发行为。研究人员也可以为新的应用开发更有效的POF指令。

    2.2 基于POF的协同存储架构

    为协同利用各个节点的存储资源, 首先需要对网络内缓存状态的整体感知。基于POF的协同存储架构利用软件定义网络控制与转发分离的优点, 如图1所示, 将管理域内缓存状态在控制器端的域内缓存状态模块进行管理, 更新通过节点周期性地上报存储状态来实现。同时, 为实现数据面对于数据包的快速转发, 将部分周边节点缓存状态保存在交换机端, 通过交换机间的信令交互以及与控制器端的周期性更新来实现。这一设计还可以降低对于向控制器高频次查询带来的压力和交互时延, 更好地支持快速转发。控制器端的缓存策略管理模块则负责对所采用的缓存策略进行管理及相应流表的生成、下发与删除等。

    图1 基于POF的协同存储架构   下载原图

    为在数据链路层实现协同存储, 设计如图2所示的帧结构。协同存储交互所需的基本功能包括:

    1) 从周边节点获取本地未缓存的数据。

    2.3 基于存储容量转发的协同存储算法

    针对单节点存储容量、速度等能力有限的问题, 在上文提出的存储架构下实现一种有效利用节点存储资源的协同存储算法。首先对管理域内的缓存状态进行维护;其次协同存储数据, 当本地无法满足存储需求时, 利用所维护的缓存状态将需存储数据转发至合适的节点存储并记录协同缓存结果;最后协同读取数据, 若在某节点内未获取所需数据, 而该数据在该节点记录已由周边节点协同缓存, 则会通过协同缓存数据的节点获取数据, 从而满足数据请求。

    2.3.1 存储状态维护

    为达到有效利用周边节点存储资源的目的, 需要对网络内各个节点剩余存储空间信息以及缓存替换信息进行维护, 并能够按照架构设计, 在控制器维护整体全局状态, 在交换机端维护周边节点状态用于快速转发。其中, 缓存替换信息为各周边节点在存储容量已满时, 依据所配置的缓存策略, 若其继续存储数据, 所需替换的缓存表征数据P。表征数据依节点具体配置的缓存替换策略所决定, 如:对于LFU策略, 表征数据为要替换缓存的访问次数;对于LRU策略, 表征数据为要替换缓存的最近访问时间。

    采用时间周期性驱动的方式, 扩展已有的链路层发现协议实现控制器端以及交换机端对于剩余存储空间和缓存替换信息的维护。首先设定节点发现帧为协同信息帧的一种, 由Tag字段来确定。控制器以软件定义网络中packet-out消息形式周期性地向域内所控制交换机的各个交换端口发送节点发现帧。

    交换机端处理流程如算法1所示。交换机接收到协同消息帧, 判定其为节点发现帧后, 则查询本地存储信息并将存储容量信息添加至节点发现帧, 然后将节点发现帧上报控制器, 修改节点发现帧Tag, 将其修改为节点发现回复帧并从接收端口返回。交换机判定接收到的协同消息帧为节点发现回复帧后, 则依此更新本地周边节点缓存状态模块, 存储周边节点缓存容量。

    算法1 存储状态维护算法

    控制器接收到交换机上报的包含剩余存储空间以及缓存替换信息的节点发现帧后, 依据此同时维护网络拓扑状态及交换机存储容量信息。同时, 在控制器端, 实时监听交换机上下线以及端口变化状态, 更新维护网络拓扑状态, 并周期性检查时间戳以删除失效链路。

    2.3.2 数据协同缓存

    若交换机需要存储接收到的数据包, 而本地存储容量不足, 则会依据本机内存储周边节点的剩余存储空间, 将数据包转发至剩余存储空间最大的节点由其缓存。若经查询, 本地及周边节点缓存容量均已满, 则比较本地及各周边节点的缓存替换信息P, 并选择缓存替换信息中优先级最低即P值最低的节点, 将数据包转发由其缓存。同时, 记录协同缓存的结果, 为后续获取该数据提供索引。

    数据协同缓存的流程如算法2所示。当交换机经上层协议解析得到需存储的报文载荷时, 若本地有存储空间, 则在本地存储;若本地没有存储空间, 则查询本地周边节点缓存状态模块中的周边节点存储容量, 将报文载荷以协同存储内容帧形式转给相应节点;若查询周边节点中也无合适的存储节点, 则比较本地及各周边节点的缓存替换信息, 依据信息中替换缓存的表征数据P来决定优先级, 选择缓存替换信息优先级最低的节点min{P1, P2, …}。若所选择节点为本地节点, 则直接在本地缓存该存储报文;若所选择节点为周边节点, 则将报文载荷以协同存储内容帧形式转给相应节点。交换机若接收到协同存数据帧, 则依据本地存储状态决定是否存储, 并将存储结果以协同消息帧形式从接收端口返回, 其中, 协同消息帧的Tag标记为协同存结果。交换机接收到协同消息帧, 以此更新周边节点缓存状态。

    算法2 数据协同缓存算法

    由于交换机内的存储数据经常变化, 为保证数据索引的实时性, 交换机的协同缓存数据在发生变化时上报控制器, 控制器依据上报信息及交换机上下线信息维护控制器端缓存映射。交换机周期性地通过控制器更新本地周边节点缓存状态模块, 存储交换机端缓存映射。

    2.3.3 数据协同读取

    交换机接收到数据请求, 而本地缓存的数据无法满足其数据请求时, 若依据交换机端缓存映射查询周边节点缓存了所需数据, 则从周边节点获取数据以满足该数据请求。

    数据协同读取的流程如算法3所示。当交换机经上层协议解析得到需读取的数据请求时, 首先查询本地存储, 若有该数据则满足数据请求。若本地无存储信息, 则查询缓存映射表, 如果匹配失败, 则交由上层协议处理;如果匹配成功, 则向相应节点发送协同兴趣帧。交换机接收到协同兴趣帧, 如果本地读取成功, 则以协同读取内容帧返回;如果本地读取失败, 则返回协同消息帧并指定Tag为读取失败。交换机若接收到协同读取内容帧则满足数据请求, 若接收到协同消息帧且Tag为读取失败则交由上层协议处理并更新周边节点存储状态。

    算法3 数据协同读取算法

    为避免可能的数据包丢失或周边节点数据获取失败等问题, 交换机周期性地对转发出的协同兴趣帧等待状态进行检查, 若超时则进行删除并交由上层协议处理。

    3 实验结果与分析

    为验证应用于本文提出架构的协同存储算法对于上层未来网络应用的支持性, 以CCNx应用为例进行相应的承载实验。同时, 对比基于控制器缓存决策并转发的方式, 验证将部分缓存状态维护在交换机端, 在数据平面进行协同存储的有效性。最后, 通过在数据链路层实现与基于特定网络协议实现的对比实验, 验证本文算法在数据链路层直接实现协同存储的高效转发性能。实验运行平台所需服务器均为64位Linux系统, 处理器为Intel (R) Xeon (R) CPU 2.10 GHz, 64 GB内存, 用于安装POF软交换机、控制器。

    3.2 转发速率对比实验

    本文算法中的协同缓存决策, 即选择哪个节点协同缓存数据以及从哪个节点获取本地没有的数据, 可放置于控制平面即控制器端, 也可以如算法所示在数据平面即交换机端直接实现。通过缓存决策在控制平面和数据平面的转发速率对比, 验证数据平面能够维护一定缓存状态, 并依据此进行协同存储决策的有效性验证。

    控制平面决策转发即在需要转发决策时将数据上报控制器, 由控制器端查询相关信息并将数据转发至相应节点。表1为局域协同转发存操作以及转发读操作报文性能对比, 采用64 KB定长的小包进行测试, 输入速率为1 000 KB/s, 实验结果取10次测试的平均值。由表1可知, 在该输入速率下, 数据平面可线速转发输入数据, 而控制平面决策并转发数据包的速率远远低于数据平面。

    3.3 数据链路层对比实验

    信息中心网络也提供基于数据链路层的实现, 实验对比了在数据链路层本文协同存储算法与基于NDN、CCNx实现的协同算法的转发速率。其中, CCNx协议栈数据包格式采用固定包头加TLV形式, NDN协议栈数据包格式采用全TLV嵌套的形式。

    实验采用64 KB定长的小包进行测试, 结果取10次测试的平均值。如图5所示, 在较低输入速率下, 三者均可按原速率将数据处理并转发出去;当输入速率提升至6 000 KB/s时, 基于CCNx及NDN实现的协同方式开始出现丢包, 而本文算法在9 000 KB/s时开始丢包;且本文算法转发速率始终优于基于CCNx及NDN实现的方式, 主要是因为CCNx和NDN需要更多指令去处理嵌套的TLV结构以获取名称等信息, 所以转发效率低于本文算法。由于CCNx具备定长的包结构, 因而信息交互的Tag可在定长结构的保留字段中实现, 而在NDN中则需要嵌套TLV结构实现, 因而基于CCNx实现协同存储转发效率高于基于NDN实现的协同存储。同时, 本文实现方式具有协议无关性, 更为灵活。

    4 结束语

    针对未来网络对网内节点存储能力的需求以及节点间缺乏有效协同存储方式的问题, 基于POF软件定义网络技术, 本文构建一种应用于数据链路层的协同存储架构, 并基于该架构提出一种有效利用节点存储空间的协同存储算法。实验结果表明, 该算法能够对上层运行的内容中心网络协议提供透明的协同存储支持, 将存储决策由控制面下沉至数据面可有效提高转发速率, 在数据链路层, 该算法比基于NDN、CCNx实现的协同算法具有更高的效率。本文算法仅考虑有效利用周边节点的存储空间, 下一步将设计更多基于该架构的协同存储算法, 以适应未来网络发展对网内存储的需求。

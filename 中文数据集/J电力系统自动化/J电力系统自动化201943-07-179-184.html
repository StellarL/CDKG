    摘    要：

    环网通信是就地化保护的一个重要技术特点, 其可靠性直接影响就地化保护的性能。文中介绍了就地化保护环网通信的通信机制和典型实现架构, 分析了通信中断、误码、延时补偿异常、网络风暴等典型通信异常对就地化保护的影响。在此基础上, 提出了基于双环双向数据判别的通信中断、误码、延时补偿异常处理策略以及基于报文识别和过滤的网络风暴处理策略, 并给出了具体的实现方案。最后通过试验证明了处理策略的可行性。

    Analysis on Communication Exceptions and Handling Strategies for Ring Network with Outdoor Installation Protection

    FU Mingjun WANG Longfei JI Xina DONG Leichao

    XJ Electric Co.Ltd.

    Abstract：

    Ring network communication is an important technical feature of outdoor installation protection, and its reliability directly affects the performance of outdoor installation protection. This paper introduces the communication mechanism and typical implementation framework of outdoor installation protection ring network communication, and analyses the influence of typical communication exceptions such as communication interruption, code error, delay compensation anomaly and network storm on outdoor installation protection. On this basis, handling strategy for communication interrupt, error code, and delay compensation based on double-loop bidirectional data discrimination, network storm handling strategy based on message recognition and filtering are proposed, and the specific implementation scheme is given. Finally, the feasibility and effectiveness of the handling strategy are verified by experiments.

    0 引言

    就地化保护是中国智能变电站技术研究的热点[1,2,3,4], 目前已成为智能变电站继电保护的重要发展方向[5]。就地化保护的一个典型技术特点是基于环网通信实现元件保护。与之前的常规变电站和三层两网架构的智能变电站保护相比, 就地化保护网络拓扑结构发生了变化。之前的变电站自动化网络主要以交换机为核心设备构建星形网络拓扑, 而就地化保护是由保护本身承担数据转发交换功能, 通过各保护子机首尾相连组成环型网络拓扑。新的拓扑结构决定了数据通信机制及产品实现架构的变化。就地化保护的网络通信重要性发生了变化。之前的变电站自动化网络通信如发生异常, 往往只是相应通信功能受影响, 一般不直接影响保护功能, 特别是主保护的动作行为。而就地化保护环网通信异常直接影响保护的采样, 进而影响保护的核心功能的性能。以上变化决定了以前用来进行网络异常处理的策略, 如利用交换机的流量抑制、利用报文过滤等功能来抑制网络风暴等手段[6,7], 已无法适用就地化保护。针对就地化保护环网通信异常的影响分析和处理策略必须结合新的通信机制和产品实现结构重新进行分析研究。

    目前业界针对环网通信的研究主要集于如何基于环网通信实现就地化保护, 如文献[8,9]研究了基于环网通信的就地化母线保护和就地化变压器保护技术实现方案, 介绍了高可靠性无缝冗余 (HSR) 环网的通信机理和基于环网通信的采样同步技术。本文侧重于介绍就地化元件保护环网通信的机制和实现架构, 分析环网通信的典型异常对就地化保护的影响, 最后提出相应的异常处理策略。

    1 环网通信机制分析及实现

    1.1 环网通信机制分析

    以就地化元件保护为例, 其由多个子机组成, 多个子机之间采用高可靠无缝冗余的双向双环通信网络和HSR协议[10]来实现信息交互传递, 共同完成元件保护功能。双向双环通信网络拓扑结构如图1所示。

    图1 双向双环通信网络   下载原图

    Fig.1 Bidirectional double ring communication network

    保护子机的启动CPU和保护CPU分别接入一个双向冗余环, 环内数据采用“接收先到报文, 丢弃后到报文”的方式。环网内传输的内容包括模拟量、开关量、自检信息、定值参数校验码。其核心内容为采样值, 自检信息用于重要告警闭锁, 定值参数校验码用于判别子机参数一致性。

    环网通信报文在以太网IEEE 802.3q标准报文的基础上, 在源以太网媒体接入控制器 (MAC) 字段后增加6字节环网特征字段, 形成环网通信的基础帧格式。环网报文本身携带其延时时间, 并利用延时修正域对报文延时进行逐级修正, 每经过一个节点修正一次, 直到所有节点传输结束, 最后利用修正域找到原始采样时刻, 进而插值采样实现采样值同步。

    1.2 环网通信的实现架构

    针对环网通信延时修正的技术需求, 目前业界主要采用现场可编程门阵列 (FPGA) 技术来实现MAC芯片的功能, 主要扩展实现以下功能:根据一定规则实现环网报文的接收和转发, 准确记录报文接收和发送的时间戳, 在报文中进行精确补偿, 同时负责与微控制器 (MCU) 芯片的应用程序进行通信。而在软件实现上, MCU上的应用程序一般又分为网卡驱动程序和上层应用程序。驱动程序负责网卡的初始化、中断处理、内存管理和收发数据, 主要是将网卡注册给应用程序, 屏蔽底层网络通信实现的技术细节。应用程序则负责网络报文的具体数据处理。典型实现架构如图2所示。

    2 环网通信异常及影响分析

    环网通信的核心内容是采样值, 其通信可靠性直接影响就地化保护采样的正确性。下面重点分析环网通信各种典型异常工况及对就地化元件保护的影响。

    2.1 通信中断

    光纤因外力断裂、保护设备PHY芯片损坏、光模块老化导致发送功率过低或接收灵敏度下降等, 均可能会导致光纤通信中断。环网通信从通信机理上降低了因通信中断带来的继电保护可靠性损失。当某个环内某一方向发生通信中断, 仍可以使用另一个方向的通信数据, 不会影响保护功能的性能。如果环内两个方向均发生通信中断, 则本环的保护功能因采样异常会闭锁保护, 根据双CPU与出口的原则, 此时整套保护因通信中断而功能丧失, 对于双套配置的保护, 此时变为单套运行方式。对于单向通信中断, 虽不影响保护功能, 但通信的整体可靠性降低, 保护应给出运行告警信息。对于双向通信中断, 明确导致保护功能退出, 保护应给出严重告警信息。

    2.2 误码

    误码是通信经常面临的问题。一般而言, 误码产生的原因主要由于传输过程中的干扰, 如电磁辐射等对继电保护设备报文发送或接收物理模块的影响, 或是光模块长时间工作老化、性能下降。实际工程应用中此类误码一般体现为报文循环冗余校验 (CRC) 错误, 可由PHY芯片直接识别, 直接做抛弃处理。采用环网通信有效降低了误码对保护的影响, 出现单方向传输误码时, 保护仍可以采用另一方向的通信数据, 不会对保护产生影响。考虑通信质量持续恶化, 可能双向产生误码, 而环网通信无数据重发机制, 因此体现到应用层, 主要的影响是采样丢点。如果是偶发的丢单点, 应用程序可对采样丢点进行数学上的线性差值补偿或抛物线差值补偿等处理[12], 有效降低丢点对保护功能性能的影响。如发生连续丢点等, 则不可避免对保护造成影响, 根据程度不同可能会导致保护动作时间延长或保护不能动作出口, 保护应给出严重告警信息。

    在实际工程应用中还有一种特殊类型的误码情况需要考虑, 即程序编码出错导致的误码, 这种误码表现为报文携带的数据错误而数据帧CRC校验正确。对于此种误码, 无法通过PHY直接丢弃, 也无法统计其帧错误计数。虽然跨间隔保护采用双环通信, 具有一定抵御误动作的机制, 但作为一种异常情况, 如果不能识别, 如双环均出现应用编码错误, 则会严重影响保护的保护行为。因此, 需要研究判别此类误码的识别机制, 用于闭锁保护和信息告警。

    2.3 延时补偿异常

    基于环网通信的采样方案本质上依赖于每个子机的延时修正准确性, 延时修正的准确性直接决定采样的正确性。环网通信主要应用于跨间隔保护, 基于环网通信实现采样值的共享, 在此基础上进行差动保护判别。如果某一子机的延时补偿机制异常, 则可能会导致差动保护的误动作。目前业界的研究主要在如何提高可靠性和加大测试力度确保延时补偿正确方面, 针对延时补偿机制异常的研究不多。

    2.4 网络风暴

    环网通信是就地化保护各子机之间的通信方式, 在物理上是封闭的通信系统, 原则上不用考虑外部网络报文带来的冲击, 保护实际运行时造成环网风暴的原因主要来自环网内部。一是环网通信机制失效, 如设备重复帧没有丢弃, 导致数据帧在环内无限重发, 进而形成网络风暴;二是子机发送机制异常, 原则上子机应该以4 000帧/s的频率发送通信报文, 由于内部时钟异常等原因导致发送频率异常, 导致网络风暴。网络风暴对保护设备的影响十分恶劣, 如果没有有效的处理策略, 将在短时间快速抢占CPU资源, 导致程序崩溃。近年来, 网络风暴的处理得到了电力系统业界的高度重视[12], 保护产品抵御网络风暴的能力成为继电保护产品检测验收的重点项目[13]。

    3 典型异常处理策略

    3.1 通信中断告警策略

    对于通信中断的判别, 实现方法相对简单, 由FPGA分别统计环内双向数据的接收计数即可, 判别一段时间内接收计数不再增加, 则判为通信中断。需要注意的是, 双向的环网数据需要分别判别告警, 以实现对环网运行状态的实时监测。

    3.2 误码告警策略

    由第2.2节分析, 误码主要有两类, 第1类为外部干扰等引起的误码, 主要表现为CRC错误, 第2类为编码错误引起的误码, 主要表现为CRC正确, 传输数据错误。

    下面研究第1类误码的告警判别策略。

    环网双向通信为报文提供了A和B两个方向的传输路径, 目前的通用做法为:正常通信时FPGA给上层应用及相邻子机转发先到的数据, 后到的数据作为设备重复帧直接由FPGA丢弃。报文处理方法如图3所示。

    图3 环网报文一般处理方法   下载原图

    Fig.3 General handling method for ring network messages

    这种做法的优势是简化了处理措施, 对上层应用屏蔽了下层环网数据接收的细节信息, 上层数据处理程序只需要接收处理一个方向的数据, CPU负荷较轻。但缺点是上层应用由于信息缺失无法对双向数据进行细化的异常判别告警, 不利于保护实际运行工况的状态监测和通信问题的排查。如果中间再增加一个环节, FPGA将接收的双向数据均上传至应用, 而专门用于双向数据异常判别, 用于保护采样和转发至相邻子机的数据仍采用先到的数据, 处理方法如图4所示。

    Fig.4 Improved handling method for ring network messages

    如果某子机数据A方向接收数据产生误码, 网口A的FPGA不再向应用发送采样值, 则由应用判为A方向某子机数据丢帧。B方向也是相同机制。需要注意的是单一方向的接收误码不对保护造成任何影响, 仅作为链路监视信息, 给运维调试提供更加丰富准确的实际运行信息。此种做法, 将环内双向数据都引入应用程序, 一定程度上增加了CPU负荷, 如果底层FPGA程序的异常报文过滤机制良好, 能有效屏蔽带外报文和抑制网络压力报文, 则透传到应用程序的仅为有效带内数据, 且只做数据接收帧连续性判别, 对CPU的负荷率影响不大。

    对于第2类误码, 利用单环的数据无法进行异常判断, 需要利用双环的数据进行判别, 其实质上为双AD采样数据的差异化判别。对双环传输的模拟量和开关量分别进行一致性判别, 模拟量的判别可以参考智能变电站保护双AD一致性判别逻辑进行处理[14], 开关量直接进行一致性判别即可, 判别异常后应及时闭锁保护。

    3.3 延时补偿异常处理策略

    之前智能变电站采用直采模式, 因为保护是单向接收采样值报文, 无法判别报文中延时的正确性。而就地化保护采用双环通信则为延时正确性的判别提供了条件。就地化保护双环通信分别应用于启动环和保护环, 双环数据经过的物理路径相同, 数据在光纤通道中的延时一致, 且环网通信的数据转发均由FPGA实现, 延时稳定, 故双环的数据链路延时基本一致。可利用此特点对双环数据的采样延时正确性进行校验, 即对比双环延时的差值大于一定门槛, 可判别为某一环延时补偿异常, 可用于告警提示, 进而排查解决。识别出延时补偿异常是否闭锁保护, 虽然双环数据独立用于启动CPU和保护CPU判别, 最后取“与”运算之后动作出口, 故单环的保护延时异常不会导致保护的误出口, 但考虑到存在两个环延时补偿均错误的可能, 笔者认为如果准确判别出两个环内延时补偿存在明显差异, 应该闭锁保护, 杜绝保护误出口的可能。具体在保护实现上还可以辅以环内延时补偿值是否发生突变作为判据, 最终确定延时补偿是否异常。需要注意的是此机制基于正常的环网通信工况, 设计时要考虑网络风暴的影响, 网络风暴的报文会导致报文处理延时抖动变大, 识别存在网络风暴时, 不可再进行一致性判别。

    3.4 网络风暴处理策略

    环网内网络风暴根据报文的特征属性可以分为带外网络报文风暴和带内网络报文风暴。符合环网通信协议格式, 链路层参数、应用层参数均满足接收要求的报文是带内报文, 否则为带外报文。结合目前就地化保护的应用情况, 一般忽略链路延时测量, 环网中只有单一类型报文即应用数据报文, 报文格式固定、传输长度不变。带内报文的判定可以方便地由FPGA对报文的固定字段进行判别即可。结合环网报文的特点, 源MAC地址为非订阅、环网号不一致、应用报文类型不符合、应用标识 (APPID) 为非订阅的报文, 均判为带外报文。如果FPGA判为带外报文则直接丢弃, 不再向下一节点转发, 也不向上层应用转发, 即可避免网络报文对CPU的冲击, 同时减少对环网通信的影响。

    根据第2.4节分析, 实际运行时产生带内网络风暴的可能性比较大。由于正常通信时带内报文只有一种类型且传输频率稳定, 基本为4帧/ms, 环网报文的处理机制为转发优先, 故子机的转发延时最大抖动为一帧报文的发送处理时间, 其典型时间实测一般小于5 μs。故每毫秒接收帧数相对固定为3～4。鉴于以上特征, 带内报文可由驱动层进行判别, 如果在短的时间窗口 (如5 ms) 内接收帧数异常, 则识别为风暴, 可通知FPGA停止接收此组报文一段时间, 同时不再向下一节点转发此组报文。间隔一段时间 (如5 s) 后重新开放短窗, 再对报文帧数是否异常进行判别。具体的接收帧异常判别门槛, 可根据具体的处理器性能进行合理设置。这样处理的优势是实现简单, 对于带外报文可完全过滤, 对于异常的带内报文由第1个转发子机直接丢弃, 则后续所有子机应用层感受到异常报文采样值均为零值无效, 确保了各子机保护动作行为的一致性。

    4 试验验证

    针对应用本文方案的就地化母线保护装置进行试验验证, 搭建试验系统如图5所示。

    图5 试验系统   下载原图

    Fig.5 Testing system

    试验的主要思路是利用专业的网络测试设备模拟各种典型异常, 进而检测对保护功能性能的影响及装置的相关告警情况, 验证本文策略的有效性。其中IXIA XM2网络测试仪负责模拟各种带内带外网络风暴, IXIA NE II损伤测试仪负责模拟报文丢帧、编码错误及报文中延时字段异常等, 同时利用继电保护测试仪来实际测试保护动作行为及性能所受的影响。

    通过试验结果可知, 就地化保护装置在通信中断试验时能可靠判别环内双向通信是否中断, 并给出告警提示。在误码试验时, 对于外部干扰等引起的第1类误码, 模拟环内任一方向通信误码, 均能给出告警提示;对于编码错误引起的第2类误码, 能够可靠提示异常并闭锁保护。在延时补偿异常试验时, 能够判别环内延时补偿错误, 给出告警提示, 同时闭锁保护。在网络风暴试验时, 实测装置不受带外报文压力的影响。模拟1 000 Mbit/s的带外风暴报文, 无死机或重启现象。模拟区内故障时, 保护装置可靠动作, 模拟区外故障时, 保护装置不误动。施加带内订阅报文风暴, 装置与订阅报文压力不相关的功能、性能不受影响, 能够正常运行, 不死机、不重启。模拟1 000 Mbit/s的带内风暴报文, 模拟区内故障时, 保护装置能可靠动作, 模拟区外故障时, 保护装置不误动。与订阅报文压力相关的保护功能、性能在风暴停止后可靠恢复。综上可知, 本文所提处理策略有效, 能够达到预期设计目标。

    5 结语

    基于双环双向数据判别的误码和延时补偿异常策略和基于报文识别和过滤的网络风暴处理策略较好地解决了环网通信的异常识别和处理问题, 对于提高就地化保护的整体运行可靠性和运维方便性有一定积极作用。本文所提策略已在相关产品中得到应用并通过了权威检测部门的验证, 挂网试运行效果良好。

    由于就地化保护目前尚处于挂网试运行阶段, 本文的处理策略仍需经受长时间运行的考验。如何结合就地化保护实际工程运行情况进一步完善处理策略是后续的研究方向。

    基于RRPP边缘节点的流量中断研究

    刘梦诗 王虹 张芮

    南京理工大学电子工程与光电技术学院

    摘    要：

    以太网快速环网保护协议 (RRPP) 在以太网环中能够防止数据环路引起的广播风暴。当RRRP环的边缘节点和辅助边缘节点之间的主环链路都出现故障时, 边缘节点收到Major-Fault报文后阻塞边缘端口, 这时的边缘节点流量不通仿若一个“孤岛”。文中针对上述问题, 将环网中的所有子环按环号从小到大存储在自定义的链表中, 当边缘节点收到了Major-Fault报文, 则选择环网中最小子环的边缘端口放开, 使得边缘端口的状态由阻塞变为放开。该方法解决了当主环链路故障时, 边缘端口的流量中断问题, 此时边缘节点去除阻塞, 不再是流量不通的一个“孤岛”。

    关键词：

    快速环网保护协议; 边缘节点; 辅助边缘节点; 子环; 主环; 孤岛;

    作者简介： 刘梦诗 (1994-) , 女, 硕士研究生。研究方向:信息与通信。;

    Research on Flow Interruption of RRPP Edge Nodes

    LIU Mengshi WANG Hong ZHANG Rui

    School of Electronic and Optical Engineering, Nanjing University of Science and Technology

    Abstract：

    RRPP can prevent broadcast storms caused by data loops. When the main link between the edge node and the assistant edge node failed, the assistant edge node will block the edge port after the edge node received Major-Fault packet. The edge node at this time just like an “island.” In this paper, all the subrings in the ring ware stored in a custom linked list from small to large according to the above problem. When the edge node received Major-Fault packet, the edge port of the smallest sub-ring in the ring network was selected, and the state of the edge port was changed from blocking to releasing. It solved the problem of traffic interruption at the edge port when the primary ring link fails.At this point, the edge node removes the blockage and is no longer an “island” of traffic failure.

    Keyword：

    rapid ring protection protocol; edge node; assistant edge node; subring; main ring; isolated island;

    Received： 2018-08-31

    Fund： Development Fundation of Nanjing University of Science and Technology (30918014106);

    RRPP组网如图1所示, 子环的主节点会从主端口每隔1 s发送Hello报文来检测链路状况, 当主环处于非断裂状态时, 经过边缘节点B和辅助边缘节点D被子环副端口接收。当B、D之间的一条主环链路故障时, 子环的Hello报文走向是B->A->C->D。此时设备D发现自己的主端口down后, 就会从副端口发送LinkDown报文通知主环主节点链路发生故障, 主节点收到LinkDown报文立刻放开它的副端口, 同时发送Common-Flush-FDB报文通知所有传输结点、边缘节点以及辅助边缘节点[1]。使其更新各自的MAC地址表项和ARP/ND表项。各节点更新完表项后, 数据流切换到正常链路上[1]。

    图1 主链路故障图   下载原图

    Figure 1. Main link failure diagram

    如果B、D结点之间的两条主环链路都发生了故障。由于边缘节点B每隔0.5 s沿B->D方向发送Edge-Hello报文。若D在2 s内没有收到该报文, 则表明B和D间的两条主环链路都出现了故障。此时, 辅助边缘节点D沿子环方向 (D->E->B) 发送Major-Fault报文, 边缘节点B收到该报文后会阻塞边缘端口, 防止子环成环[1]。但这时边缘节点B的所有端口都被阻塞, 流量不通, 在主环链路故障期间, 该设备毫无用武之地。本文在现有技术的基础上, 对该现象做了改进, 提供了一种完整的技术方案。

    1 硬件实现

    1.1 硬件环境

    本方案的组网环境为6台H3C S58系列交换机, H3C S58系列交换机是H3C公司自主开发的下一代数据中心级万兆以太网交换产品[2]。分为S5800、S5810和S5820X共3个子系列, 为数据中心提供丰富的服务器接入方案。也可以用于园区网的汇聚层或接入层以及中小企业核心[2]。

    S5800系列机箱式交换机支持H3C创新的IRF2 (Intelligent Resilient Framework 2, 第二代智能弹性架构) 技术, 用户可以将多台S5800交换机连接, 形成一个逻辑上的独立实体, 从而构建具备高可靠性、易扩展性和易管理性的新型智能网络[2]。

    本文网络拓扑为相交环, 其中一个主环, 两个子环 (也可以多子环, 原理相同) , 如图3所示。

    Figure 2. Network diagram

    1.2 硬件配置

    RRPP是国内某公司的一种配置型协议[6], 需要进行的配置有:Device A～Device F构成RRPP域1, 主控制VLAN设为VLAN 100, 保护VLAN设为VLAN 200 to 300[1]。

    主环由设备A, B, C, D构成。子环1由设备B, D, E构成。子环2由设备B, D, F构成。主环的主节点为设备A, 指定设备A的主副端口。主环的传输结点与子环的边缘端口为设备B。主环的传输节点与子环的辅助边缘节点为设备D。子环1和2的主节点分别为设备E和F, 指定设备E和F的主副端口。边缘端口和辅助边缘端口只接入子环的端口设为边缘端口[3]。

    2 算法实现

    2.1 选出N个子环中最小子环算法

    对于RRPP多子环的情况, 即网路拓扑中有两个或两个以上的子环, 各环有两个公共结点, 且这两个公共结点相同时, 可以定义一个RRPP域, 选择其中一个为主环, 其他为子环[1]。子环的编号可以自行定义, 一般从2开始递增。将子环按环号从小到大存储在自定义的链表中。当边缘节点收到了Major-Fault报文, 说明主环链路发生故障, 边缘节点的边缘端口被阻塞, 此时选择链表中最小环的边缘端口放开, 既不会导致8字成环, 也不会形成“孤岛”[7]。选择最小子环放开的流程图如图3所示。

    2.2 环删除时环选择算法

    如果发生故障后, 边缘节点已经选择了最小环的边缘端口放开。而此时连接边缘端口的某条链路发生了故障。这时边缘节点感知到是否是放开了的那条链路发生了故障, 如果不是, 则保持当前状态不做改变[8];如果是, 则需在剩下健康的子环里重新选择一个最小环的边缘端口放开, 流程图如图4所示。

    4 结束语

    本文针对主环链路故障时, 阻塞边缘端口的问题做了改进[15]。该技术方案选择一个最小环对应的边缘端口放开, 可以让该设备的流量不中断, 不会有资源的浪费。同时只放开一个端口, 避免了多个子环间形成“8”字环路, 造成严重的广播风暴[16]。对于环删除的处理, 只需要判断是否是已放开的端口链路发生故障, 再重新选择一个端口放开, 对于添加环的处理也非常得容易。

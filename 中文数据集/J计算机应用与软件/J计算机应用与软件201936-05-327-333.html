    摘    要：

    为了在不修改处理器结构的基础上检测代码重用攻击, 提出面向代码重用攻击检测的安全调试架构。针对通用基础调试结构可利用的调试信号, 设计总线跟踪模块, 为执行内存访问控制提供接口。研究基于性能计数器的函数级代码重用攻击检测方法, 增加跟踪检测微控制单元, 使安全调试架构与性能计数器配合工作, 在不插桩的前提下完成检测。实验结果表明, 安全调试架构传输延时小, 资源消耗少, 能够有效检测代码重用攻击。

    代码重用攻击; 嵌入式设备安全; 调试架构; 性能计数器;

    RESEARCH AND DESIGN OF SECURE DEBUG ARCHITECTURE FOR DETECTING CODE REUSE ATTACK

    Li Yang Dai Zibin Li Junwei

    Information Engineering University

    Abstract：

    In order to detect code reuse attacks without modifying the structure of the processor, we proposed the secure debug architecture for detecting code reuse attacks. For the debugging signals available in the general basic debugging structure, a bus trace module was designed to provide an interface for executing memory access control. We studied the function-level code reuse attack detection method based on performance counter. We added a trace detecting micro controller unit to make the security debugging architecture work with performance counter and to complete the detection without source code instrumentation. The experimental results show that the security debugging architecture can detect code reuse attacks effectively with less transmission delay and less resource consumption.

    Keyword：

    Code reuse attacks; Embedded device security; Debug architecture; Performance counters;

    代码重用攻击[1]作为一种新型软件攻击, 已严重威胁到嵌入式设备的安全。针对代码重用攻击, 许多检测方案将硬件监控单元集成在处理器的流水线中[2,3,4,5], 将检测所需的信号从不同流水线阶段转发到硬件监控单元, 硬件监控单元利用这些信号检测控制流的完整性, 一些方案是在被测系统的指令集中扩展特殊指令来完成检测[6,7,8,9], 这些方案都需要改动处理器流水线。但是在一些特殊场景, 并不能通过修改处理器流水线部署安全机制, 例如某些第三方处理器供应商禁止修改处理器内部逻辑, 或者修改处理器内部逻辑会耗费较高的成本。这时许多方案利用现有的调试接口监视程序的行为, 这些方案符合SoC的设计规则, 能够以低成本实现。

    目前, 已有众多学者在不修改处理器结构的基础上, 提出了检测方案。Corey等[10]利用性能计数器计数结果之间的数学关系, 建立模型来区分正常程序和恶意程序, 但是建模过程复杂, 检测精度较低;Wang等[11]将检测代码固化在Boot ROM中, 在加载固件时检测代码重用攻击, 能够确保固件执行的完整性, 但是不能够保证应用程序执行的完整性。现有的利用性能计数器检测代码重用攻击的方案基本都是纯软件实现, 需要借助软件工具在可执行文件中插桩检测代码。除了利用性能计数器, 许多方案利用调试接口部署检测策略, Lee等[12]观测到ARM CoreSight调试器能够生成的控制流跟踪信息仅仅包括间接分支目标地址, 为了部署硬件影子堆栈, 还需要获取分支类型和源地址, 为此, 通过离线分析二进制文件生成需补充的元数据。Lee等[13]通过ARM的核心调试接口CDI (Core Debug Interface) 实施跟踪检测, CONVERSE使用IEEE-ISTO Nexus 5001标准调试接口部署检测策略[14]。利用调试架构检测通常需要补充元数据, 并且容易产生跟踪丢失。此外, 对设备调试接口规范的属性具有依赖性, 例如CDI作为高级调试接口较少在低端嵌入式设备中使用。

    本文将性能计数器检测方法与调试接口检测方法相结合, 提出一种面向代码重用攻击检测的安全调试架构, 制定了函数级检测方案, 能够在不修改处理内部逻辑、不插桩的前提下完成检测, 克服了现有技术的不足。

    1 攻击模型假设

    代码重用攻击利用漏洞改变程序在内存中原有代码、执行顺序来达到攻击的目的。代码重用攻击通常通过缓存溢出部署, 通过覆盖返回地址或其他参数重定向控制流, 链接具有不同功能的短指令片段完成恶意操作。本文假设攻击者能够反汇编库文件和可执行文件, 能够查找到可被用于构建攻击的短指令片段, 假设用于编写应用程序的高级语言中存在可被攻击者利用的漏铜。这些漏洞是广泛存在的, 这也是代码重用攻击种类多、防御困难和发展迅速的主要原因。同时假设加载进内存的代码不可修改, 数据不可执行, 例如系统受到NX (No eXecute) 机制的保护[15], 能够防御代码注入攻击[16]。

    代码重用攻击能够利用程序漏洞修改基本块末尾条件分支指令的条件变量, 在程序控制流图CFG (Control Flow Graph) 有效路径之内改变执行方向, 如图1中虚线a所示;或改变循环执行的次数, 如图中1虚线b所示, 例如LOP (Loop Oriented Programming) 攻击[17];此外, 代码重用攻击能够通过修改基本块末尾代码指针, 执行不在程序控制流图有效路径限定内的非法路径, 如图中1虚线c所示, 例如ROP (Return Oriented Programming) 攻击[18]和JOP (Jump Oriented Programming) 攻击[19]。

    2 关键技术研究

    2.1 调试规范研究

    在SoC中集成基础调试架构, 能够深入了解执行软件的系统, 掌握软件在芯片上具体的执行情况。一个高效强大的调试系统可以大大缩短系统的开发周期。现今异构多核设计给软件开发人员带来了新的挑战, 跨多个CPU和硬件加速器执行并发软件、复杂的I/O接口交互和严格的实时要求是新常态, 由此产生许多新型漏洞, 例如竞争条件、死锁和严重的性能降级。查找此类错误, 断点调试已不适用, 因为设置断点会干扰不同执行线程之间的时间关系, 发生“探测效应”, 导致原始问题的消失。这些问题也会干扰对代码重用攻击的检测。为了解决以上问题, 现有的调试架构基本都支持非侵入式的跟踪调试, 将观察结果传输到片外, 然后在片外重建程序流并分析程序行为。

    目前大多数芯片供应商提供的基础调试架构主要遵循两种规范:ARM CoreSight[20]或NEXUS 5001 (官方称为“IEEE-ISTO 5001”) [21], 都支持运行控制调试和跟踪调试。但是这两种规范都不是开源的, 需要支付费用才可以使用, 这阻碍了增强调试架构功能的研究。

    开源调试架构OSD (Open SoC Debug) 的出现解决了这个问题[22]。OSD是一项完全开源的规范, 无需任何委员会成员资格或支付许可使用费, 任何人都可以自由分享和修改规范本身, 以及出于任何目的创建和分发规范的实现。OSD是一种可扩展的模块化规范, 如图2所示, 规范中定义了具有不同功能的调试模块组件:

    (1) SCM SCM (Subnet Control Module) 主要用于提供系统信息, 例如调试网络中调试模块的数量, 子网络编号和支持调试数据包的大小。其还能够复位子网络或复位处理器。

    (2) CDM CDM (Core Debug Module) 主要用于运行控制调试, 调试前需要挂起处理器, 调试过程中, 能够通过访问处理器中的特殊寄存器控制处理器的运行, 这些特殊寄存器中包括了性能计数器的使能寄存器、复位寄存器和计数寄存器。调试结束后, 解除处理器的挂起。

    (3) CTM CTM (Core Trace Module) 主要用于跟踪模式 (机器模式和用户模式) 的更改和函数的执行。CTM通过跟踪调用指令和返回指令的执行, 以及这些指令的PC指针值 (内存地址) , 在elf可执行文件中查找指令位置后, 能够跟踪函数的调用和返回。

    (4) MAM MAM (Memory Access Module) 提供访问系统内存的接口, 与CDM配合使用。

    (5) HAM HAM (Host Authentication Module) 主要用于令牌匹配和身份认证。调试架构会将系统内部信号暴露, 为了防止攻击者滥用调试架构, 需要经过HAM的认证, 主机接口模块才能够与HAM以外的调试模块通信。

    图2 OSD调试架构   下载原图

    OSD遵循IP复用技术, 通过复用OSD规范中的组件、接口和软件工具, 能够完成定制设计。OSD的开源特性也使其非常具有竞争力, 未来必然会被广泛使用, 所以本文以OSD为基础调试架构设计一款面向代码重用攻击检测的安全调试架构。

    2.2 基于性能计数器的代码重用攻击检测技术

    性能计数器几乎被集成在所有的主流处理器中, 本质是一组特殊寄存器, 能够在处理器运行程序的过程中, 对处理器内部某些硬件相关的动作进行计数, 配合处理器外部调试接口, 设立时间断点, 可以获得处理器的性能参数。性能计数器通常被用于硬件系统的性能评估和低功耗优化。

    本文将性能计数器用于安全检测, 是基于现代RISC处理器中提供的性能计数器基本都能够实现实时的事件响应和精确无漏的采样。本文通过性能计数器与调试架构的配合, 利用性能计数器在执行正常程序和异常程序产生计数结果的差异, 检测代码重用攻击。本文选用具备嵌入式设备典型特征的PULPino SoC平台作为实验平台[23], 如表1所示, PULPino中RISC处理器性能计数器能够对十个事件进行计数, 计数结果分别存储在名称为PCCR0-PCCR10的寄存器中。

    PCCR1    		INSTR对执行指令的数目计数

    PCCR2    		LD_STALL对由读取内存引起的数据冲突计数

    PCCR3    		JR_STALL对由访问跳转寄存器引起的冲突计数

    PCCR8    		BRANCH对分支数目计数

    PCCR10    		RVC对压缩指令执行的数目计数

    利用性能计数器检测代码重用攻击分为离线阶段和在线阶段, 离线阶段收集并存储检测点正常的性能事件值, 在线阶段跟踪程序的执行, 执行到检测点时, 通过比较实际性能事件值与正常性能事件值判断是否遭受到攻击。为了有效检测代码重用攻击, 需要解决以下四方面的问题:

    (1) 性能事件的选择

    代码重用攻击不会明显影响所有的性能事件, 为了有效区分正常程序和异常程序的执行, 需选择合适的性能事件。通常实施代码重用攻击, 首先会通过缓存区溢出劫持控制流, 覆盖程序中原有的局部变量, 若变量为分支判定条件, 则会使程序偏离原有的执行路径, 从而影响程序中采取的条件分支指令数BTAKEN。又因为代码重用攻击的shellcode是由连续的以返回指令、跳转指令或分支指令结尾的短指令片段组成, 在完成恶意操作之前, 无条件跳转的指令数JUMP和分支指令数BRANCH会明显增加。此外, 由于代码重用攻击的配件分散在整个内存空间, 不具备空间局部性, 分支预测的成功率大大降低。因此, 本文还选取等待取指的时钟周期数IMISS来检测代码重用攻击。

    (2) 收集性能事件值的粒度

    性能计数器在实际运用中, 即使没有被攻击, 执行相同代码也会出现不同的性能计数值, 这是由于各个性能事件发生的次数会受多种不确定因素的影响, 例如程序被加载到内存不同的位置执行, 程序中产生了随机数或计数过程中产生噪声, 都会造成计数结果的不同。

    代码重用攻击的shellcode通常由100多条指令组成, 对性能事件值的影响通常集中在完成恶意操作前, 因此将整个应用程序作为计数单位, 产生的误差较大。同一函数内的指令种类差异较小, 性能事件值变化较小, 如果受到攻击, 将会产生明显差异, 因此本文以函数粒度收集性能事件值。

    (3) 触发检测的时机

    如果在每次调用函数前都执行检测, 则检测效率低。通过研究发现, 无论是哪种代码重用攻击, 最终都要利用操作系统提供的执行命令、访问文件和网络通信等功能, 才能达到控制系统或窃取敏感信息的目的。本文将系统调用前的一个函数作为敏感函数进行检测, 定义敏感函数集并监控程序的执行, 在调用系统函数前, 检测性能事件值是否存在异常。

    (4) 检测阈值的确定

    如上所述, 性能事件值会受到多种干扰因素的影响, 并不是一个确定值, 需要通过统计分析, 确定性能计数值的正常范围, 在正常范围之外的计数结果视为系统遭受攻击。本文借助“盒须图”统计模型来确定检测阈值。

    3 安全调试架构总体架构

    面向代码重用攻击检测的安全调试架构以OSD为基础设计得到, 但不代表本设计适用范围局限, 大多数调试规范都具备与OSD一样的功能, 本设计同样适用于其他调试规范。此外, 开源调试规范OSD功能全面, 与现有的商用调试规范相比, 具有强大的市场竞争力, 未来OSD会得到广泛使用, 本文研究增强了OSD的功能, 研究具有前瞻性, 未来用户可根据需要在OSD中添加检测代码重用攻击的组件。同时, 本文选择的PULPino SoC具备嵌入式设备的典型特征, PULPino结构简单, 其性能计数器跟踪的事件类型, 大多数嵌入式设备都能够跟踪。因此本文利用性能计数器执行检测的方案也适用于其他设备。

    图3为安全调试架构的总体架构, 图左侧为被测SoC系统。本文在原有OSD基础调试架构中添加了ATM (AXI Trace Module) 和TDMCU (Trace Detecting Micro Controller Unit) , 如图3阴影部分所示。在对OSD的研究过程中, 发现没有跟踪总线的调试模块。片上系统访存操作都需要访问总线, 通过ATM引出的总线信号, 可以检测到非授权访问, 用于防御代码重用攻击的部署。TDMCU用于跟踪检测敏感函数的性能事件值是否异常, 可用于检测代码重用攻击的实施。

    3.1 ATM

    代码重用攻击者能够利用内存泄漏, 从内存中查找和定位用于构建攻击的短指令片段。内存中数据和指令具有极大的关联性, 真正控制指令执行的还是数据, 攻击者利用程序漏洞对控制转移指令的目标地址和条件分支指令的条件进行篡改, 使程序不按照原始路径执行。

    在攻击模型假设中已分析到, 攻击者能够注入数据的地方只能是普通数据段和地址段, 但是由于普通数据段和地址段在内存中存储的方式没有差别, 空间上没有分离, 这就给攻击者篡改的机会, 导致代码重用攻击的发生。内存访问控制能够防止越界访存, 有效防御了代码重用攻击。执行内存访问控制需要从总线中提取访存信号, ATM为跟踪AXI总线上的信号提供了接口, 为执行内存访问控制提供了便利。如图4所示为ATM、AXI总线和内存访问控制模块三者之间的关系, ATM能够对AXI总线上的信号进行行为分析, 判断当前访问的读或写行为, 并将跟踪到的总线信息以数据包的形式发送到Debug接口, 内存访问控制模块从Debug接口接收信息, 并执行边界检查, 一旦发现越界访存, 利用调试接口中的BREAK功能立即停止设备的运行。本文重点研究检测代码重用攻击, 因此不对内存访问控制模块进行介绍。

    3.2 TDMCU

    调试架构中的组件能够从片上系统内部提取用于执行检测策略的内部信号, 性能计数器能够跟踪记录程序执行过程中各个事件类型发生的次数。安全调试架构的工作原理是:在系统功能模式下, 利用调试架构在程序运行过程中的特定时机获取性能事件值, 将实时获取的性能事件值与离线分析正常的参照性能事件值进行比较, 从而达到检测攻击的目的。TDMCU作为安全调试架构的核心, 控制整个检测过程的进行, 其主要功能包括:

    (1) TDMCU与被测系统通信, 通过配置OSD中的CTM, TDMCU能够从CTM接收跟踪数据包, 跟踪程序中函数的执行, 当发现执行到敏感函数入口时, TDMCU通过配置OSD中的CDM复位CPU的性能计数器, 该过程需要挂起CPU。然后解除CPU挂起, 继续执行程序, 性能计数器开始计数。

    (2) TDMCU跟踪函数的执行, 当发现执行到敏感函数出口时, 控制CDM挂起CPU, 并读取性能计数器的值。

    (3) 将读取的性能事件值与预存的性能事件值进行比较, 如果满足阈值要求, 则解除CPU挂起, 继续执行程序;如果不满足阈值要求, 配置OSD产生BREAK停止设备的运行。

    本文设计充分利用了调试架构的跟踪、访存、通信和BREAK等功能, 通过TDMCU的控制, 在不修改处理器和不插桩的前提下, 完成对运行在系统上程序的代码重用攻击检测。OSD的调试接口中包含一个FIFO存储器, 在TDMCU读取跟踪信息之前, FIFO能够缓存跟踪信息。如果没有足够快地读取信息, FIFO会被填满, 导致跟踪信息的丢失。虽然OSD能够对丢失的数据包进行计数, 当能够再次传输数据时, 发送代表溢出的数据包, 但是包中不含有跟踪信息, 无法进行检测。TDMCU从FIFO提取跟踪数据包的速率受到被测系统执行函数的速率、FIFO存储深度和调试接口位宽等因素的影响。为了分析跟踪丢失的情况, 定义以下变量:

    Rb:被测系统执行函数的最大速率。这取决于被测系统上运行的程序和被测系统中处理器的工作频率。

    B:调试接口的位宽。OSD调试接口的位宽为16位, 所以称OSD的调试字为16位。TDMCU能够通过调试接口从FIFO中提取数据。

    Re:TDMCU从FIFO提取跟踪信息的最大速率。

    Rp:TDMCU处理跟踪信息的平均速率。TDMCU的功能包括复位性能计数器和读取比较性能事件值, 因此Rp被定义为平均速率, Rp取决于TDMCU的工作频率和采用的检测方法, 例如需要比较的性能事件值个数, 个数越多, 速率越小。

    定义TDMCU工作的最大速率为Rf=min (Re×B, Rp) 。即TDMCU的工作速率受提取速率和处理速率的限制。综上可得, 分支丢失速率Rdrop存在式 (1) 的关系。当Rf>Rb时, 不会产生跟踪丢失的情况;其他情况时, Rdrop与Rb呈线性关系。为了防止跟踪丢失, Rf应大于Rb, Rf中的B为固定值, 所以应提高Re和Rp, 即TDMCU的工作频率应大于处理器的时钟频率。

    4 实验与验证

    4.1 实验平台搭建

    实验的硬件平台为Zedboard开发板, 在开发板FPGA部分搭建了如图3所示的硬件结构, 整个硬件系统包括PULPino和安全调试架构, PULPino为32位系统。安全调试架构中的ATM和TDMCU采用Verilog硬件描述语言编写, TDMCU位宽为32位。实验的软件部分为FreeRToS实时操作系统。系统选用C Tuning套件中的程序作为测试例程[24], 并利用测试例程中的代码构建代码重用攻击, 在实验平台上执行正常测试例程和包含攻击的例程, 验证设计的功能。

    4.2 实验结果

    实验结果显示, 安全调试架构能够检测到模拟构建的多种代码重用攻击, 达到设计预期的功能。在65 nm工艺库下, 使用Design Compiler综合工具对PULPino中的CPU、安全调试架构中的ATM和TDMCU进行了综合分析, 其等效门数和频率结果如表2所示。由表可见ATM和TDMCU等效门总数比CPU小, 因此当ATM和TDMCU被添加进整个SoC时, 产生的资源开销在可接受的范围之内。由于本文选择了4个性能事件进行检测, 所以复位性能计数器会使被测系统产生6个时钟周期的延迟。TDMCU的工作频率是被测系统工作频率的两倍多, 不会出现跟踪丢失的情况, 确保每个敏感函数都能够得到检测验证, 但是检测性能事件值会使被测系统产生8个时钟周期的延迟。虽然本设计会产生延迟, 但是相对于利用软件工具对程序插桩来检测代码重用攻击所花费的时间, 本设计能够节省较高的检测和开发成本。

    5 结 语

    本文提出了面向代码重用攻击检测的安全调试架构, 设计了总线跟踪模块ATM和检测跟踪微控制单元TDMCU, 研究了基于性能计数器的函数级代码重用攻击检测方法, 包括离线收集正常的性能事件值。通过配置TDMCU, 在运行过程中自动检测程序是否遭受代码重用攻击, 并且无需修改处理器内部逻辑和对程序进行插桩。目前的设计仅支持检测单核系统, 下一步计划利用通用基础调试架构中可扩展的模块化属性, 使设计的安全调试架构支持多核系统的代码重用攻击检测。

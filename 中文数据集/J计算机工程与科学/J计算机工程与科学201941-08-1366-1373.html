    摘    要：

    伴随着云计算的快速发展, 海量数据等业务需求的处理无法只依赖单体应用程序。微服务软件架构模式以其模块化、可扩展、高可用的应用优势为应用程序的开发带来了新的设计思路。容器是基于共享Linux内核、面向应用的一种新兴的轻量级虚拟化技术, 以Docker为代表的容器技术为微服务提供了理想的载体。同时, 以Kubernetes为代表的容器编排工具则极大地简化了容器化微服务创建、集成、部署、运维的整个流程。在开发和运维向“面向容器”的转变中, 会带来数量庞大且关系复杂的服务组合, 此时微服务的创建与部署则变得尤为重要。从易用性角度出发, 提供了一种容器编排的可视化方法, 实践分析显示, 利用此方法进行的微服务部署不仅为研发人员提供了友好型服务创建界面, 而且还便利了服务创建过程, 提高了开发效率。

    Visualization of container orchestration in microservice environment

    ZHANG Li-min GAO Jing LI Wu-bin LIU Chen

    School of Computer Science, North China University of Technology

    Abstract：

    With the rapid development of cloud computing, processing of massive data cannot depend on any single specific application program. The microservice software architecture model brings new design paradigms for the application development due to its modular, scalable, and highly available application advantages. Containers are an emerging lightweight virtualization technology, which are based on the shared Linux kernel and application-oriented. Container technologies (Docker, a typical representative) provide an ideal carrier for microservices. At the same time, container orchestration tools (Kubernetes, a typical representative) greatly simplify the entire process of creating, integrating, deploying, and maintaining containerized microservices. In the transition from “development and maintenance” to “container-oriented”, there will be large and complex combinations of services. Thus the creation and deployment of these microservices become even more important. Considering the usability, we present a visualization method for container orchestration. Experiments demonstrate that the microservice deployment using the proposed method not only provides a friendly service creation interface for developers, but also facilitates the service creation process and improves development efficiency.

    1 引言

    随着大数据和云计算的飞速发展, 单体式应用越来越不适用于复杂的业务需求。微服务架构的出现则将规模庞大的应用分解为小的、互相连接的服务, 成功地解决了单体应用所存在的问题。此外, 由微服务组成的服务集群在传统虚拟机或物理机方式下搭建步骤繁多, 搭建逻辑复杂, 集群的安装和部署都有一定的局限性, 如配置文件之多、配置节点数量之大, 部署过程涉及计算机网络、Linux操作系统、SSH无密码登录、jdk环境配置、Shell脚本等一系列纷繁复杂的操作, 动辄分布式集群的部署以失败告终, 且无从下手找出故障根源[1,1], 这就在一定程度上拖慢了开发进度。而随着大数据与云计算的飞速发展, 服务集群的需求度也明显上升, 如何快速搭建服务集群也成了业内人士研究的重点。

    近几年, 容器为用户打开了一扇通往新世界的大门, 真正进入这个容器世界后, 却发现新的生态系统如此庞大。在生产使用中, 不论科研单位还是企业, 都会提出更复杂的需求。这时, 我们需要众多跨主机的容器协同工作, 需要支持各种类型的工作负载。企业级应用开发更是需要基于容器技术, 实现支持多人协作的持续集成、持续交付平台[2,2]。但是, 目前容器的编排主要是在Linux命令行模式进行, 导致在操作过程中出现很多冗余编写, 比如容器编排多个文件中的参数关键字, 降低了开发效率。

    本文利用容器和可视化技术实现容器的编排, 通过容器编排可视化界面来完成微服务的部署。基于以上思路, 本文提出了一种基于Docker容器技术, 利用Kubernetes容器编排工具[3,3], 选取Node.js前端开发框架实现容器编排的可视化方法。以可视化方式来创建容器微服务将极大地缩短服务的开发周期, 简化服务创建过程, 提高开发效率, 同时能清晰地展示服务编排过程中所需的参数信息, 以及服务包含的各类资源信息。

    2 相关工作

    近些年, 随着大数据和服务计算的飞速发展, 在容器和微服务方面的研究也愈来愈多。文献[4,4]主要对搭建微服务及开发框架的实现方式进行了研究分析。文献[5,5]详细介绍了微服务架构的优势, 从电网容器云平台自身实践出发, 给出了微服务架构的云端应用实践。文献[6,6]利用网络性能测试工具IPerf3对常见的虚拟化网络软件进行性能评估, 将网络软件的特性功能与Kubernetes集成相结合, 建立了微服务容器化的应用平台。文献[7,7]通过分析Docker的特点, 设计了一种基于Docker的开发者服务平台, 为开发者提供代码托管、编译发布和运行环境等服务。文献[8,8]基于Docker容器构建了Web集群, 并通过实际测试, 验证此系统达到了扩容要求并能完成部署的工作。文献[9]提供了有关容器的云编排方法的全面描述, 以及现有的解决方案和所面临的问题和挑战。文献[10]介绍了自主容器编排问题, 分析了现有技术解决方案并讨论了开放性的挑战。文献[11]简要分析了PassSSOA和Kubernetes的编排特性, 概述了2种方法的相似点和不同点, 并研究了实现2种技术联合的一些想法, 提出了Docker Container-as-a-Service层上的Jolie微服务的架构组合。文献[12]则在工业物联网 (IIoT) 领域提出了一种基于轻量级虚拟化的模块化和可扩展的架构, 并利用此架构的模块化与Docker提供的轻量级虚拟化编排相结合, 简化了管理并实现了分布式部署。

    虽然在容器和微服务方面有了上述相关研究, 但是目前很少有针对容器编排可视化方面的实践研究。

    3 容器编排可视化设计

    本节主要介绍容器编排可视化设计的原理及其实现。

    3.1 容器编排原理

    随着容器技术的发展, 以Docker为代表的容器技术在细粒度、松耦合、可移植、轻量级和虚拟化等特性上与微服务的理念相辅相成, 为微服务提供了匹配的实现机制, 在应用开发和发布的方式上有了实质性的改变。而以Kubernetes为代表的容器编排工具, 其构建在Docker技术之上, 通过它可以轻松实现容器应用实例的部署、扩展、调度等其它运营维护管理。

    容器编排利用Dockerfile[13,13]文件定制容器服务的镜像, 基于此镜像完成Kubernetes资源描述文件的编写, 进而生成容器化微服务。

    3.1.1 定制镜像

    Dockerfile是一种被Docker程序解释的脚本, Dockerfile由一条一条的指令组成, 每条指令对应一条Linux命令。相比image黑盒子, Dockerfile这种显而易见的脚本更容易被使用者接受, 它明确地表明image是怎么产生的。有了Dockerfile, 当我们需要定制自己额外的需求时, 只需在Dockerfile上添加或者修改指令, 重新生成image即可[1,1]。另外, 研发人员可以对image进行上传 (Push) 、拉取 (Pull) 、保存 (Save) 和加载 (Load) 操作。然后直接Run container, 可以对container进行Stop、Start、Restart操作。图1展示了Docker操作涉及的3大基础组件:镜像 (Images) 、容器 (Containers) 和仓库 (Repository) 。

    图1 Docker的3大基础组件   下载原图

    Figure 1 Three basic components of Docker

    如下是Docker利用Dockerfile构建的容器镜像:

    #设置继承自我们创建的 jdk7镜像

    FROM birdben/jdk7:v1

    #下面是一些创建者的基本信息

    MAINTAINER zlm

    #设置环境变量, 所有操作都是非交互式的

    ENV DEBIAN_FRONTEND noninteractive

    /*添加supervisord的配置文件, 并复制配置文件到对应目录下面*/

    COPY supervisord.conf /etc/supervisor/conf.d/supervisord.conf

    #设置hadoop的环境变量

    ENV HADOOP_HOME /software/Hadoop-2.6.5

    ENV PATH ${HADOOP_HOME }/bin:$PATH

    #复制hadoop-2.6.5 文件到镜像中

    ADD hadoop-2.6.5 /software/Hadoop-2.6.5

    #创建HDFS存储路径

    RUN mkdir -p $HADOOP_HOME/hdfs/name

    RUN mkdir -p $HADOOP_HOME/hdfs/data

    #授权HADOOP_HOME路径给admin用户

    RUN sudochown-R admin /software/hadoop-2.6.5

    #容器需要开放Hadoop 9000端口

    EXPOSE 9000

    /*执行supervisord来同时执行多个命令, 使用supervisord的可执行路径启动服务。*/

    CMD ["/usr/bin/supervisord"]

    使用docker build 命令快速生成所需容器镜像。将生成的定制化容器镜像作为image值引用到Kubernetes的Yaml文件中进行容器服务的编排操作。

    3.1.2 编写资源描述文件

    基于Kubernetes技术构建容器微服务的时间处于秒数量级, 研发人员只需编写资源描述文件 (本文中为Yaml文件) , 指定服务的名称、镜像、副本数、各类资源数量如CPU数目和内存量等, 使用kubectl create命令便可快速地生成特定服务。若需求增加, 可通过scale来扩展服务数量。Yaml文件部署服务示例如下所示:

    apiVersion: v1

    kind:ReplicationController

    metadata:

    name:hadoop-master

    labels:

    component: hadoop-master

    spec:

    replicas:1

    template:

    metadata:

    labels:

    component: hadoop -master

    spec:

    imagePullSecrets:

    -name:bfdreg

    containers:

    -name:hadoop -master

    image: yancey1989/hadoop-master:1.2

    env:

    -name: ZOOKEEPER_ADDR

    value: 10.97.97.88:2181

    -name: HDFS_HOST

    value: 10.102.206.145

    ports:

    -containerPort:8088

    hostPort: 18088

    name: dashboard

    protocol: TCP

    volumeMounts:

    -mountPath: /data

    name: storage

    volumes:

    -name: storage

    emptyDir:{ }

    -name:hadoop-slave

    以镜像的方式将开发环境分发给所有开发人员, 结合 Kubernetes的Yaml 配置文件, 简化了开发环境搭建的过程, 可以快速构建一套可复用的开发环境, 这样可以让开发者将精力更加集中在应用程序的开发上, 同时运维和测试也可以并行执行, 保证了开发、测试、发布和运维的一体化[14,14]。

    3.2 可视化设计与实现

    Node.js作为一个新兴的前端框架, 使用Module模块划分不同的功能, 以简化应用的开发。在程序中使用模块是十分方便的, 只需引入相应类库, 并将此类库的引用存放在一个变量中, 便可以使用此类库中包含的丰富的各类函数进行相应操作。这样就可以快速地设计出可视化页面, 并利用Godaddy Kubernetes API实现对容器资源相应函数的调用。

    Kubernetes通过编写服务的Yaml文件来创建服务资源。Yaml文件以key:value格式指定服务的配置参数, 比如服务的name、所依赖的image、暴露的各类port和内存量等参数。一般情况下研发人员在命令行下创建Yaml文件, 然后编写服务配置, 最后使用kubectl create命令生成服务。本文则将配置文件中的必要参数抽离出来, 进行参数的可视化, 在前台可视化界面中完成Yaml文件的生成及服务的创建。

    本文先选取Yaml文件中必要参数进行可视化, 比如服务名称name、所需镜像image等, 在界面手动输入参数值, 此参数值需要符合服务命名规范, 并能够正确描述出服务信息, 此值将被传入Yaml文件中对应的value处。其中image参数是通过选择模式填入, 此时调用Kubernetes API, 选择镜像仓库中所存在的image。将参数填写完整后, 生成的Yaml文件将会保存在Kubernetes集群环境下的某一路径。“确认”后将调用Kubernetes API的kubectl create命令生成服务。可视化设计的思路如图2所示。

    4 实验与评价

    为了测试微服务环境下容器编排可视化的开发效率, 实验选取Spark服务集群作为测试对象, 从3个角度进行测试比较, 依次是虚拟机环境下Spark服务的搭建、基于Docker的以Yaml文件方式在命令行模式下构建Spark服务和基于Docker的以可视化方式生成Spark服务, 并对服务的部署时间、服务启停速度和扩展效率3个性能指标进行比较。

    4.1 实验环境

    为了对开发的容器编排可视化功能进行验证, 本文基于的实验环境为:3台64位的ubuntu16.04 server虚拟机, 操作系统为GNU/Linux #110-Ubuntu, Docker容器v1.26, 容器编排软件Kubernetes v1.10.3, 前端Node.js框架node v8.11.2, npm v5.6.0, Node.js依赖的gcc v5.4.0。实验所在的Kubernetes集群设置1个master节点, 2个slave节点。

    4.2 实验设置

    4.2.1 在虚拟机方式下部署

    实际应用中, Spark主要部署在Linux系统的集群中。如果要完整地使用Spark, 需要预先安装Hadoop, 因此在Linux系统中安装Spark需要预先安装JDK、Scala等依赖。首先在master节点和slave节点上完成基础步骤的安装, 包括JDK安装、Scala安装、配置SSH免密码登录、Hadoop安装配置及启动, 然后再开始进行Spark集群的安装与部署[15,15]。

    虚拟机环境下部署Spark的主要步骤如下所示:

    (1) 下载并解压Spark安装文件。

    :～$ wget https://d3kbcqa49mib13.cloudfront.net/spark-2.2.0-bin-hadoop2.7.tgz

    :～$ tar -zxvf spark-2.2.0-bin-hadoop2.7.tgz

    文件释放并改名:

    :～$ rm -rf spark-2.2.0-bin-hadoop2.7.tgz

    :～$ mv spark-2.2.0-bin-hadoop2.7 spark-2.2.0

    (2) 配置Spark的环境变量。

    在/etc/profile文件结尾追加:

    export SPARK_HOME=/usr/local/spark-2.2.0

    export PATH=$PATH:$ SPARK_HOME/bin

    保存退出后, 重新运行profile, 使其生效:

    :～$ source /etc/profile

    (3) 配置Spark的相关文件。

    此处需要配置的文件有spark-env.sh和slaves 2个。首先把缓存的文件spark-env.sh.template改为spark识别的文件spark-env.sh:

    $ cp conf/spark-env.sh.template conf/spark-env.sh

    修改spark-env.sh文件, 在文末添加以下内容:

    export JAVA_HOME=/usr/java/jdk1.8.0_141

    export SCALA_HOME=/usr/scala-2.11.7

    export HADOOP_HOME=/usr/local/hadoop-2.7.2

    export HADOOP_CONF_DIR=/usr/local/hadoop-2.7.2/etc/hadoop

    export SPARK_MASTER_IP=SparkMaster

    export SPARK_WORKER_MEMORY=4g

    export SPARK_WORKER_CORES=2

    export SPARK_WORKER_INSTANCES=1

    修改slaves文件, 进入conf/slaves文件, 将slave节点添加进去。

    (4) 同步slave1和slave2节点的配置。

    使用scp命令将master节点上配置好的Spark文件复制到2个slave节点上。

    (5) 启动Spark。

    至此, Spark服务集群搭建完成, 可以通过Web查看Spark集群的运行状态, 在浏览器输入masterIP:Port进行访问。

    4.2.2 基于Docker的Yaml文件方式

    (1) 构建Spark镜像。

    通过Dockerfile文件定义镜像, 利用Docker镜像打包命令docker build生成定制化镜像, 如下所示:

    FROM java:openjdk-8-jdk

    ENV spark_ver 2.1.1

    # Get Spark from US Apache mirror

    RUN mkdir -p /opt && \

    cd /opt && \

    curl http://mirrors.tuna.tsinghua.edu.cn/apache/spark/spark-${spark_ver}/spark-${spark_ver}-bin-hadoop2.6.tgz | \

    tar -zx&& \

    ln -s spark-${spark_ver}-bin-hadoop2.6 spark &&\

    echo Spark ${spark_ver} installed in /opt

    ADD start-common.sh start-worker.sh start-master.sh/

    RUN chmod +x /start-common.sh /start-master.sh /start-worker.sh

    ENV PATH $PATH:/opt/spark/bin

    (2) 编写Yaml文件, 生成服务。

    Spark master的Yaml文件如下所示:

    apiVersion: extensions/v1beta1

    kind: Deployment

    metadata:

    labels:

    name: spark-master

    name: spark-master

    spec:

    replicas:1

    template:

    metadata:

    labels:

    name: spark-master

    spec:

    containers:

    -name: spark-master

    image: spark-2.1.1-bin-hadoop2.6

    imagePullPolicy:IfNotPresent

    name: spark-master

    ports:

    -containerPort:7077

    protocol: TCP

    command:

    - "/bin/bash"

    - "-c"

    - "--"

    args:

    - './start-master.sh ; sleep infinity'

    spark-master-service.yaml

    apiVersion: v1

    kind: Service

    metadata:

    name: spark-master

    labels:

    name: spark-master

    spec:

    type: NodePort

    ports:

    # the port that this service should serve on

    -name:webui

    port:8080

    targetPort:8080

    nodePort:31000

    -name:spark

    port: 7077

    targetPort:7077

    selector:

    name: spark-master

    Spark slave的Yaml文件如下所示:

    spark-worker.yaml

    apiVersion: extensions/v1beta1

    kind: Deployment

    metadata:

    labels:

    name: spark-worker

    name: spark-worker

    spec:

    replicas:1

    template:

    metadata:

    labels:

    name: spark-worker

    spec:

    containers:

    -image: spark-2.1.1-bin-hadoop2.6

    imagePullPolicy: "IfNotPresent"

    name: spark-worker

    ports:

    -containerPort:7078

    protocol: TCP

    command:

    - "/bin/bash"

    - "-c"

    - "--"

    args :

    - './start-worker.sh ; sleep infinity'

    利用Kubernetes容器编排工具, 编写Spark服务的Yaml文件, 在命令行输入kubectl create -f xxx.yaml命令生成服务。

    本实验采用4.1节的测试环境, 随机抽取此领域水平相当的多人对4.2节3种实验的设置分别进行实现, 并记录每次测试所花费的时间, 着重对比3种方式下所构建服务的性能指标。性能指标如表1所示。根据表1的对比可以看出, 构建服务方式很多, 但每种方式的部署过程复杂度是不同的。

    图6是通过实验得到的服务部署效率对比图, 主要从时间级别来验证, 时间级别设定为ms (毫秒) 、s (秒) 、min (分钟) 、h (小时) 。从图6可知, 基于容器技术和可视化方式构建的微服务所需时间是秒数量级, 比传统虚拟机方式构建服务所需的时间减少很多, 极大地提高了微服务的开发效率;在Kubernetes系统下, 容器的启停速度控制在毫秒级别, 并且通过调整服务的副本数可以快速响应计算节点的伸缩活动, 保证资源的快速分配和回收。

    5 结束语

    构建复杂的应用确实非常困难, 而微服务架构模式可以使复杂应用构建变得简单化。本文遵循微服务架构设计理念, 基于Docker容器技术, 利用Kubernetes容器编排工具、Node.js前端开发框架以及借助Godaddy的Kubernetes客户端库实现了容器编排的可视化, 并通过实验分析得出:本文提出的可视化服务编排方式, 大大简化了繁琐的部署工作。从传统服务生成时间处于分钟级别和可视化容器服务生成时间处于秒级别甚至是毫秒级别来看, 也极大地提高了服务生成效率。

    在下一步工作中, 我们主要进行服务的组合和可视化作业的操作。如对ZooKeeper、HBase和Hadoop 3种服务进行服务组合, 来生成服务集群或平台, 我们将通过可视化容器编排方式实现3种服务的构建和组合。另外, 我们利用可视化容器编排生成的服务去实现作业的提交和处理等功能。这样研发人员就不用进入命令行而是直接在界面上手动部署多种微服务并完成作业的运行等操作。

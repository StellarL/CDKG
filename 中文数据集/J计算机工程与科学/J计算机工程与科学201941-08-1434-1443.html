    摘    要：

    软件体系结构在软件重用中有着特殊的意义。缺乏显式的描述并使用支持体系结构重用过程的信息和缺乏有效的重用方法是软件体系结构难以重用最根本的原因。在软件设计阶段, 将元信息、元建模、反射和软件体系结构结合起来, 构造了一种支持软件体系结构重用的反射式软件体系结构。给出了基于反射式软件体系结构的软件体系结构重用过程。给出了反射式软件体系结构的具体化过程的基本原理和具体化操作的Object-Z形式化描述, 即创建元组件、创建元连接器、创建元组合件操作的形式化描述。给出了基于反射式软件体系结构的软件体系结构重用的支撑工具的设计过程。

    A reflective software architecture supporting software reusing in design stage and its formalization

    LUO Ju-bo YING Shi LIU Tian-shi

    School of Computer Science, Xi'an Shiyou University School of Computer Science, Wuhan University

    Abstract：

    Software reusing in software architecture is especially important. Reusing software architecture is a very difficult problem in the realm of software engineering. There are two main reasons for the problem: the lack of information which can explicitly describe and support the process of reusing software architecture, and the lack of an effective reusing approach. Combining meta information with meta modeling, reflection and software architecture, we construct a reflective software architecture to support software architecture reusing at software design stage. We provide the software architecture reusing process based on the reflective software architecture. We also offer the basic principles of the concrete process of the reflective software architecture, and give the Object-Z formalization description of concrete operations such as the creation of meta-component, meta-connector and meta-composite. The design process of the supporting tool of software architecture reusing based on the reflective software architecture is also illustrated.

    1 引言

    SRI International (斯坦福国际研究所) Manuel等人[1]提出了推演系统和泛理论的概念, 并基于Russion Dolls模型提出了一种Russion Dolls with logic reflection的分布式对象模型, 为统一描述反射奠定了理论基础。一个反射系统可以分为2级:基级 (Base-level) 描述了系统正常的业务功能, 元级 (Meta-level) 负责建立和维护系统自述。

    在软件体系结构的形式化描述方面[2,2], 有以下一些研究:Wright基于进程代数CSP描述, Darwin和π-ADL[3]基于π演算 (一阶或高阶) 描述, Rapide基于偏序事件集Posets描述, XYZ/ADL基于线性时序逻辑描述。Object-Z是形式规格说明语言Z的面向对象扩充[4,4], 基于集合论与数理逻辑, 具有严密的逻辑性, 它适合精确地描述大型软件系统, 并且可以对其规格说明进行形式推理。

    运行时软件体系结构动态演化方面的研究有软件体系结构动态演化[5]、软件服务的在线演化[6,6]等。文献[7,8,7,8]分别给出了面向服务的物联网软件体系结构设计、基于特征模型的软件产品自动导出方法方面的一些研究成果。

    对于在软件设计阶段软件体系结构的重用, 国内外研究机构提出了很多不同的方法, 下面是目前具有代表意义的研究成果的介绍和分析:

    (1) 面向领域的体系结构, 如产品线体系结构[9,9]、特定领域的体系结构[10,10]等的重用。产品线体系结构的重用方法需要定义一个庞大的领域通用的体系结构模型, 实际系统的体系结构可以通过对该模型进行剪裁修改后形成。特定领域的体系结构重用方法需要针对特定领域。这类重用方法缺点在于:只针对特定领域, 缺乏通用性;同时, 构造一个完善可用的领域体系结构资源集具有很大的难度。

    (2) 软件体系结构设计知识, 如体系结构风格[11]、软件体系结构模式、参考体系结构等的重用。体系结构设计的通用知识并不针对某个特定领域, 因此可以得到更广泛的应用。然而, 把体系结构设计知识以软件制品的形式进行重用时, 仍然面临着许多重大的技术障碍。

    还有软件框架的重用技术, 如目前流行的Struts、Spring等各种MVC (Model-View-Controller) 框架。软件框架可以将体系结构的设计方案连同其实现代码一起进行重用。这种重用是详细设计阶段的重用技术, 主要重用代码资源, 是支持实现阶段而非设计阶段的重用。

    总的来说, 软件体系结构重用存在的主要问题是: (1) 缺乏统一的体系结构建模方法; (2) 基于不同的体系结构描述语言; (3) 缺乏支持重用过程的信息。

    本文定义并构造了一种在设计阶段支持软件体系结构重用的反射式软件体系结构。基于反射式软件体系结构重用软件体系结构的基本方法是:将通用的体系结构语言所描述的体系结构作为基级, 将体系结构重用元信息在元级中建模, 通过定义具体化操作和反射操作, 体系结构设计人员和工具可以在元级对体系结构重用元信息这一高抽象层次设计元素进行组合与重用, 从而达到重用已有的体系结构设计结构, 构造满足需求的软件体系结构。基于反射式软件体系结构重用方法是一种更通用、更便捷的体系结构制品本身的重用方法, 它具有统一的体系结构建模方法的信息。此外, 给出了反射式软件体系结构的具体化过程的基本原理和具体化操作基于Object-Z的形式规约。此外, 还给出了基于反射式软件体系结构的软件体系结构重用的支撑工具的设计过程。

    2.3 PMB协议

    元级和基级从不同角度描述了同一个体系结构, 因此必须显式描述元级和基级之间的关联关系。元级和基级的因果关联是由PMB协议来实现的, PMB协议包括:定义元级体系结构和基级体系结构之间的交互和互操作 (包括反射操作和具体化操作, 这2种操作都为支持软件重用的操作服务, 是动态的) ;定义元级和基级的静态因果关联, 是静态的。PMB协议保证了元级体系结构的改变能同时改变基级结构, 基级体系结构的改变也能在元级软件体系结构中得到相应的改变, 确保反射式软件体系结构中元级与基级之间的因果关联始终是正确的。

    2.4 基于反射式软件体系结构的软件体系结构重用过程

    基于反射式软件体系结构, 任何一个重用操作首先需要从元级获取有关的元信息;然后根据所执行的操作, 修改元级体系结构;最后根据修改后的元级体系结构, 反射生成基级体系结构, 使元级始终正确地表示当前基级体系结构的元信息。这种基于反射的软件体系结构重用方法借鉴了组件的重用技术, 为已有的体系结构制品 (分为组件、连接器、体系结构、风格4种) 添加支持重用的元信息, 将重用元信息和原有体系结构制品封装成为自包含、自描述的大粒度的实体, 来支持“组件式”的体系结构重用。这些体系结构制品并不局限于某个特定的领域和某种特定的体系结构建模语言, 可以如同组件一样被发布、检索、组合, 同时支持适应性修改, 从而在体系结构设计过程中进行更加灵活的重用。

    3 反射式软件体系结构的具体化过程及具体化操作Object-Z形式规约

    反射式软件体系结构的具体化过程是将基级体系结构中的元信息抽取出来, 并将这些元信息的值赋给元级体系结构中所对应的元信息, 从而在元级体系结构中显式地展现这些设计的元信息。

    更确切地说, 反射式软件体系结构的具体化过程所做的工作如下:对于基级体系结构中的每个组件、连接器和体系结构本身, 在元级中分别构造一个对应的元组件、元连接器和元组合件。元组件封装并描述基级中相应组件及其类型的各种元信息。元连接器封装并描述基级中相应连接器及其类型的各种元信息。元组合件封装并描述基级中体系结构本身的元信息, 包括体系结构的外观元信息、构成元信息和配置元信息。

    下面用Object-Z对反射式软件体系结构的具体化操作进行形式化描述, 也就是创建元组件、创建元连接器、创建元组合件操作的形式化描述。

    4 基于反射式软件体系结构的软件体系结构重用方法的支撑工具

    为了支持基于反射式软件体系结构的软件体系结构重用方法的实施, 我们开发了一个基于Eclipse插件技术的软件工具。工具主要有4个用途:制作、发布和修改可重用反射式体系结构制品, 基于反射式软件体系结构的软件体系结构设计。

    4.1 工具的设计方案

    工具的体系结构设计成3层结构:表示层、逻辑层和持久层。表示层负责显示用户界面以及和用户交互;逻辑层负责处理具体的业务逻辑。持久层则负责数据的持久化以及对体系结构资源库的访问。工具的体系结构由表示层、逻辑层和持久层组成, 如图4所示。

    4.2 表示层设计

    表示层各个部分的介绍如下:

    GEF (Graphical Editor Framework) 是一个功能强大的可视化模型编辑框架, 通过GEF, 我们运用Eclipse能够快速开发出图形编辑器。

    (1) 导航视图 (Navigator) :

    显示当前工作区内项目的文件目录。

    (2) 可重用资源视图:

    显示项目当前检索处理的所有可重用制品。

    (3) 图形编辑区:

    显示当前编辑的体系结构的结构图。

    (4) 体系结构制品文本显示区:

    用于显示基级体系结构的ADL文本或者元级体系结构的MetaADL文本。

    (5) 工具箱:

    存放各种体系结构元素的模板, 可从工具箱拖放元素模板到图形编辑区中。

    (6) 属性窗口:

    显示图形编辑区中当前选中的元素的元信息列表。

    4.3 逻辑层设计

    从总体上看, 逻辑层包括4个部分:元级元素模型、基级元素模型、协调器和检索服务。

    元级元素模型和基级元素模型是反射式可重用体系结构制品的元级和基级在工具运行时的对象模型。协调器控制处理体系结构的制作操作和重用操作。检索服务提供在资源库中检索适合当前需求的可重用的体系结构制品的功能。设计人员可按不同的主题, 以关键字检索的方式在资源库中检索满足要求的可重用制品。

    4.4 持久层的设计

    持久层主要用于对可重用反射式体系结构制品进行持久化并将其保存到可重用资源库中。

    5 结束语

    本文研究并构造了一种在设计阶段支持软件体系结构重用的反射式软件体系结构, 主要包括基级体系结构模型和元级体系结构模型。基于反射式软件体系结构重用方法是一种更通用、更便捷的体系结构制品本身的重用方法。本文还研究了基于反射式软件体系结构的软件体系结构重用过程, 研究并给出反射式软件体系结构的具体化过程的基本原理和具体化操作基于Object-Z的形式化描述。此外, 给出了基于反射式软件体系结构的软件体系结构重用的支撑工具的设计过程。

    今后的工作将开展基级和元级的一致性研究以及基级和元级一致性验证的自动实现研究。

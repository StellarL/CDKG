    摘    要：

    针对软件定义网络中, 控制器无法保证下发的网络策略能够在转发设备上得到正确执行的安全问题, 提出一种新的转发路径监控安全方案。首先以控制器的全局视图能力为基础, 设计了基于OpenFlow协议的路径凭据交互处理机制;然后采用哈希链和消息验证码作为生成和处理转发路径凭据信息的关键技术;最后在此基础上, 对Ryu控制器和Open vSwitch开源交换机进行深度优化, 添加相应处理流程, 建立轻量级的路径安全机制。测试结果表明, 该机制能够有效保证数据转发路径安全, 吞吐量消耗比SDN数据层可信转发方案 (SDNsec) 降低20%以上, 更适用于路径复杂的网络环境, 但时延和CPU使用率的浮动超过15%, 有待进一步优化。

    Software defined network path security based on Hash chain

    LI Zhaobin LIU Zeyi WEI Zhanzhen HAN Yu

    Department of Electronics and Communication Engineering, Beijing Electronic Science and Technology Institute

    Abstract：

    For the security problem that the SDN (Software Defined Network) controller can not guarantee the network strategy issued by itself to be correctly executed on the forwarding devices, a new forwarding path monitoring security solution was proposed. Firstly, based on the overall view capability of the controller, a path credential interaction processing mechanism based on OpenFlow was designed. Secondly, Hash chain and message authentication code were introduced as the key technologies for generating and processing the forwarding path credential information. Thirdly, on this basis, Ryu controller and Open vSwitch open-source switch were deeply optimized, with credential processing flow added, constructing a lightweight path security mechanism. The test results show that the proposed mechanism can effectively guarantee the security of data forwarding path, and its throughput consumption is reduced by more than 20% compared with SDNsec, which means it is more suitable for the network environment with complex routes, but its fluctuates of latency and CPU usage are more than 15%, which needs further optimization.

    软件定义网络 (Software Defined Network, SDN) 通过分离控制层与转发层大幅提升其网络部署的灵活性、网络管理的动态性以及网络传输的高效性。随着SDN架构的应用场景不断丰富, 其安全问题也逐渐凸显, 成为影响该技术进一步发展的巨大障碍, 特别是受限于转发层的分布式环境以及设备认证机制的缺失, SDN的数据转发过程存在较大的问题, 由于控制器缺乏针对流表实际执行情况的有效监测机制, 且在数据转发过程中交换机并不关注上一跳设备的身份合法性, 攻击者通过控制一个恶意节点即可以偏离流表规定路径的方式来隐蔽地篡改或丢弃数据包。此外, SDN还缺乏有效的数据包完整性校验机制, 导致接收者并不能确定收到的数据包是否未经篡改。

    目前SDN相关研究中还没有提供较为有效的流策略安全方案。文献[1]中提出的SDN数据层可信转发方案 (SDNsec) 对每跳交换机的流表进行编码, 并将其放入数据包报头中来保证流策略同步和转发路径安全, 但是其复杂的计算方式和较大的数据量无法适应转发路径较复杂的环境。文献[2]中提出的轻量级SDN数据包转发验证方案 (Lightweight Packet Forwarding, LPV) 通过读取转发节点的流转发统计值来判断数据转发是否出现问题, 该方案对控制器提出了较高的要求, 同时也增加了南向接口的数据量。此外, 还有部分方案[3,4,5,6,7]利用复杂的数学模型对网络拓扑和流策略进行评估, 然后结合评估结果下发流策略, 但是这些方案为控制器和南向接口带来了较大的压力, 无法直接在实际环境中部署这些方案。为保证流策略在数据转发层得到正确执行, 防止出现转发路径偏移, 亟需一种能够对整条转发路径上的流表执行情况进行监测, 且占用较少计算资源的路径安全机制。

    本文提出了一种基于哈希链的SDN路径安全机制, 以轻量级的路径校验技术为基础, 结合消息验证码技术, 保证了网络策略的一致性以及数据负载的完整性。该机制以控制器随机生成并下发的哈希链作为转发路径上交换机的身份验证标识, 交换机通过验证上一跳交换机的标识来决定执行既定的流策略或是向控制器汇报路径偏移风险。通过使用开源控制器Ryu和软件交换机OVS (Open vSwitch) 来实现该机制, 并在Mininet模拟环境中进行测试, 证明该机制可以在较低网络开销的情况下有效实现路径验证和数据包完整性验证。

    1 相关研究

    1.1 SDN架构研究

    开放式网络基金会发布的软件定义网络白皮书规定, SDN网络分为三层结构:应用层、控制层和转发层, 其结构如图1所示。应用层主要是各种用来解释用户意图的上层应用程序;控制层是网络的控制中心, 负责网络内交换路径和边界业务路由的生成, 并负责处理网络状态变化事件;转发层主要由转发设备和连接转发设备的线路构成, 负责执行用户数据的转发, 转发过程中所需要的转发表项是由控制层生成的。

    图1 SDN体系架构   下载原图

    Fig. 1 SDN architecture

    SDN架构和传统网络架构的不同之处[8]在于, 通过控制器对网络进行集中控制, 并实现转控分离、集中控制和开放接口。控制器位于控制层, 该层与应用层之间的接口称为北向接口, 与转发层之间的接口称为南向接口。OpenFlow协议是目前在南向接口上应用最为广泛的协议, SDN控制器通过OpenFlow协议控制转发层进行数据包转发。

    1.2 安全转发相关研究

    保证数据包的安全转发是网络的基本要求, 在SDN架构中, 控制层和转发层的安全运行则是实现这一要求的基础, 一旦控制层或转发层设备出现异常或遭到攻击, 网络的安全性便会受到巨大损害。因此, 数据包的安全转发在SDN安全研究中获得了广泛的关注[9], 其研究方向大致可分为3个:

    1) SDN控制器的策略安全配置[10,11,12]。这类方法将控制器下发策略的安全性作为核心研究内容。通过改进现有的开源控制器或是设计包含安全机制的新控制器, 保证控制器能够提前评估网络并分析得出适当的策略下发方式, 进而保证数据转发的安全, 但这类方法复杂度较高, 且新增或改进的功能也给控制器带来了较大负担。此外, 这些新增模块本身的安全风险也是一个值得考虑的问题。

    2) 数据转发情况监控和检测技术[13,14,15,16,17]。流策略能否在交换机上得到正确执行直接关系着数据的转发安全, 目前较为普遍的方法是通过对流数量进行分析或是发送探测数据包来检测转发层的实际传输情况。这类方法的优点是可以在不改变现有网络策略的情况下获取转发层状态信息并保证数据转发安全, 但是也为网络带来了巨大的流量压力。

    3) 基于标签的数据包安全技术[18,19,20,21]。这类方法能够快速检测出数据包是否被篡改, 控制器也可以利用标签分析出交换机的实际转发情况, 但是在数据包中加入标签的方法会为交换机带来一定的通信和计算开销, 并且SDN交换机的计算能力较弱, 因此在实际的使用中需要考虑尽可能地降低资源消耗。

    2 安全需求及关键技术分析

    SDN架构建立在对转发层设备完全信任的基础上[22], 这为该技术的实际应用带来了巨大风险, 攻击者可以以较低的成本实施对整个网络的攻击。为保证SDN的数据转发安全, 本文主要解决了转发路径校验和数据包完整性两个问题。

    2.1 SDN转发层路径校验

    SDN控制器缺乏对数据流在转发层实际转发路径的检测手段[1], 这在实际应用环境中显然是不安全的, 图2 (a) 中攻击者一旦控制转发路径上的恶意交换机, 即可隐蔽地完成数据包窃取或篡改。为解决此问题, 转发层设备需要对上一跳交换机进行路径合法性校验, 通过一定的路径校验凭证判断上一跳交换机是否来自控制器制定的转发路径, 进而保证转发数据的路径合法, 效果如图2 (b) 所示。

    图2 路径校验效果   下载原图

    Fig. 2 Effect of path validation

    针对SDN交换机计算能力有限[8]的特点, 本文选择哈希链作为路径的校验凭证。与传统公钥算法相比, 该技术能够在保证校验凭证安全的前提下, 提高生成速度并降低存储空间占用。控制器在选择一个密码学安全哈希函数h和一个秘密种子s后, 迭代h共N次即可生成一个哈希链[23], 链尾hN (s) 记为w。如式 (1) 所示:

    w=hΝ (s) =h (hΝ-1 (s) ) =h (h (h⋯h (s) ) ) Ν          (1)

    链尾hN (s) 称为哈希链的根节点, 它类似于公钥技术中的公钥, 也称哈希链验证锚。只知道hN (s) , 但不知道s的情况下无法计算出hN-1 (s) ; 而给出hN-1 (s) 时, 只需对其进行一次哈希运算, 并将得到的值与hN (s) 比较即可判断正确性。利用哈希链的单向验证特性, 本机制可以把哈希链上的数据hi (s) 逆向发送给转发路径上的交换机SN-i+1, 由交换机根据自己的哈希值进行一次哈希运算得到hi+1 (s) , 再将生成的新哈希值与接收到的数据包中SN-i的哈希值标签进行比对, 根据结果判断是否更新哈希值标签并转发至下一跳交换机。

    2.2 SDN数据包完整性

    数据包完整性是判断一个网络是否安全可靠的重要标准, 而标准的SDN架构中尚未引入转发层数据包的完整性校验机制, 这也为SDN架构在现实环境中的应用带来了不安全因素[24]。特别是在一些对数据安全性要求较高的环境中, 接收端能否完整地接收到传输数据关系着整个网络是否可用。为保证数据包具有较强的安全性, 使用户能够接收到完整可靠的数据, 一种常用的方法便是引入消息验证码 (Message Authentication Code, MAC) 。该技术利用密钥对需要认证的消息产生新的数据块并对数据块进行哈希运算, 借助哈希函数的抗强碰撞性可以有效保护消息的完整性。本文中通信双方通过密钥协商获得消息验证码的密钥K, 发送者利用该密钥计算消息m的消息验证码MAC=h (K‖m) , 并将其封装进待转发的数据包中转发出去。接收者在收到数据包后提取其中的消息m和消息验证码, 然后根据之前获得的密钥计算出当前的消息验证码MAC′, 将其与接收到的MAC进行比较, 判断接收数据包的完整性。

    3 SDN路径安全机制设计与实现

    3.1 SDN路径安全机制设计

    鉴于目前SDN网络中缺乏高效、可靠的数据安全转发路径选择和校验机制, 本文提出了一种基于哈希链的SDN路径安全机制, 该机制主要包含数据转发路径校验模块和数据完整性校验模块:数据转发路径校验模块中由控制器采集数据流五元组信息, 计算出哈希链数据, 然后分别下发给对应的交换机, 由交换机再进行彼此之间的路径校验;数据完整性校验模块需要控制器完成消息验证码和会话密钥的生成和下发, 然后由交换机根据会话密钥和加密后的消息验证码对数据包完整性进行验证。该机制的基本模型如图3所示。

    3.2 SDN路径安全机制流程及安全性分析

    3.2.1 SDN路径安全机制流程

    本机制的流程如图6所示, 共分为5个步骤:

    ①入口交换机收到新的数据包, 流表匹配失败后利用Packet-in消息将其发送到控制器。

    ②控制器收到Packet-in消息后提取出五元组信息P, 在为其规划数据转发路径并获得交换机跳数N后, 利用哈希算法h生成哈希链Cn=hn (P) , 其中1≤n≤N。然后将逆向的哈希链数值和生成的随机数K封装进Packet-out消息中和流表一起发送给路径上的各跳交换机, 其中密钥只发送给入口交换机和出口交换机。

    ③入口交换机S1利用接收到的随机数K计算出数据净载m的消息验证码MAC=h (K‖m) , 然后将自己的哈希链数值CN和消息验证码MAC插入网络数据包中发送给下一跳交换机。

    ④中间交换机Sn收到数据包后, 提取其中的哈希链数值CN-n+2, 然后对Sn从控制器上获得的CN-n+1进行一次哈希运算后得到h (CN-n+1) =hN-n+2 (P) 。若该数值与数据包中的CN-n+2相等, 则将数据包中的哈希链数值更新为自己的CN-n+1, 并转发至下一跳交换机。

    ⑤出口交换机SN收到数据包后, 首先提取并校验其中的哈希链数值, 同步骤④。然后利用随机数K计算出数据净载m的消息验证码MACnew=h (K‖m) , 若MACnew与数据包中的MAC相同, 则完整性校验通过, 交换机删除数据包中的多余数据并将其发送到服务器。

    3.2.2 安全性分析

    SDN路径安全机制采用了哈希链和消息验证码两种安全技术, 其安全性分析如下:

    1) 转发路径的安全性。数据传输路径上的交换机都是通过单向哈希链值来进行身份认证, 进而保证路径的安全可靠。当数据包的转发路径发生偏移时, 由于恶意交换机Si只拥有自己的哈希链数值CN-i+1=hN-i+1 (P) , 且无法通过计算得出其之后路径上交换机的哈希链数值, 因此会被其他交换机发现并及时上报控制器。例如转发路径上的正常交换机Si+2会因为收到数据包中的哈希链数值不等于h (CN- (i+2) +1) =hN-i (P) 而触发报警机制。

    2) 数据完整性。该机制中使用消息验证码来保证数据的完整性, 在每一个数据包中都附带一个消息验证码, 只要攻击者修改其中任一内容, 出口交换机就能通过计算消息验证码并与传递过来的值进行比较判断出来。如果攻击者篡改了消息m, 但它无法修改消息验证码MAC=h (K‖m) , 因为这种修改会打破哈希函数的强单向性。

    3.3 关键模块设计

    本机制中的关键模块为安全数据解析存储模块和安全数据插入验证模块。其中:安全数据解析模块包含数据的生成、下发、解析存储三部分内容, 涉及控制器和交换机两种平台;安全数据插入验证模块包含插入、验证两部分内容, 部署在交换机平台上。

    3.3.1 安全数据生成存储模块

    由于SDN控制器具有全局视图能力, 并且能够在接收到的Packet-in消息中提取出转发层数据包的五元组信息, 因此本模块的计算部分由控制器完成。控制器在接收到Packet-in消息之后, 首先根据最短路径算法生成转发路径, 并获得路径上的交换机数量。然后提取出数据包的五元组信息, 将其作为哈希链的生成元, 为路径上的交换机生成一个哈希链, 使每跳交换机都能获得自己的哈希值。为保证交换机计算消息验证码时的密钥具有实时性, 控制器还需要根据数据包的五元组信息提取出固定长度的随机字符串作为交换机计算消息验证码的密钥。最后由控制器消息封装模块将哈希链数据和密钥封装进Packet-out消息中发送给相对应的交换机。为使OpenFlow协议能够支持这种新的Packet-out消息内容, 本文对OpenFlow协议进行了深度优化, 因此, 当控制器需要向交换机发送密钥时, 也可以利用这种方式发送。转发层交换机在接收到来自控制器的Packet-out消息后, 需要对数据的类型进行判断, 若是携带安全数据的消息, 则对数据包进行解析并提取出安全数据, 按照一定格式存储在交换机中。整个过程如图7所示。

    3.3.2 安全数据插入验证模块

    传统SDN交换机只能进行简单的数据处理、转发和状态收集, 为实现安全数据插入和验证功能, 需要对交换机进行优化, 添加安全数据识别和提取、简单的安全数据计算和安全数据比对校验功能。根据Open vSwitch交换机的实际软件架构和工作流程, 本机制添加了安全数据插入和安全数据验证两个安全动作, 经流表触发后, 交换机便为数据包开辟新的空间插入数据或是对已有安全数据进行校验。

    本模块的交换机执行动作内容如图8所示。路径校验动作面向数据包转发路径上的所有交换机, 由控制器计算后发送给各跳交换机, 其中首跳交换机需要对数据包进行包重构后将路径校验数据插入数据包中, 之后的交换机则要提取数据包中的路径校验数据, 校验通过后插入自己的路径校验数据, 最后在出口交换机上自动移除安全数据。完整性校验动作面向数据包转发路径上的首跳交换机和出口交换机, 由控制器将实时生成的密钥发送给这两个交换机, 然后首跳交换机根据该密钥计算出消息验证码并插入到重构后的数据包中进行转发, 当出口交换机接收到数据包后, 首先完成路径校验, 然后根据数据净载和密钥计算出当前数据包的消息验证码, 并与数据包中携带的消息验证码进行比较, 校验通过则自动移除相关数据。

    5 结语

    针对数据转发层缺乏实际转发路径校验机制的安全问题, 本文在充分研究国内外相关研究现状后, 结合多种方案的特点, 提出了一种基于哈希链的轻量级路径安全机制。利用控制层的全局视图能力, 以轻量级的哈希链和消息验证码技术作为凭借, 通过对现有的控制器和开源交换机进行深度优化, 实现了数据转发层的路径校验, 保证了控制层的安全策略在转发层的正确执行。

    本文设计并实现的路径安全机制在Mininet网络环境中进行了测试, 结果表明该机制能够根据用户需求动态添加安全策略, 在保证数据路径安全的前提下, 将通信资源和硬件设备资源的消耗保持在相对正常的范围内。

    下一步工作将考虑引入DPDK技术对开源交换机的工作机制进行改进, 通过优化交换机用户态、内核态和底层硬件之间的传输机制, 降低不同状态交换机切换时的资源占用, 同时提高交换机用户态的工作效率, 进而实现硬件资源占用的大幅降低。
